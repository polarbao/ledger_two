# LedgerTwo 项目设计文档分析报告

## 1. 项目定位与核心价值
**LedgerTwo** 是一款面向固定双人（如情侣、合租伙伴）的私有化共享记账 Web 应用。其核心价值在于剥离了“谁付款”与“谁承担”的耦合，通过“分摊”和“结算”机制，清晰回答双人生活中的财务关系（谁花了多少、谁欠谁多少）。项目目标不在于大而全的资产管理，而在于解决双人共同开销的痛点。

## 2. 技术栈架构与选型
文档中明确了极简和高可控的技术选型，符合 KISS 原则和私有化部署场景：
- **后端**：Go + SQLite。选择 Go 保证了低资源消耗与高并发性能，非常适合群晖 NAS 部署；SQLite 以单文件形式极大简化了备份与迁移成本，且足以应对双人低并发写入场景。
- **前端**：React + TypeScript + Vite。生态成熟，适合构建复杂的 Web 交互（如账单明细抽屉、统计图表），同时为后续 PWA 或跨端应用打下基础。
- **部署与环境**：通过 Docker Compose 部署到群晖 NAS，使用 Tailscale 实现内网穿透与安全访问。

## 3. 核心领域建模与数据库设计
项目的数据库设计充分体现了“财务严谨性”和“历史可回溯性”：
- **资金精度**：全系统金额采用**整数分**存储，避免浮点数精度丢失，在 UI 层转换为元展示。
- **软删除**：涉及资金变动的 `transactions` 表采用软删除，并引入 `audit_logs` 审计表，保证所有金额修改均可追踪。
- **核心表关联**：
  - `transactions`：记录流水主体。
  - `transaction_splits`：分离分摊逻辑，支持“平均分摊”、“仅付款人承担”等多种模式。
  - `settlements`：记录双方的结算行为。结算时不仅是修改历史账单状态，而是新增一条独立的结算流水。

## 4. 业务规则与交互逻辑
- **账单可见性**：通过 `private`、`partner_readable` 和 `shared` 三种状态，既满足了个人隐私需求，又实现了家庭账目的公开透明。
- **UI/UX 设计**：针对高频的“流水明细”和“记一笔”操作，桌面端采用抽屉式设计，移动端采用底部弹窗，操作流连贯。

## 5. 阶段规划建议
从《MVP Roadmap》与《PRD》可见，项目具备良好的迭代节奏。建议在 MVP 开发阶段：
1. **严格限制边界**：优先跑通“记账 -> 分摊 -> 结算”核心主线，暂缓复杂的数据导入和报表功能。
2. **重点测试分摊逻辑**：核心算法（净额 = 实际支付 - 应承担）需引入单元测试覆盖，确保各类退款、修改、删除操作下的结算金额正确。
3. **接口标准化**：严格按照 `07_DATABASE_API.md` 定义 RESTful 接口，并建议早期引入 Swagger/OpenAPI 进行前后端联调约束。

> 总结：LedgerTwo 的需求聚焦、架构轻量但具备良好的业务扩展性。其“分摊与结算”的领域设计非常清晰，Go + SQLite 的选型能够完美契合长期的私有化低成本维护需求。
