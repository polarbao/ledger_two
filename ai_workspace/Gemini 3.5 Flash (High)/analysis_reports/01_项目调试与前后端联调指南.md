# LedgerTwo 项目调试与前后端联调指南

本指南详细记录了 LedgerTwo 项目（Go 后端 + React/TypeScript 前端）的本地调试、断点排查以及前后端联调配置方法。

---

## 1. 技术栈与运行环境

*   **后端**：Go 1.22+，基于 `chi` 路由，数据库采用轻量化本地 SQLite。
*   **前端**：React + TypeScript + Vite + Tailwind CSS。
*   **推荐宿主环境**：Windows 11 + WSL2 (Ubuntu) + VSCode。
    *   *注：建议将项目置于 WSL2 的原生 ext4 文件系统内（如 `/home/polar/Projects/ledger_two`），以获得更好的磁盘 I/O 性能和完美支持 Docker 卷挂载。*

---

## 2. 后端单独调试 (Go)

后端的程序入口为 `backend/cmd/server/main.go`。

### 2.1 方法 A：VSCode IDE 断点调试（强烈推荐）

利用 VSCode 配合 Go 扩展以及 Delve 调试器进行源码级断点调试。

1.  **打开项目**：确保 VSCode 已连接至 WSL2，且当前工作目录为项目根目录。
2.  **配置文件**：确保 `.vscode/launch.json` 已存在并包含以下配置：
    ```json
    {
      "version": "0.2.0",
      "configurations": [
        {
          "name": "Debug Backend",
          "type": "go",
          "request": "launch",
          "mode": "auto",
          "program": "${workspaceFolder}/backend/cmd/server",
          "cwd": "${workspaceFolder}/backend",
          "env": {
            "APP_ENV": "development",
            "HTTP_ADDR": ":8080",
            "DB_PATH": "./data/ledger-dev.db",
            "BACKUP_DIR": "./data/backups",
            "UPLOAD_DIR": "./data/uploads",
            "SESSION_SECRET": "dev-only-change-in-production"
          }
        }
      ]
    }
    ```
3.  **新建目录**：在首次启动前，确保 `backend/data` 目录及其子目录已被创建，否则写入数据库或备份可能会报错：
    ```bash
    mkdir -p backend/data/backups backend/data/uploads
    ```
4.  **启动调试**：
    *   在 VSCode 中按 `Ctrl+Shift+D` 切换到“运行和调试”侧边栏。
    *   下拉菜单选择 **"Debug Backend"**。
    *   在后端的业务代码（例如 `router.go`、具体 handler 或 service 逻辑中）点击行号左侧设置断点（红点）。
    *   按 **`F5`** 启动，程序会在断点处挂起，此时可检查调用栈和变量状态。

### 2.2 方法 B：使用 Air 工具进行热重载实时调试

在日常开发中，如果不需要单步断点，可以使用 `air` 工具，它会在检测到代码修改时自动重新编译并热重载服务。

1.  **安装 Air**：
    ```bash
    go install github.com/air-verse/air@latest
    ```
2.  **启动服务**：
    ```bash
    cd backend
    air
    ```
    *启动后，后端将在默认端口（通常是 `8080`）提供 REST 服务。*

### 2.3 方法 C：单元测试调试

对于非 UI 部分的核心算法、业务逻辑（例如结算 settlement 或记账逻辑），通过运行测试用例是最高效的调试手段：
```bash
cd backend
go test -v ./...
```
在 VSCode 中安装 Go 插件后，可以直接在 `xxx_test.go` 源码的测试函数上方点击 `Run Test` 或 `Debug Test` 进行局部的断点调试。

## 3. 前端单独调试 (React + TS + Vite)

前端代码位于 `frontend` 目录。

### 3.1 依赖安装与启动

1.  **安装依赖**：
    ```bash
    cd frontend
    pnpm install
    ```
2.  **启动开发服务器**：
    ```bash
    pnpm dev --host 0.0.0.0
    ```
    *使用 `--host 0.0.0.0` 可以确保局域网或 WSL2 桥接网络可以直接访问。*

### 3.2 调试技巧

1.  **浏览器开发者工具 (F12)**：
    *   **Network 选项卡**：监控所有以 `/api` 开头的请求，检查 Payload（入参）和 Response（出参）的数据结构。
    *   **Console 选项卡**：查看 React 渲染异常或未捕获的 Promise 错误。
2.  **前端测试调试**：
    对于金额工具函数（`money.ts`）、日期处理函数（`date.ts`）等纯函数，可以通过 Vitest/Jest 脚本进行单独验证：
    ```bash
    cd frontend
    pnpm test
    ```

## 4. 前后端联调与跨域代理配置

开发模式下，前端和后端部署在不同的端口（例如前端 `5173`，后端 `8080`）。为了使前端的相对路径请求 `/api/...` 能够顺利到达后端服务，需要利用 Vite 的开发服务器代理功能。

### 4.1 步骤一：配置 Vite 代理 (vite.config.ts)

修改 `frontend/vite.config.ts`，加入 `server.proxy` 配置：

```typescript
import { defineConfig } from 'vite'
import react from '@vitejs/plugin-react'

// https://vite.dev/config/
export default defineConfig({
  plugins: [react()],
  server: {
    proxy: {
      // 将所有前端发送至 /api 的请求代理到后端的 8080 端口
      '/api': {
        target: 'http://127.0.0.1:8080',
        changeOrigin: true,
        // 因为后端路由本身就是以 /api 为前缀，所以无需重写路径
      }
    }
  }
})
```

### 4.2 步骤二：本地联调启动流程

1.  **启动后端**（监听 `8080` 端口）：
    可以使用 VSCode 调试运行（`F5`）或在 `backend` 目录下通过命令行启动。
2.  **启动前端**（监听 `5173` 端口）：
    在 `frontend` 目录下运行 `pnpm dev`。
3.  **访问与测试**：
    打开浏览器访问 `http://localhost:5173`。前端向 `/api/init/status` 或 `/api/auth/me` 发起请求时，Vite 会拦截该请求并无缝转发至后端的 `http://127.0.0.1:8080`，此时可完美复用 Cookie 凭证且不会触发跨域限制。

## 5. 常见问题与踩坑指南

### 5.1 跨域及 401 Unauthorized 问题
*   **现象**：前端调用登录接口成功，但后续请求依然报 401。
*   **原因**：前端请求未携带 Cookie，或代理配置中丢失了 Credentials。
*   **解决**：在前端封装的 API 客户端（`frontend/src/api/client.ts`）中，确保 fetch 配置中包含 `credentials: 'include'`。

### 5.2 数据库锁定 (database is locked)
*   **原因**：SQLite 不支持高并发的并发写入，或调试过程中某些长连接未正确释放关闭。
*   **解决**：确保在后端初始化 DB 连接时显式设置合理的连接池上限（例如最大闲置连接为 1，最大打开连接为 1，以配合 SQLite 的并发限制）。调试中断时，若有残留进程占用文件，可通过杀死残留的 Go 进程或 Delve 进程释放锁。

### 5.3 端口占用
*   **报错**：`listen tcp :8080: bind: address already in use`
*   **解决**：在 WSL 中运行 `ss -ltnp | grep 8080` 查找占用端口的 PID，并使用 `kill -9 <PID>` 终止该进程。或者在 `backend/cmd/server/main.go` 中通过环境变量临时修改监听端口。
