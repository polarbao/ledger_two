# Demo 版本范围锁定 v0.3

## 1. 目的

本文档用于锁定 Demo 版本边界，避免 AI 编码模型一次性实现过多功能，导致代码不可控、业务逻辑发散或数据库设计过早复杂化。

## 2. Demo 版本只支持的能力

### 2.1 用户与账本

1. 系统只支持一个共享账本。
2. 系统只支持两个成员。
3. 两个成员在初始化阶段创建。
4. 默认两个成员都具备管理员权限。
5. 不支持公开注册。
6. 不支持邀请码。
7. 不支持多账本切换。

### 2.2 账单类型

Demo 版本只实现：

| 类型 | 说明 | 是否进入消费统计 | 是否进入结算计算 |
|---|---|---:|---:|
| `expense` | 普通支出 | 是 | 否 |
| `income` | 收入 | 否，单独统计 | 否 |
| `shared_expense` | 共同支出 | 是 | 是 |
| `settlement` | 结算记录 | 否 | 是，用于抵扣净额 |

Demo 暂不实现：

1. 转账 `transfer`
2. 退款 `refund`
3. 周期账单
4. 附件/小票
5. 预算
6. 微信/支付宝导入

### 2.3 分摊方式

Demo 只实现两种：

1. 平均分摊 `equal`
2. 仅付款人承担 `payer_only`

预留但不实现：

1. 按比例 `ratio`
2. 按金额 `amount`
3. 按份数 `shares`

### 2.4 页面范围

Demo 只实现以下页面：

1. 登录页
2. 初始化页
3. 首页 Dashboard
4. 流水页 Transactions
5. 记一笔抽屉/弹窗
6. 账单详情抽屉
7. 结算中心 Settlement
8. 统计页 Analytics
9. 设置页 Settings，初版只做分类/标签/导出入口

### 2.5 统计范围

Demo 只实现：

1. 本月总支出
2. 本月收入
3. 分类支出 Top N
4. 标签支出 Top N
5. 成员实际支付金额
6. 成员实际承担金额
7. 当前待结算净额

## 3. Demo 版本明确不做

1. 多账本。
2. 多成员。
3. 复杂权限组。
4. 复杂账户余额核算。
5. 对账单导入。
6. OCR。
7. 预算提醒。
8. 多币种汇率。
9. 公网部署安全加固自动化。
10. App 客户端。

## 4. Demo 核心验收路径

Demo 成功的唯一核心路径：

```text
初始化两个用户
-> 登录
-> 新增个人支出
-> 新增共同支出
-> 查看流水
-> 查看统计
-> 查看谁欠谁
-> 生成结算记录
-> 导出数据
-> Docker 本地运行
-> NAS 可部署
```

## 5. AI 编码模型禁止事项

AI 不得擅自：

1. 引入 PostgreSQL、Redis、消息队列。
2. 引入微服务。
3. 引入复杂权限框架。
4. 引入 GraphQL。
5. 引入 ORM 后绕过文档中的金额和权限规则。
6. 用 float 处理金额。
7. 在前端直接计算最终结算金额作为唯一结果。
8. 删除历史账单来完成结算。
9. 直接提交真实密钥到仓库。

## 6. Demo 成功标准

只要满足以下条件，就视为 Demo 成功：

1. 两个用户可以登录。
2. 可以新增、编辑、删除支出。
3. 可以新增共同支出。
4. 可以正确计算双方净额。
5. 可以生成结算记录。
6. 首页、流水、结算、统计页面可用。
7. SQLite 数据持久化。
8. Docker Compose 可启动。
9. 群晖 NAS 可按部署文档运行。
10. 自动化测试覆盖核心金额计算。
