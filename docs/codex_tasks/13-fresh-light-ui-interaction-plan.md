# Task13：Fresh Light UI/UX 协同开发计划

状态：计划基线已冻结，代码实现未开始<br>
创建日期：2026-07-13<br>
适用阶段：v1.2 发布收口后、v1.3 业务扩展前的体验质量专项；后续业务任务中的 UI 变更持续复用本计划

## 1. 目标

将 Fresh Light 从设计目标转成可执行、可并行但不互相覆盖的前端任务，同时解决两个问题：

1. 按页面逐步迁移现有 Dark Glass 界面，不一次性重写应用。
2. 让 UI 专项与业务 Task 同步推进，避免同一组件被不同任务重复设计、重复实现或改变冻结业务口径。

本计划是 UI-FL-01 至 UI-FL-10 的执行入口。设计细节以 `docs/ui/figma/ledger-two-fresh-light-implementation-spec-2026-07-13.md` 为准，业务、金额、权限和 API 仍以 PRD、技术契约、OpenAPI 和后端实现为准。

## 2. 当前状态

- Task41-Task49 已完成，其业务行为和 `Task47U/48U/49U` 验收事实不得被本专项重写。
- Task49X 的 XLSX 业务与发布收口继续由 `docs/codex_tasks/12-v1.2-xlsx-import-special-plan.md` 管理；`UI-FL-08` 只能在其契约上更新界面。
- Fresh Light 规范清单与本地审阅包均覆盖 29 个 Frame，三个页面简称已有显式映射。
- 本地 HTML、SVG、Manifest 是生成审阅证据，不等于线上 Figma 节点、Variables 或 Auto Layout 已完整同步。
- Dark Glass 是当前已验收实现和回滚基线；Fresh Light 代码实现尚未开始。

## 3. 范围

### 3.1 包含

- Fresh Light 语义 Token、基础组件和双主题映射。
- AppShell、Dashboard、记账抽屉、流水、结算、设置、导入和分析页面的分阶段迁移。
- 375/390/430/1440 视口、可访问性和真实业务路径验收。
- 设计稿、任务文档、代码、截图和验收记录的同步更新。
- 后续业务任务涉及 UI 时的组件归属和冲突协调机制。

### 3.2 不包含

- 修改金额、分摊、结算、权限、导入去重、备份或审计规则。
- 未经独立评审新增 API、migration 或业务状态。
- 银行同步、OCR、复杂预算、原生 App 和直接通知共同支付。
- 为了视觉统一删除已有功能，或把所有页面一次性替换。

## 4. 事实源与冲突优先级

发生冲突时按以下顺序处理：

1. 已冻结 PRD、金额/权限/隐私规则。
2. 技术契约、OpenAPI、migration 和后端服务行为。
3. 当前 Task 规格与已完成验收记录。
4. Fresh Light 实施规格、Token、Variables、组件库和 Frame Manifest。
5. 在线 Figma Frame、截图和本地生成审阅包。

生成预览只能提出差异，不能反向修改业务规则。确需改变业务或 API 时，必须退出 UI-FL 任务，另开 PRD/DEV 评审和提交。

## 6. 共享组件归属

| 归属任务 | 唯一负责范围 | 其他任务使用规则 |
|---|---|---|
| UI-FL-01 | Token、按钮、Chip、Segment、Dialog、Sheet、焦点和状态语义 | 页面任务只能组合和提出缺口，不得创建局部颜色、圆角或同名组件 |
| UI-FL-02 | AppShell、导航、断点、移动 FAB、全局状态入口 | 页面任务不得自行复制导航或定义新全局断点 |
| UI-FL-04 | 表单字段分组、表单区块和 Footer 编排 | 模板、复制和页面入口复用同一表单 orchestration |
| UI-FL-05 | `ResponsiveDataList`、筛选栏、筛选 Chip 和移动列表外壳 | 导入、元数据等工作台优先复用，不另建近似实现 |
| UI-FL-10 | 跨页视觉、可访问性、回归和证据汇总 | 只登记缺陷并回派原归属任务，不直接重写业务模块 |

共享组件的 API 或视觉状态尚未冻结时，不允许两个页面任务同时修改该组件。确需并行时，先由归属任务提交基础契约，页面任务再基于该提交开发。

## 7. 同步协调机制

### 7.1 双向登记

任何业务 Task 触及前端时，任务文档必须增加或确认以下信息：

- 关联的 UI-FL 编号；没有关联时写明原因。
- 涉及路由、页面、共享组件和文件所有者。
- API、权限、金额和状态机是否保持不变。
- 与其他进行中任务是否存在同文件或同组件冲突。
- 需要同步的 Figma handoff、Frame、截图和验收记录。

UI-FL 任务启动时也必须反向列出所复用的历史/当前业务 Task，禁止只按截图改代码。

### 7.2 四个同步点

1. **任务启动**：冻结文件范围、共享组件归属、依赖和验收视口。
2. **契约冻结**：确认组件 props、状态、响应式结构和 API 不变声明。
3. **实现完成**：同一提交或同一 Task 内同步代码、任务状态和必要 handoff。
4. **验收完成**：补截图/CDP、真实业务路径、缺陷结论和回滚结果。

### 7.3 并行条件

允许并行必须同时满足：

- 不修改同一个共享组件，或组件归属任务已先冻结契约。
- 不修改同一路由的同一交互状态。
- 不需要互相等待未冻结 API、DTO 或业务规则。
- 每个任务有独立测试和截图范围。

不满足时按“基础组件 -> 页面骨架 -> 业务状态 -> 验收”顺序串行。业务缺陷优先于视觉迁移；发布阻断缺陷可暂停对应 UI-FL 波次。

## 8. 准入门槛

开始代码迁移前必须满足：

- v1.2 业务范围和 Task49X 发布边界已冻结；未完成项有明确归属，不混入 UI-FL。
- 工作树有可追踪提交基线，真实账单、数据库、密钥和原始 Figma 包未进入 Git。
- 当前应用关键页面已保留 Dark Glass 基线截图或等价验收记录。
- Frame Manifest 与本地预览清单 29/29 对齐。
- Fresh Light Token、组件状态和目标页面有本地规范；线上 Figma 未同步项明确标注。
- 主题切换或提交级回退路径明确，不能以删除 Dark Glass 作为迁移前提。

## 10. 原子任务

### UI-FL-01：Token 与基础组件

范围：语义 Token、Fresh Light/Dark Glass 映射、按钮、状态 Chip、Segment、Dialog、Sheet、空/错/权限状态和焦点样式。<br>
完成标准：页面不再新增局部主题常量；状态不只靠颜色；Dark Glass 仍可回退。<br>
验证：组件测试、lint、build、组件矩阵截图。

### UI-FL-02：AppShell 与全局导航

范围：桌面侧栏、移动底栏、账本选择、同步/草稿状态和记账 FAB。<br>
完成标准：五个一级导航稳定；375px 无遮挡；切换账本、离线和权限行为不变。<br>
验证：导航测试、375/390/1440 截图、账本切换与草稿入口回归。

### UI-FL-03：Dashboard

范围：月度摘要、结算行动、周期提醒、最近流水和分类摘要。<br>
完成标准：5 秒内找到记账和待结算金额；不显示 UUID 或开发型文案。<br>
验证：Dashboard 测试、周期确认/跳过回归、375/1440 截图。

### UI-FL-04：记账抽屉

范围：金额优先、高低频字段分层、共同支出预览、固定 Footer 和移动 Bottom Sheet。<br>
完成标准：普通支出 10 秒、共同支出 20 秒；保存并继续无脏字段继承。<br>
验证：表单纯函数和组件测试、普通/共同/复制/模板/草稿真实路径、375/390/1440 截图。

### UI-FL-05：流水工作台

范围：桌面表格、移动卡片、搜索、筛选、详情和行操作。<br>
完成标准：移动端无宽表；历史归档名称、可见性和共同/个人均有文字。<br>
验证：筛选与软删除回归、长标题和空状态、375/390/1440 截图。

### UI-FL-06：结算解释与确认

范围：范围切换、转账行动、paid/share/raw_net/settlement/final_net、复制文案和登记确认。<br>
完成标准：登记明确生成 settlement 且不改历史账单；复制不改变状态。<br>
验证：结算组件测试、复制失败兜底、真实登记路径、375/1440 截图。

### UI-FL-07：设置与元数据

范围：设置分区、分类/标签/账户、备份恢复、诊断和角色态。<br>
完成标准：owner/editor/viewer 边界不变；高风险动作说明影响并二次确认。<br>
验证：权限、归档恢复、备份诊断回归，375/1440 截图。

### UI-FL-08：导入工作台

范围：Entry、Preview、移动卡片、Row Editor、Commit Confirm 和结果状态。<br>
完成标准：CSV/XLSX、new/duplicate/suspicious/invalid/adjusted 状态清楚；preview 不写正式账单。<br>
验证：Task47U/48U/49U/49XU 用例、真实脱敏文件 preview、375/390/1440 截图。

### UI-FL-09：分析与钻取

范围：趋势、分类、成员、标签和到流水的筛选钻取。<br>
完成标准：图表有数据表述和空状态；成员展示支付、承担、净垫付；settlement 不进入消费统计。<br>
验证：统计口径回归、钻取筛选、375/1440 截图。

### UI-FL-10：全局验收

范围：视觉一致性、可访问性、响应式、真实业务路径、Dark Glass 回退和证据归档。<br>
完成标准：P0/P1 缺陷关闭；375/390/430/1440 无重叠和横向滚动；关键流程均有证据。<br>
验证：前端完整门禁、浏览器截图/像素检查、登录到结算/导入的端到端回归。

Figma/Frame/本地规范：

每个任务单独提交。若设计规范、代码和验收记录属于同一原子任务，可在该 Task 内分成“规范冻结”和“实现验收”两个提交，不应跨 Task 混合。

## 12. 验证门禁

基础前端门禁：

```bash
cd frontend
npm run lint
npm run test
npm run build
```

UI 验收至少覆盖：

- 375、390、430、1440 视口。
- 默认、加载、空、错误、离线、无权限和高风险确认。
- 键盘焦点、抽屉焦点返回、44x44 触控目标、文字对比和非颜色状态提示。
- 登录、记账、流水筛选、结算、元数据、导入 preview/commit 等受影响真实路径。

一旦 UI 改动触及业务行为，还必须执行对应后端、API、金额、权限、导入或迁移测试，不能只运行前端构建。

## 13. 风险与回滚

| 风险 | 控制 | 回滚 |
|---|---|---|
| 双主题造成样式分叉 | 先统一语义 Token，不复制页面代码 | 回退主题映射或当前 UI-FL 原子提交 |
| 并行任务修改同一组件 | 共享组件唯一归属、契约先提交 | 暂停页面任务并回到归属任务修复 |
| 视觉迁移改变业务口径 | API 不变声明、真实业务回归 | 回退页面提交，保留后端与 Dark Glass |
| Figma 与代码再次漂移 | 四个同步点、Frame/截图/任务反向登记 | 以已验收代码和高优先级事实源为准，记录设计差异 |
| 一次性改动过大 | 按 UI-FL 原子提交和波次推进 | 逐任务回退，不做整分支大回滚 |

## 14. 完成定义

UI 专项只有同时满足以下条件才可标记完成：

1. UI-FL-01 至 UI-FL-10 均有独立提交和验收记录。
2. 业务 Task、UI-FL、Figma handoff、实现和验收之间可双向追踪。
3. Fresh Light 成为默认体验前已完成 Dark Glass 回退验证。
4. 未引入未经评审的 API、migration 或业务范围。
5. 线上 Figma 未完成项继续明确标注，不以本地预览代替同步证明。
