# Task50：v1.3 多账本正式化详细实施计划

状态：Task50.1-Task50.6 已完成；v1.3.0-rc 本机独立 staging 已通过<br>
创建日期：2026-07-15<br>
前置评审：Task50P.1-P.6 已完成<br>
目标版本：v1.3，目标 schema 21

## 1. Goal

在不扩展多人分摊、不改变历史账单和结算口径的前提下，把现有“已有多账本基础”升级为可信的多账本闭环：生命周期、唯一 Owner、显式 LedgerContext、跨模块隔离、确定性当前账本、Fresh Light 管理流程以及可恢复的升级发布。

## 2. Scope

1. migration 020/021、账本生命周期、版本并发、成员约束和独立实例管理员。
2. 所有账本内请求显式提供并验证账本上下文，清理首账本 fallback。
3. 创建、重命名、归档、恢复、Owner 移交、成员离开和实例运维 API。
4. active ledger 确定性回退、无账本状态、缓存和迟到响应隔离。
5. Fresh Light 账本管理、成员管理、归档只读和无账本交互。
6. 交易、结算、报表、元数据、周期规则、模板、导入、附件、导出、备份和审计的跨账本回归。

## 3. Non-goals

1. 第三名及以上成员、聚会分摊、ratio/shares/custom 分摊和最小转账算法。
2. 邀请链接、短信/邮件/微信通知或直接通知共同支付。
3. 银行同步、OCR、原生 App、企业多租户和账号硬删除。
4. 把账本 Owner 等同于实例管理员。
5. 重写 UI-FL 已完成页面、引入第二套组件库或在 Task50.1 处理前端分包债务。
6. 在 v1.2 NAS 发布窗口混入 migration 020/021。

## 4. Current state

1. 当前 schema 19 已有 `ledgers`、`ledger_members`、owner/editor/viewer 和基础创建/成员 API。
2. 后端已有 `LedgerContext` 和部分上下文测试，但 metadata、transaction、settlement、reports、safety 等路径仍存在首账本 fallback 或兼容分支。
3. 前端已有持久化 `activeLedgerId`、账本级 query key 和 `X-Ledger-Id` 注入，但 `AppInitGuard` 在失效时直接选择列表第一项，尚无 archived/no-active/version 状态。
4. UI-FL-01 至 UI-FL-10 已完成，Fresh Light 为默认体验，Dark Glass 为显式回退。
5. v1.2 WSL staging 仍固定 schema 19；Task50 开发只能使用 `backend/data/development/task50/`。

## 5. Proposed approach

按“数据不变量 -> 上下文与 Guard -> 业务事务/API -> 前端状态 -> 页面 -> 全模块发布”顺序串行推进。每个 Task 完成后独立提交，后续 Task 不得绕过前一 Task 的门禁。

```text
Task50.1 数据基础
  -> Task50.2 显式上下文与统一 Guard
  -> Task50.3 生命周期/成员/实例 API
  -> Task50.4 active ledger 与缓存状态
  -> Task50.5 Fresh Light 管理 UI
  -> Task50.6 全模块隔离、升级与发布收口
```

实施事实源优先级：PRD 31 -> Tech 25 -> OpenAPI v1.3 -> Fixture 32 -> UI 16 -> Figma handoff -> 本计划。任何规则变化先更新上游事实源，不直接在实现中创造新口径。

## 6. 开发环境门禁

1. 浏览器仅使用 `http://127.0.0.1:5173`，后端使用 development channel；`http://localhost:38088` 保留给 WSL staging。
2. 数据库固定为 `backend/data/development/task50/ledger.db`，上传、备份和日志使用同级独立目录。
3. 自动化测试继续使用 `t.TempDir()` 或测试临时库，不连接 development/staging。
4. migration 020/021 不在 WSL staging 或 NAS 执行。
5. 每次提交前执行工作树数据审计，不提交 `.db/.sqlite/.xlsx/.xls/.env`、真实账单、上传或备份。

详细依据见 `docs/project_analysis/2026-07-15-task50-p6-development-readiness.md`。

### Task50.2：显式 LedgerContext 与统一 Guard

目标：让账本上下文成为所有账本内业务请求的强制边界，并统一 archived/RBAC 拒绝逻辑。

预计修改：

- `backend/internal/ledger/context.go`、`http_context.go`、`rbac.go`
- `backend/internal/http/router/router.go`
- transaction、settlement、dashboard、reports、metadata、recurring、template、importer、safety/attachment 相关 service/repository
- `frontend/src/api/client.ts` 与必要的 API 契约测试

实施步骤：

1. 先建立缺 header、非成员、Path/Header mismatch、archived 写入和角色矩阵失败测试。
2. 将账本内路由归入强制 LedgerContext middleware；公开/实例级路由不得伪造账本上下文。
3. 新增并集中使用 `LifecyclePolicy`、`RolePolicy`、`InstancePolicy`，handler 只负责输入输出映射。
4. 清理所有生产路径的 `GetFirstLedger*`/首账本 fallback；兼容测试代码也不得授权业务请求。
5. repository 的对象查询同时包含 ledgerID，A 账本对象 ID 在 B 上表现为不可访问。
6. 前端请求在需要账本时无 active ledger 则不发送；不得静默省略 header 后依赖后端 fallback。

绑定验收：`T50-API-005~008`、`T50-ISO-001~010`。

完成标准：所有账本内 API 缺显式上下文均稳定拒绝；archived 读/写例外符合契约；全模块 fallback 搜索清单归零；既有单账本路径回归通过。

回滚：回退 Task50.2 应用提交；schema 21 保留且不破坏旧字段读取，但不得让回退镜像进入 production。

实施结果（2026-07-16）：

1. 账本内路由已统一强制显式 `LedgerContext`，缺失 header、Path/Header 不一致、非成员、角色越权和 archived 写入分别返回冻结错误码。
2. `LifecyclePolicy`、`RolePolicy`、`InstancePolicy` 已集中落地；实例管理员不再由账本 Owner 推导，全局运维路由不消费账本 header。
3. dashboard、transaction、settlement、reports、metadata、shared expense、safety 等生产 service 已移除首账本 fallback；`/auth/me` 只返回用户和实例管理员身份。
4. 交易、模板、导入批次/明细等对象查询与写入同时校验 `ledgerID`，跨账本对象访问稳定表现为不可访问。
5. 前端 API client 已对账本请求 fail closed：无 active ledger 时不发送请求；身份、初始化、账本列表/创建和实例运维请求显式声明为全局作用域。
6. `T50-API-005~008` 与 `T50-ISO-001~010` 所涉及模块的代表性 middleware、router、service/repository 和前端 client 边界测试已通过；完整 Fixture 32 A/B 组合矩阵仍保留在 Task50.6，既有后端、前端测试、lint 和 build 回归通过。
7. 本任务未开放 Task50.3 生命周期/成员 mutation API，未改造 Task50.4 active-ledger 状态机，也未部署 WSL staging 或 NAS。

验收证据：`docs/project_analysis/2026-07-16-task50-2-ledger-context-acceptance.md`。

### Task50.3：生命周期、成员与实例运维 API

目标：基于 Task50.1/50.2 的不变量开放冻结 API，所有高风险操作具备 version、审计和稳定错误码。

建议按同一 Task 内三个可审阅子切片串行实现，全部完成后再关闭 Task50.3：

#### Task50.3A 生命周期

1. 实现账本创建与名称规则（Task50 不设总数上限）、详情、重命名、归档预检、归档和恢复 service/handler。
2. 所有 mutation 强制 `If-Match`，成功响应返回新 `ETag`/version。
3. 归档前检查 ready import batch；未结清净额只返回警告信息，不阻断。
4. archived 状态禁止业务写入，允许冻结契约定义的读取与导出。

实施结果（2026-07-16）：

1. 已实现 active/archived/all 列表、创建、详情、重命名、归档预检、归档与恢复；创建/详情/mutation 响应携带冻结格式的 ETag，mutation 强制 `If-Match`。
2. 名称按 trim 后 1-60 个 Unicode 字符校验，同名与连续创建不受任意账号级阈值限制；创建者成为唯一 Owner，初始为 active/version 1。
3. 归档事务以 version claim 作为首个写入，重新统计未过期 ready 批次，并在失败时回滚 version、过期批次收敛和审计；未结清余额复用 settlement 可信计算且不创建结算。
4. 新增 Owner-only `POST /imports/{batchId}/discard`，只把 ready 批次收敛为 expired，保留预览行和 hash，不创建 transaction，并写 `import_batch_discard` 审计。
5. 生命周期与 discard 审计写入 `actor_role`；归档 after_json 包含新 status/version、未结清快照和 ready batch count。
6. 前端已同步最终 DTO、生命周期 API、ETag helper 与显式 discard API；未提前实现 Task50.4 active-ledger 状态机或 Task50.5 页面。

目标：把当前账本变成确定性、可解释且不会串缓存的前端状态机。

预计修改：

- `frontend/src/stores/ledger.store.ts`

- `frontend/src/components/layout/AppShell.tsx`
- `frontend/src/api/queryKeys.ts`、`client.ts`、`ledger.api.ts`
- 新增 `LedgerSwitcher`、`NoActiveLedgerShell` 及相关 model/test

实施步骤：

1. 冻结 active/archived/all 响应模型和确定性 fallback：保留有效 active；否则选符合规则的 active；没有 active 则进入 no-active。
2. 切换账本时取消旧请求、使旧 ledger query 失效，并阻止 A 的迟到响应覆盖 B。
3. 当前账本被归档、移除成员或失去访问权时显示原因，不静默循环选择。
4. no-active shell 只请求身份和账本列表，不挂载交易、报表、导入等业务查询。
5. 刷新后只恢复仍有效的 active ledger；归档账本通过显式入口访问。

绑定验收：`T50-WEB-001~004`、active/fallback/cache/late-response/no-active 用例。

1. active ledger store 升级为 `validating/active/no-active/error` 状态机，只持久化账本 ID 与最近使用时间；旧 role/status 快照通过 v2 migration 清理。
2. 启动校验不再使用列表第一项，身份失败与账本列表失败分离；偏好失效时按最近使用的可访问 active 账本回退并显示一次原因。

5. 离线草稿与记账表单的分类、账户、标签、付款人等本地偏好按 ledger ID 隔离；旧版草稿迁移到最后持久化账本。
6. 设置页创建第二账本复用同一安全切换路径，不再先切换后全局 invalidate。
7. 前端全量 35 个测试文件、133 项测试、lint 和 production build 通过；既有大 bundle warning 保持非阻断。
8. 本任务未新增 migration/后端路由，未连接 development 实库，未部署 WSL staging 或 NAS。

## 8. Risks

| 风险 | 控制 | 阻断条件 |
|---|---|---|
| migration 约束与旧脏数据冲突 | 预检先于迁移，异常 Fixture 明确错误 | 不能证明一账本一 Owner 或数据守恒 |

| 前端切换串缓存 | ledger-scoped key、取消旧请求、迟到响应测试 | A 数据可渲染到 B |
| UI-FL 被业务任务反向重写 | 复用现有原语和文件所有权 | 新建平行 Token/组件库或改金额权限口径 |
| v1.2 数据被误迁移 | host、channel、路径和 schema 四重检查 | 目标不是独立 development/staging |
| NAS 门禁混入 Task50 | RC05 与 Task50.6 独立维护窗口 | 未获得用户明确部署确认 |
