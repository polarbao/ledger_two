## 1. Goal

把 Task50.3 到 Task50.6 从路线级描述收敛为可直接执行、可独立验收的任务入口，同时明确 Task50 完成后的准备顺序，避免多人分摊或通知能力在契约未冻结时提前进入代码。

## 2. Scope

1. 冻结 Task50.3 生命周期、成员和实例运维 API 的实现边界、兼容策略、测试入口和提交边界。
2. 判断 Task50.4、Task50.5、Task50.6 的准备程度和阻塞关系。
3. 补齐账本总数、归档预检、`leave` 并发版本和历史参与者展示四项契约缺口。
4. 建立 Task51P 多人分摊开发前准备门禁，并保持 Task52 为研究任务。

## 3. Non-goals

1. 本文不实现 Task50.3 代码，不执行 migration 020/021，不部署 WSL 或 NAS。
2. 不开放第三名成员、邀请链接、多人分摊、最小转账或直接通知共同支付。
3. 不把线上 Figma 同步状态当作本地 UI 实现的阻断条件，也不宣称线上文件已同步。
4. 不提前冻结 Task51 的版本号、成员上限、分摊算法或 migration 编号。

### 4.2 本轮关闭的契约缺口

1. **账本总数**：Task50 不设置账号级账本总数上限；创建仅校验 trim 后名称长度 1-60 个 Unicode 字符。若未来需要资源配额，应作为独立运维策略评审，不得在 handler 中写任意阈值。
2. **归档预检**：新增 `GET /ledgers/{ledgerId}/archive-preflight`，只返回未结清净额与未过期 `ready` 导入批次快照，不写审计、不生成结算；正式归档仍在事务内重新计算。
3. **离开账本版本**：`POST /ledgers/{ledgerId}/leave` 成功后必须返回新 `ETag`，与其他成员 mutation 使用同一版本并发口径。
4. **历史参与者展示**：成员移除不改写历史账单、分摊、结算和审计。历史姓名只能从“当前账本对象实际引用的用户 ID”定向加载；禁止返回全局用户目录，也不能只依赖当前 `ledger_members`。

### 5.1 Task50.3A：生命周期 API

状态：已完成，验收证据见 `docs/project_analysis/2026-07-16-task50-3a-lifecycle-acceptance.md`。

实现范围：

1. `GET /ledgers?status=active|archived|all`、`POST /ledgers`。
2. `GET /ledgers/{ledgerId}`、`PATCH /ledgers/{ledgerId}`。
3. `GET /ledgers/{ledgerId}/archive-preflight`。
4. `POST /ledgers/{ledgerId}/archive`、`POST /ledgers/{ledgerId}/restore`。
5. 归档阻断后使用既有导入批次放弃接口清理指定账本的 `ready` 批次；不得通过归档隐式删除批次。

主要文件：

- `backend/internal/ledger/model.go`
- `backend/internal/ledger/repo.go`
- `backend/internal/ledger/service.go`
- `backend/internal/ledger/handler.go`
- `backend/internal/http/router/router.go`
- `backend/internal/errors/errors.go`
- `frontend/src/api/ledger.api.ts`，本切片只同步 DTO，不实现 Task50.4/50.5 页面

测试先行清单：

1. 列表状态过滤与稳定排序，非成员不可读取详情。
2. 名称 trim、1-60 Unicode 字符、同名允许、连续创建不触发任意总数上限。
3. `If-Match` 缺失、格式错误、旧版本和成功后 `ETag`。
4. 仅 Owner 可重命名、预检、归档和恢复。
5. 预检不写审计、不生成结算、不改变导入批次或账本版本。
6. `ready AND (expires_at IS NULL OR expires_at > now)` 批次阻断归档；冲突必须回滚 version claim 和过期批次清理。只有归档可继续时，已到期 ready 批次才与归档在同一事务内收敛。
7. 未结清净额必须显式确认但不自动结算；归档成功后业务写入继续由 lifecycle guard 拒绝。

完成标准：生命周期路由、DTO、错误码和 OpenAPI 一致，竞争请求不能绕过 version 或 `ready` 批次阻断。

### 5.2 Task50.3B：成员与 Owner 不变量

状态：已完成，验收证据见 `docs/project_analysis/2026-07-16-task50-3b-member-acceptance.md`。Task50.1 的最多两人/唯一 Owner trigger 与 Task50.3B 事务 service 共同形成数据库/应用双层兜底。

实现范围：

1. `GET/POST /ledgers/{ledgerId}/members`。
2. `PATCH/DELETE /ledgers/{ledgerId}/members/{userId}`。
3. `POST /ledgers/{ledgerId}/members/{userId}/transfer-owner`。
4. `POST /ledgers/{ledgerId}/leave`。

兼容策略：当前代码存在 `PUT /members/{userId}`。Task50.3B 以 OpenAPI 的 `PATCH` 为正式接口，前端切换到 `PATCH`；后端暂时保留 `PUT` 兼容路由并标记 deprecated，二者调用同一 service。兼容路由的删除必须另行评审，不能在 v1.3 内静默破坏既有客户端。

事务规则：

1. 所有成员 mutation 在同一事务内校验账本 version、角色、成员上限和唯一 Owner；首个写语句只增加一次 ledger version，后续业务写入不得重复增加。
2. mutation、账本 version 增量和审计日志同事务提交；任一步失败必须整体回滚。
3. Owner 移交必须一次完成“目标成为 Owner、原 Owner 成为 Editor”，不可暴露零 Owner 或双 Owner 中间态。
4. 移除或离开只删除成员关系，不改写历史账务对象；历史参与者资料按当前账本对象引用定向解析。

测试先行清单：

1. 精确 username、用户不存在、重复成员、第三名成员、非法角色和归档写入。
2. Viewer/Editor/Owner 权限矩阵与实例管理员不自动获得账本成员权限。
3. Owner 移交成功、旧 version 冲突、目标非成员和故障注入回滚。
4. Owner 不能直接离开；Editor/Viewer 离开返回新 `ETag`。
5. 成员替换后历史 private/partner-readable/shared 可见性按 PRD 冻结，历史参与者名称可显示且不泄漏全局用户目录。

完成标准：数据库 trigger 与 service 双层证明无法产生第三名成员、零 Owner 或双 Owner，历史账务数据数量和金额不变。

## 8. Risks

| 风险 | 控制 | 阻断条件 |
|---|---|---|
| SQLite 并发下 version 先读后写 | 条件更新或 `BEGIN IMMEDIATE` 事务，检查 affected rows | 两个旧版本请求均成功 |
| 归档预检与提交间状态变化 | 预检仅展示，提交事务重新计算 | 可在新 `ready` 批次存在时归档 |
| 历史姓名查询泄漏用户目录 | 只解析当前账本对象引用的 user ID | 响应包含未被当前账本对象引用的用户 |
| `PUT`/`PATCH` 行为分叉 | 双路由调用同一 handler/service，契约测试覆盖 | 两种方法返回不同权限或事务结果 |
| Task50.4 提前固化临时 DTO | Task50.3 收口后才放行 | 前端状态机依赖未冻结字段 |
| Task51 破坏 schema 21 约束 | Task51P 的 ADR、migration 和历史可见性先冻结 | 未定义历史行为即开放第三成员 |
