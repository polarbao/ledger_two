目标版本：v1.3，候选 schema 22

## 1. Goal

在不改变 CSV/XLSX 解析、导入去重、整批事务提交和账务金额口径的前提下，完成“默认元数据 -> 分级归类 -> 人工修正 -> 显式学习 -> 后续复用”的确定性闭环。

## 2. Scope

1. migration 022、稳定 `system_key`、规则来源/应用模式和预览解释持久化。
2. `basic_cn_v1` / `empty` 默认元数据模板，覆盖首次初始化、新建账本和既有账本显式应用。
3. 人工、批量、用户规则、学习规则、内置建议和兜底的分级 resolver。
4. 行级解释、重新分类、批量接受/应用、显式记住商户和元数据保护 API。
5. Fresh Light 导入工作台、规则管理、基础包预览和兜底分类替代交互。
6. 匿名 Fixture、迁移守恒、浏览器、feature flag、回滚和发布准入。

## 3. Non-goals

1. 不引入云端 AI/LLM、外部分类 API、本地 ML 或后台隐式学习。
2. 不增加银行同步、OCR、正则/脚本规则、任意 AND/OR 规则编辑器。
3. 不实现二级分类、标签分组、分类合并、跨账本规则复制或自动账户学习。
4. 不修改历史 migration 001-021，不直接升级 WSL/NAS 数据库。
5. 不与 Task50.4-Task50.6 并行修改共享文件，不推进 Task51/Task52 代码，不改变两人约束。

## 4. Current state

1. importer 已有稳定 preview/commit、selected/suggested 字段和 Task49 规则 CRUD。
2. 现有规则只写建议，首个命中规则生效，缺少来源、置信度、冲突和解释状态。
3. 初始化只创建 5 个支出分类和 1 个标签；新建账本不创建默认分类/标签。
4. metadata 已有 Owner-only CRUD、排序、归档和恢复，可作为默认包与兜底保护基础。
5. 前端已有导入行编辑器、预览列表、规则管理和 Fresh Light 组件，不需要新建第二套 UI 框架。

## 7. Task53.2：纯分类器与历史规则兼容

### 7.1 Implementation

1. 新建 `backend/internal/importer/classifier/`，包含 normalization、candidate provider、resolver 和 built-in knowledge。
2. 使用 NFKC、Unicode 小写、trim、连续空白折叠和已验证来源噪声清理；保留原始展示值。
3. 按 `manual > bulk > user_rule > learned_rule > builtin > fallback` 解析。
4. 用户/学习规则按 priority、origin、specificity、created_at、id 稳定排序；禁止依赖 map 顺序。
5. 新增 `merchant_equals`；保留既有四种 match type，不引入正则或脚本。
6. built-in 只引用 `system_key`，只产生 medium/low 建议；兜底产生 fallback，不添加标签。
7. 高置信标签取稳定并集，超过 8 个返回冲突，不静默截断。

### 7.2 TDD first

1. Fixture CT-R01 至 CT-R12、Normalization 和 Tag limit 全部作为 table-driven tests。
2. 同输入重复 100 次输出一致。
3. equals 优先 contains；同级不同分类返回 conflict。
4. archived/missing/cross-ledger 元数据使整条候选失效。
5. 历史 `suggest` 规则仍只产生建议；新 auto/learned 才可产生 selected。
6. duplicate、invalid、skipped 不进入 classifier。

## 8. Task53.3：预览持久化、解释与重分类

### 8.1 Implementation

1. 将 classifier 接入 preview，替换现有单规则 `applyImportRules`，但 parser 和 import_hash 不变。
2. high auto 同时写 `selected_*` 和兼容 `suggested_*`；medium/low 只写 `suggested_*`；fallback 写兜底 `selected_category_id`。
3. 持久化 status、confidence、source、reason、matched rule IDs；commit 不重新运行分类器。
4. 扩展 batch summary，返回 auto/suggested/fallback/manual/bulk/conflict 数量。
5. 实现 `POST /imports/{batchId}/reclassify`，默认 dry-run；只处理 ready 批次和 eligible 行。
6. manual/bulk 行不可被 reclassify 覆盖；变化摘要可审计但不生成 transaction。

### 8.2 TDD first

1. high/suggest/fallback/conflict 的 selected/suggested 和解释快照一致。
2. preview 多次运行确定，commit 不因规则后来变化而改变结果。
3. dry-run 不写库；执行模式只更新 eligible 行。
4. stale batch、committed/expired batch、跨账本 batch 返回稳定错误。
5. 分类状态统计与行明细一致。

## 9. Task53.4：批量调整、显式学习与元数据保护

### 9.1 Implementation

1. 实现 `bulk-adjust` 的 `accept_suggestions` 和 `apply_values`，成功行标记 `bulk`。
2. `accept_suggestions` 允许同批行接受不同持久化建议；`apply_values` 可解决 conflict。
3. 实现独立 learn API，使用规范化商户精确匹配，默认当前账本/当前来源。
4. 本行保存与学习规则事务解耦；学习失败不能回滚已成功的行调整。
5. 学习规则可编辑、归档、恢复；不得覆盖显式用户规则。
6. metadata 列表返回规则引用数；引用归档对象的规则显示 stale。
7. 归档兜底分类必须在同事务指定同类型替代分类，并更新 profile/fallback 指向；不改写历史账单分类。

### 9.2 TDD first

1. accept_suggestions 接受不同分类且不创建规则。
2. apply_values 跳过 invalid/duplicate/skipped，可覆盖用户明确选中的 manual/bulk 行。
3. learn 空商户、跨账本、非法来源、显式规则冲突和故障注入。
4. 同规范化商户重复 learn 更新同一 learned rule，不重复创建。
5. 标签上限、分类类型和 archived metadata 返回稳定错误。
6. 兜底替代成功/失败事务与历史交易不变性。

## 13. Risks

| Risk | Control | Blocker |
|---|---|---|
| 历史规则升级后突然自动选择 | migration 固定 suggest，兼容测试 | 任一历史规则变为 auto |
| 兜底掩盖低识别率 | fallback 单独统计和筛选 | fallback 计入成功识别 |
| 学习污染长期规则 | 仅显式 checkbox/command，独立 learn API | 后台自动生成规则 |
| 批量接受误建长期规则 | bulk 与 learn 分离 | bulk 请求产生 learned rule |
| 分类与交易类型错配 | metadata type 校验 | 支出使用收入分类或反向 |
| 标签过多/结果不稳定 | 稳定并集、8 上限、冲突 | 静默截断或顺序漂移 |
| Task50 文件竞争 | Task53 暂缓，等待 Task50.6 后重新登记所有权 | 同时修改 ledger/router/shared UI |
| schema 22 误部署 | 独立 DB/端口/镜像、NAS 单独确认 | 直接升级现有 WSL/NAS |

## 14. Rollback

5. 默认包失败停止入口，已创建分类/标签按普通元数据保留，不自动删除或改写历史交易。
