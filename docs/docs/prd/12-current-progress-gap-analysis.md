# PRD：当前开发进展与文档差异分析

状态：供审核  
目标：从产品、技术、UI、任务计划四个角度核对 Task30 后当前项目是否与文档存在明显差异。

## 1. 差异等级

| 等级 | 含义 |
|---|---|
| P0 | 会影响后续开发方向或数据安全，必须先处理 |
| P1 | 会影响可维护性或用户体验，应在 v1.1 前处理 |
| P2 | 可后续优化，但需要记录 |

## 2. P0 差异

### 2.1 文档阶段描述不一致

现象：

- README/CHANGELOG 已描述 v1.0.0 发布。
- 旧文档仍描述 v0.3/Demo/v0.2。

影响：

- Codex/Gemini 会读取错误的产品边界。
- 后续开发可能误删或绕过已完成能力。

处理：

- 更新 `docs/00_DOCUMENT_INDEX.md`。
- 更新 `docs/prd/00-product-roadmap.md`。
- 更新 `docs/18_POST_DEMO_AI_CODING_TASKS.md`。

### 2.2 配置变量不一致

现象：

- Docker Compose 和后端配置读取的变量名需要统一。
- 生产环境必须明确强制密钥策略。

影响：

- 可能导致登录 Cookie、JWT Secret、HTTP/HTTPS 行为异常。
- 可能在生产使用默认开发密钥。

处理：

- 统一 `JWT_SECRET` 或 `SESSION_SECRET`。
- 统一 `PORT` 或 `HTTP_ADDR`。
- 更新 Docker Compose、`.env.example`、config.go、部署文档。

### 2.3 账本上下文不统一

现象：

- 前端通过 `X-Ledger-Id` 传 active ledger。
- 后端多个 service 自行查询当前用户 ledger，且存在 fallback 到第一个 ledger 的模式。

影响：

- 多账本情况下可能导致数据落错账本。
- 权限判断难统一测试。

处理：

- 引入 `LedgerContext`。
- 引入 `MembershipGuard`。
- 所有业务服务依赖统一上下文。

## 3. P1 差异

### 3.1 分类、标签、支付账户管理能力不足

现象：

- UI 文档已要求排序、归档等。
- 当前实现主要是列表读取和记账时关联。

影响：

- 长期真实记账会出现分类不可维护、标签混乱、账户不可归档的问题。

处理：

- 新增 category/tag/account 模块。
- 新增管理 API 和设置页二级页面。

### 3.2 前端设置页承载过多

现象：

- 设置页同时承载数据导出、备份恢复、周期规则、导入入口、账本成员管理等。

影响：

- 后续再加入分类/标签/账户管理和邀请管理会变得拥挤。

处理：

- 设置页拆分为多个分区或二级页面。
- 数据安全、账本成员、分类标签账户、系统诊断分离。

### 3.3 附件访问控制不足

现象：

- 附件以静态目录形式暴露，缺少账单权限校验。

影响：

- private 账单附件有潜在泄露风险。

处理：

- 改为受保护附件 API。
- 下载/预览时校验当前用户是否可见关联账单。

### 3.4 API 契约文档不足

现象：

- 当前 API 已经较多，但缺少 OpenAPI 或完整 contract。

影响：

- 后续 Web/PWA/移动端复用成本高。
- AI 编码容易误用字段。

处理：

- 新增 OpenAPI 草案或自动生成流程。
- 稳定分页、错误、DTO、时间和金额规范。

## 4. P2 差异

1. 代码风格规范未集中在 `docs/codex_tasks/`。
2. PR/Commit 规范不够机器可读。
3. E2E 尚未作为必选门禁。
4. 诊断中心和审计日志 UI 可后续增强。
5. PWA/离线草稿能力需要进一步核对浏览器兼容性和测试覆盖。

## 5. 进入 v1.1 前的最低完成标准

1. 文档事实源完成。
2. 配置安全完成。
3. LedgerContext/RBAC 完成。
4. 分类/标签/账户管理至少完成后端 contract 和基础 UI。
5. API contract 完成。
6. 测试矩阵覆盖多账本、权限、迁移、导出和附件。
7. `docs/codex_tasks/` 成为后续任务唯一入口。
