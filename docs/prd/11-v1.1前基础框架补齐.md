# PRD：v1.1 前基础框架补齐

状态：供审核  
目标版本：Foundation before v1.1  
说明：本 PRD 不定义 v1.1 具体业务功能，而是定义进入 v1.1 前必须补足的基础框架能力。

## 1. 背景

Task01-Task30 已完成后，LedgerTwo 已经具备 v1.0/MVP 级别功能。但由于项目早期是固定双人账本 Demo，后续在开发中又逐步加入了多账本、多成员、导入、备份、PWA、离线草稿等能力，导致当前产品文档、技术文档和代码实现之间出现差异。

为了避免 v1.1 继续在未统一的基础上开发，本阶段先补足基础框架。

## 2. 目标用户

本阶段直接服务对象不是最终记账用户，而是后续开发者、AI 编码工具和项目维护者。

| 角色 | 诉求 |
|---|---|
| 产品负责人 | 知道当前 v1.0 已完成什么、v1.1 前需要补什么 |
| 开发者 | 知道基础架构边界、模块职责和测试要求 |
| Codex/Gemini | 有清晰任务、输入文档、代码风格和验收标准 |
| 运维者 | 知道部署配置、备份恢复、日志和诊断如何验证 |

## 3. 核心问题

本阶段解决以下问题：

1. 文档事实源不一致。
2. 配置变量和部署文档不一致。
3. 多账本上下文缺少统一框架。
4. 角色权限散落在业务模块中。
5. API 合同没有正式冻结。
6. 分类/标签/支付账户管理能力不完整。
7. 前端状态管理不适合长期多账本使用。
8. 测试矩阵不覆盖 v1.1 前置风险。
9. AI 开发规范不够严格，容易导致大范围无边界改动。

## 4. 功能需求

### 4.1 文档事实源收口

系统文档必须统一描述当前阶段：

- Task01-Task30 已完成。
- v1.0/MVP 已完成。
- v1.1 具体业务规划尚未审核。
- 当前先做 Foundation before v1.1。

验收标准：

1. `docs/00-文档索引.md` 不再描述 v0.3 Demo 为当前状态。
2. `docs/prd/00-产品定位与版本路线.md` 不再写“下一阶段进入 v0.2”。
3. `docs/18-演示后AI编码任务与提示词.md` 标记 Task30 完成并指向 `docs/codex_tasks/`。
4. README、CHANGELOG 与 docs 状态不冲突。

### 4.2 配置与部署安全

产品必须能让用户明确知道：生产部署需要哪些环境变量，缺失会怎样，HTTP/HTTPS 如何影响登录 Cookie。

验收标准：

1. Docker Compose、`.env.example`、config.go、部署文档使用同一套变量名。
2. 生产环境缺少强 `JWT_SECRET` / `SESSION_SECRET` 时拒绝启动。
3. 局域网 HTTP 模式和 HTTPS 反代模式有明确说明。
4. `/api/healthz` 返回版本、数据库状态、schema version 和必要诊断信息。

### 4.3 LedgerContext 与权限框架

所有业务 API 必须明确当前账本上下文。

验收标准：

1. 当前用户、当前账本、当前角色可以统一解析为 `LedgerContext`。
2. 业务 API 不再默认静默落到“用户第一个账本”。
3. owner/editor/viewer 权限矩阵写入文档。
4. 后端通过统一 Guard 校验角色，而不是各 service 复制 SQL。
5. private / partner_readable / shared 在多成员账本中的含义被明确。

### 4.4 API 契约

API 必须能长期被 Web、PWA、未来移动端复用。

验收标准：

1. 当前 `/api` 兼容策略明确。
2. `/api/v1` 规划明确。
3. 成功和错误响应结构稳定。
4. 金额、时间、分页、排序、筛选字段命名稳定。
5. 输出 OpenAPI 草案或生成流程。

### 4.5 分类、标签、支付账户管理

分类、标签、支付账户是后续个人、情侣、家庭记账的基础配置，不能只作为初始化默认数据。

验收标准：

1. 分类支持新增、编辑、排序、归档、恢复。
2. 标签支持新增、编辑、排序、归档、恢复。
3. 支付账户支持新增、编辑、排序、归档、恢复。
4. 已归档项不出现在新增账单默认选择器。
5. 历史账单仍保留并展示已归档项。
6. 已被账单使用的分类/标签/账户不能直接物理删除。

### 4.6 前端基础框架

前端必须支持长期多账本、角色权限和后续扩展。

验收标准：

1. active ledger 通过统一 Provider 管理。
2. 所有 TanStack Query key 包含 ledgerId。
3. 切换账本时不依赖 `window.location.reload()`。
4. RoleGate / PermissionGate 统一控制按钮和页面入口。
5. 设置页拆分为更清晰的信息架构。

### 4.7 测试与质量门禁

验收标准：

1. CI 包含后端测试、前端 lint/test/build、Docker build。
2. 新增多账本隔离测试。
3. 新增 owner/editor/viewer 权限矩阵测试。
4. 新增分类/标签归档后历史账单展示测试。
5. 新增迁移回归测试。
6. 新增导出和附件权限测试。

## 5. 不做事项

1. 不实现 v1.1 未审核业务闭环。
2. 不做公开邀请链接。
3. 不做邮件/短信/微信通知。
4. 不引入微服务。
5. 不强制迁移 PostgreSQL。
6. 不做移动原生 App。
7. 不为了重构而大规模改 UI 风格。

## 6. 风险

| 风险 | 说明 | 应对 |
|---|---|---|
| 文档与代码继续偏离 | AI 会读取错误上下文 | Task31 优先完成 |
| 配置安全漏洞 | 生产可能使用开发密钥 | Task32 阻断 |
| 多账本数据串账 | fallback 第一个账本导致数据落错 | Task33 统一 LedgerContext |
| 分类标签不可维护 | 长期记账分类混乱 | Task35 补齐 |
| 大模型一次性重构过大 | 破坏 v1.0 功能 | codex_tasks 采用小步任务 |

## 7. 交付物

Foundation before v1.1 完成后，应至少交付：

1. 更新后的 PRD / Tech / UI 文档。
2. `docs/codex_tasks/` 任务和代码风格规范。
3. 配置安全文档和 `.env.example` 对齐说明。
4. RBAC 权限矩阵。
5. API 契约文档。
6. 测试矩阵和质量门禁。
7. 后续 v1.1 业务规划的输入模板。
