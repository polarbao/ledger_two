# PRD：产品复盘与定位重整

状态：建议采纳  
适用阶段：Task30 后，Foundation before v1.1  
角色视角：产品经理 + 开发经理

## 1. 一句话定位

LedgerTwo 不是大而全的个人财务软件，而是面向情侣、夫妻、合租伙伴和小家庭的私有化共同生活账本。

它最重要的价值不是“记下一笔账”，而是长期回答：

1. 谁实际付款。
2. 谁实际承担。
3. 哪些账单双方可见，哪些保持私人。
4. 当前谁该给谁多少钱。
5. 数据是否安全、可备份、可恢复、可迁移。

## 2. 当前产品事实

当前项目已经完成 Task01-Task30，具备 v1.0/MVP 级别能力：

- 登录、初始化、账本基础。
- 普通账单、共同支出、分摊、结算。
- 统计、导入导出、备份恢复、PWA、离线草稿。
- 多账本/成员/RBAC 已有雏形。
- NAS Docker Compose 部署文档已补齐。

但文档、代码、实际产品定位存在历史层叠：

- 早期 Demo 文档强调固定双人单账本。
- 后续实现已经进入多账本、多成员、导入、PWA、恢复备份等能力。
- v1.1 业务尚未正式审核，不应直接开发。

结论：不重写产品，不推翻技术栈，先做事实源收口和基础框架补齐。

## 3. 核心用户场景

### 3.1 高频核心场景

| 场景 | 用户问题 | 产品解法 |
|---|---|---|
| 日常生活支出 | 今天谁付了饭钱/超市钱 | 快速记账、默认分类、默认付款人 |
| 共同支出 | 一笔钱两个人或多人怎么承担 | split 明细、后端结算计算 |
| 阶段性结算 | 月底谁该给谁多少钱 | 结算中心、历史结算记录 |
| 隐私账单 | 我自己的账不想让对方看到 | private / partner_readable / shared |
| 长期数据保存 | 换设备、NAS 故障、升级怎么办 | 备份、导出、恢复、迁移规范 |

### 3.2 中频增强场景

| 场景 | 价值 | 优先级 |
|---|---|---|
| 复制一笔 / 模板 | 降低重复记账成本 | 高 |
| 周期账单提醒 | 房租、水电、订阅类固定账单 | 中 |
| 微信/支付宝 CSV 导入 | 降低历史补账成本 | 高 |
| 分类标签账户管理 | 让长期统计可维护 | 高 |
| 年度/月度报告 | 提供复盘价值 | 中 |

### 3.3 低频或远期场景

| 场景 | 判断 |
|---|---|
| 直接通知共同支付 | 延后。更适合聚会/旅行/多人临时账本，不是日常双人长期账本刚需 |
| OCR 小票 | 延后。成本高、错误风险高，不如先做好 CSV 导入 |
| 银行自动同步 | 不做近期规划。安全、维护、接入成本都高 |
| 原生移动 App | 延后。先用 PWA 和移动 Web 验证长期使用 |
| 复杂预算系统 | 延后。先做好记账、分摊、结算和统计口径 |

## 4. 记账软件的软肋复盘

### 4.1 记账动作太重

问题：用户记账是反人性的，字段越多越容易放弃。

策略：

- 默认值优先：最近分类、最近账户、默认付款人、最近标签。
- 高频路径少字段，低频路径折叠到高级设置。
- 移动端金额输入必须顺手。
- 复制一笔和模板优先于复杂自动化。

验收：

- 普通支出 10 秒内可完成。
- 共同支出 20 秒内可完成。
- 保存后 Dashboard、流水、结算自动刷新。

### 4.2 共同支出容易算错

问题：付款人、承担人、创建人、归属人混淆后，用户会失去信任。

策略：

- 后端是结算唯一可信来源。
- 前端只展示，不自行计算最终结算。
- 结算不修改历史账单，只生成 settlement。
- 修改历史共同支出时明确提示会影响当前待结算。

验收：

- A 付 200、B 付 80、B 结算 60 后归零稳定通过。
- 删除/修改 shared_expense 后余额重新计算。
- 多人建议转账路径可解释。

### 4.3 隐私和共同账本边界敏感

问题：情侣/家庭账本中，隐私泄露比功能缺失更严重。

策略：

- private、partner_readable、shared 必须由后端过滤。
- 附件访问也必须跟随账单权限，不允许裸静态路径绕过。
- 多账本后必须通过 LedgerContext 和 membership 校验。

验收：

- private 账单和附件对非授权成员不可见。
- viewer 不能写入。
- 非成员不能通过 ledger_id 访问账本数据。

### 4.4 数据安全决定长期留存

问题：账务数据一旦丢失，用户不会再信任系统。

策略：

- SQLite 单文件适合 NAS，但必须重视备份和恢复演练。
- 升级前自动备份。
- 恢复流程必须三段式确认或手动替换引导。
- 导出文件提示隐私风险。

验收：

- 手动备份、备份列表、导出 CSV/JSON 可用。
- 恢复前自动留 pre_restore 备份。
- NAS 文档明确 data/backups/uploads/logs。

### 4.5 文档事实源混乱会放大 AI 开发风险

问题：AI 读取旧 Demo 文档后可能误删新能力，或提前实现未审核 v1.1。

策略：

- 当前事实源：`docs/README.md`、`docs/prd/20-23`、`docs/codex_tasks`、Foundation 文档。
- 旧 v0.3 / Demo 文档保留为历史，不作为当前阶段唯一依据。
- 后续任务必须有 PRD + Tech/UI + Codex task。

验收：

- AI 开始开发前能明确当前阶段、非目标、验收标准。

## 5. 产品原则

1. 可信赖优先于功能数量。
2. 快速记账优先于复杂配置。
3. 结算准确优先于视觉效果。
4. 数据安全优先于自动化便利。
5. PWA 和移动 Web 优先于原生 App。
6. CSV 导入优先于 OCR 和银行同步。
7. 直接通知共同支付延后，先做复制结算文案。

## 6. 当前不做

- 不做银行自动同步。
- 不做 OCR。
- 不做企业级多租户。
- 不做复杂预算和资产负债。
- 不做原生 App。
- 不做自动通知共同支付。
- 不做未审核 v1.1 邀请机制。

## 7. 产品成功指标

短期：

- 普通支出 10 秒内完成。
- 核心结算测试稳定通过。
- 文档事实源统一。
- 生产部署配置不允许弱密钥。

中期：

- 连续 30 天记账留存。
- 用户能完成 CSV 导入并确认去重。
- 移动端无横向滚动。
- 分类/标签/账户长期可维护。

长期：

- 家庭/旅行/装修等多账本场景可稳定使用。
- 多人分摊和结算建议可解释。
- 年度报告和长期趋势有复盘价值。
- 数据可备份、可恢复、可迁移。
