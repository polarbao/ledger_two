# PRD：短期、中期、长期产品路线图

状态：建议采纳  
适用阶段：Task30 后

## 1. 路线图总原则

LedgerTwo 后续规划分三层：

1. 短期：可信赖地长期使用。
2. 中期：降低记账成本，提高真实使用频率。
3. 长期：扩展为轻家庭/小范围成员共同账本。

不以功能数量衡量进度，而以真实账务可用性、结算可信度、数据安全和移动端高频体验衡量。

## 2. 短期规划：Foundation + v1.1

建议周期：2-4 周。

目标：把当前 v1.0/MVP 从“功能完成”推进到“可放心长期使用”。

### 2.1 Foundation before v1.1

范围：

- 文档事实源收口。
- 配置与部署安全。
- LedgerContext。
- RBAC。
- API 契约和 OpenAPI 草案。
- 分类、标签、支付账户管理基础。
- 前端 LedgerProvider 和 query key。
- 设置页信息架构重组。
- 迁移、测试和质量门禁。
- 附件访问控制。
- 审计与系统诊断。

对应任务：

- `docs/codex_tasks/05-foundation-task-plan.md` Task31-Task40。

细化文档：

- `docs/prd/24-short-mid-module-breakdown.md`
- `docs/prd/25-prd-v1.1-module-specs.md`
- `docs/tech/18-short-mid-architecture-slices.md`
- `docs/ui/14-v1.1-v1.2-module-flows.md`

验收：

- 文档不再互相冲突。
- 后端业务 API 不再隐式 fallback 到用户第一个账本。
- viewer 无法写入。
- private 账单与附件不泄露。
- 生产环境缺少强密钥时拒绝启动。
- `go test ./...`、前端 lint/test/build、Docker build 可作为质量门禁。

### 2.2 v1.1：可信赖和高频记账版

目标用户：

- 情侣、夫妻、合租伙伴、小家庭。

核心能力：

1. 快捷记账默认值。
2. 复制一笔和账单模板体验统一。
3. 周期账单提醒只做“待确认”，不自动入账。
4. 分类/标签/账户管理进入设置页稳定入口。
5. 移动端记账、筛选、结算优化。
6. 结算页增加“复制结算文案”，不做自动通知。

非目标：

- 不做直接通知共同支付。
- 不做 OCR。
- 不做银行同步。
- 不做原生 App。
- 不做公开邀请机制。

验收：

- 普通支出 10 秒内完成。
- 共同支出 20 秒内完成。
- 月底结算路径清楚。
- 设置页能管理分类、标签、账户。
- 移动端 375px 无横向滚动。

## 3. 中期规划：v1.2 数据导入与省时间

建议周期：4-8 周。

目标：减少手工录入，让用户愿意补历史账并保持长期记账。

### 3.1 CSV 导入

范围：

- 微信/支付宝 CSV 基础解析。
- 上传后预览，不直接写库。
- 字段映射。
- 去重分析。
- 用户确认后事务落库。
- 导入批次可追踪。

验收：

- 同一文件重复导入不会产生重复账单。
- 导入失败当前批次回滚。
- 导入记录写审计日志。

细化文档：

- `docs/prd/26-prd-v1.2-import-module-specs.md`
- `docs/codex_tasks/09-task41-49-detailed-plan.md`

### 3.2 导入规则

范围：

- 商户/描述关键词规则。
- 自动推荐分类、标签、账户。
- 预览页允许人工调整。
- 规则只辅助，不覆盖用户确认。

验收：

- 常见商户能命中规则。
- 用户修改优先于规则。

### 3.3 附件与凭证

范围：

- 附件上传、查看、删除。
- 权限受控下载。
- 附件目录纳入备份说明。

非目标：

- 不做 OCR。
- 不把附件存数据库 BLOB。

## 4. 长期规划：v1.3-v1.5 轻家庭账本

建议周期：8 周以上，按模块推进。

目标：从双人长期账本扩展到小家庭、旅行、装修、合租等小范围账本。

### 4.1 v1.3 多账本与成员正式化

范围：

- 多账本创建、切换、归档。
- 成员角色 owner/editor/viewer。
- 账本成员管理。
- 多账本数据隔离。

暂不做：

- 公开邀请链接。
- 邮件/短信邀请。

验收：

- A 账本数据不会出现在 B 账本。
- viewer 不能新增/编辑账单。
- owner 可管理成员和账本设置。

### 4.2 v1.4 多人分摊和结算体验

范围：

- 多人 equal、amount、ratio、shares。
- 最小转账建议。
- 结算路径解释。
- 聚会/旅行账本体验。

可选：

- 直接通知共同支付进入调研，不默认实现。

### 4.3 v1.5 复盘和健康状态

范围：

- 月度/年度报告。
- 长期趋势。
- 备份健康检查。
- 恢复演练向导。
- 长时间未结算提醒。

## 6. 里程碑定义

- Task31-Task40 完成。

### M2：v1.1 日常可用完成

### M3：v1.2 数据导入完成

### M4：v1.3 轻家庭账本完成

- 报告、趋势、备份健康、恢复演练。

## 7. 决策：直接通知共同支付

结论：延后。

原因：

- 日常双人长期账本更需要阶段性结算，而不是每笔通知。
- 自动通知会引入渠道、状态、提醒策略和关系压力。
- 聚会/旅行是更合适场景，但这属于轻家庭/临时多人账本成熟后的增强。

短期替代：

- 结算页提供“复制结算文案”。
- 用户自行粘贴到微信/短信。
- 不跟踪支付状态，不做自动提醒。
