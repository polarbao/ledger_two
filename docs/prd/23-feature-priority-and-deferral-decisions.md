# PRD：功能优先级与延后决策

状态：当前优先级基线，持续维护<br>
更新时间：2026-07-16<br>
目的：避免后续开发把低频或高风险功能过早做进主线。

## 1. 决策原则

1. 高频优先于低频。
2. 低风险闭环优先于外部依赖。
3. 私有数据安全优先于自动化便利。
4. 双人/家庭长期账本优先于聚会 AA。
5. 先验证 PWA，再考虑原生 App。

## 2. 必做功能

| 功能 | 原因 | 阶段 |
|---|---|---|
| 文档事实源收口 | 防止 AI 误判阶段 | Foundation |
| 配置安全 | 真实部署基本要求 | Foundation |
| LedgerContext/RBAC | 防串账、防越权 | Foundation |
| 附件访问控制 | private 附件不可泄露 | Foundation |
| 快捷记账默认值 | 直接影响每日使用 | v1.1 |
| 分类标签账户管理 | 长期账本可维护 | v1.1 |
| 移动端优化 | 高频入口在手机 | v1.1 |
| CSV 导入预览去重 | 降低补账成本 | v1.2 |
| 分类标签分级自动化 | 降低导入后的逐条修正成本，同时保持可解释和可回退 | Task53，准备完成；Task50.6 后复评实现顺序 |

Task53 的自动化边界：人工调整最高优先；只允许用户规则和用户明确创建的学习规则进行高置信自动选择；内置本地知识只建议；批次提交始终由用户确认；不使用云端 AI/LLM 或后台隐式学习。

## 3. 应做但不急

| 功能 | 原因 | 建议阶段 |
|---|---|---|
| 账单模板 | 降低重复录入 | v1.1 |
| 周期账单提醒 | 固定费用有价值 | v1.1/v1.2 |
| 年度/月度报告 | 提升复盘价值 | v1.5 |
| 备份健康提醒 | 提升数据安全感 | v1.5 |
| 多人旅行/聚会账本 | 场景明确但非当前核心 | v1.3+ |

## 4. 延后功能

### 4.1 直接通知共同支付

结论：延后到多人临时账本成熟后再评估。

原因：

- 日常双人账本更多是滚动记账和阶段性结算，不需要每笔通知。
- 通知会引入渠道选择、阅读状态、支付状态、重复提醒、关系压力。
- 如果接微信/短信/邮件，会增加部署、安全、隐私、运维复杂度。
- 聚会/旅行才是更自然场景，但这属于多人临时账本体验，不是 v1.1 刚需。

短期替代：

- 结算页提供“复制结算文案”。
- 用户自行粘贴到微信。
- 系统不跟踪通知状态，不自动提醒。

未来触发条件：

- 多人临时账本使用频率明确。
- 用户真实反馈需要提醒和支付状态。
- 已有 Role/RBAC/通知偏好/审计基础。

### 4.2 OCR 小票

结论：延后。

原因：

- OCR 错误会污染账务数据。
- 小票上传和查看价值先于识别。
- CSV 导入能更直接降低录入成本。

未来触发条件：

- 附件功能稳定。
- 用户有大量小票留存需求。
- 能提供人工确认预览。

### 4.3 银行自动同步

结论：近期不做。

原因：

- 接入成本、法律合规、隐私风险和稳定性成本都高。
- 私有化 NAS 场景不适合强依赖外部金融接口。

### 4.4 原生 App

结论：延后。

原因：

- PWA 和移动 Web 足以验证长期使用。
- 原生 App 会让维护成本翻倍。

未来触发条件：

- PWA 长期使用后仍有系统能力缺口。
- 有明确推送、相册、离线同步等原生需求。

### 4.5 复杂预算系统

结论：延后。

原因：

- 预算不是 LedgerTwo 当前核心痛点。
- 先把记账、分摊、结算、统计做好。

## 5. 不做功能

| 功能 | 原因 |
|---|---|
| 企业报销审批 | 偏离家庭/双人账本 |
| 复杂复式会计 | 不符合用户心智 |
| 投资资产管理 | 另一个产品方向 |
| 公开多租户 SaaS | 与私有化 NAS 定位冲突 |

## 6. 每次新需求评审问题

任何新需求进入开发前，必须回答：

1. 这是日常高频还是偶发低频？
2. 是否会影响账务数据安全？
3. 是否需要外部服务？
4. 是否能先用低成本替代方案验证？
5. 是否有明确验收标准？
6. 是否会破坏当前核心路径：记账 -> 分摊 -> 结算 -> 备份？

如果不能回答，不进入开发。
