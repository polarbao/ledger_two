# PRD：v1.1 模块级需求规格

状态：建议采纳  
目标版本：v1.1  
前置条件：Foundation before v1.1 完成

## 1. 快捷记账模块

当前实现状态：已完成服务端默认值持久化、`GET /api/transaction-defaults` 读取、表单默认值回填、普通账单账户选择、保存并继续记、金额输入移动端优化和 375px 真实 UI 提交验收；验收证据见 `docs/project_analysis/v1.1-ui-submit-acceptance-2026-07-07/`。

### 1.1 目标

减少每日录入阻力，让用户在手机上完成一笔普通支出不超过 10 秒，共同支出不超过 20 秒。

### 1.2 范围

必须支持：

- 最近使用分类、账户、标签、付款人、可见性。
- 普通支出默认填充最近配置。
- 共同支出默认两人参与、equal 分摊。
- 保存并继续记。
- 金额输入聚焦和数字键盘优化。
- 保存后刷新流水、Dashboard、结算相关缓存。

暂不支持：

- AI 自动分类。
- 根据商户自动推荐。
- 离线正式提交。

### 1.3 规则

- 默认值只能作为建议，用户可修改。
- 默认值不得越权使用用户不可见的分类、账户、标签。
- 已归档分类、标签、账户不作为新账单默认值。
- 保存并继续记时保留分类、账户、付款人、可见性，清空金额和备注。
- 默认值按“账本 + 当前用户”隔离保存，普通账单保存账户默认值，共同支出不保存账户默认值。

### 1.4 验收

- 新增普通支出只填金额即可保存，前提是已有合法默认值。
- 新增共同支出默认生成两个人的 split preview。
- 保存并继续记不会复制上一笔金额。
- viewer 不显示新增入口，后端也拒绝写入。

## 2. 复制与模板模块

### 2.1 目标

把重复记账从“重新填写表单”变成“复制、微调、保存”。

### 2.2 范围

复制一笔：

- 普通支出、收入、共同支出都可复制。
- 复制保留标题、分类、账户、付款人、标签、可见性、分摊方式、备注。
- 发生时间默认当前时间。
- 金额默认保留，但用户保存前必须可见。

账单模板：

- 可从已有账单创建模板。
- 可从模板生成新账单。
- 模板可编辑、归档、恢复。
- 模板不进入统计、结算和导出正式账单列表。

暂不支持：

- 模板共享市场。
- 模板自动触发入账。

### 2.3 规则

- 复制和模板生成都必须重新校验分类、账户、成员和权限。
- 原账单被删除后，已创建模板不受影响。
- 模板引用已归档分类时，生成账单前提示用户确认或替换。

### 2.4 验收

- 复制账单不会修改原账单。
- 从模板生成账单后，统计只增加新账单，不统计模板。
- 删除模板不影响历史账单。

## 3. 周期账单待确认模块

### 3.1 目标

帮助用户不要漏记房租、水电、订阅等固定支出，同时避免自动入账造成错误。

### 3.2 范围

必须支持：

- 每周、每月、每年周期。
- 起始日期、下次到期日。
- 固定金额或待确认金额。
- 到期生成 pending instance。
- 用户确认后生成真实账单。
- 用户可跳过本期。

暂不支持：

- 系统推送通知。
- 自动扣账。
- 自动生成 settlement。

### 3.3 规则

- pending instance 不进入统计和结算。
- 确认后生成真实账单，并记录来源 recurring_rule_id。
- 跳过本期写审计日志。
- 删除周期规则不删除已经确认生成的历史账单。

### 3.4 验收

- 到期后能在设置页或 Dashboard 看到待确认项。
- 确认后流水出现真实账单。
- 跳过后本期不再提醒。

## 4. 分类、标签、账户管理模块

### 4.1 目标

让长期账本保持可维护，避免分类失控、标签混乱、账户重复。

### 4.2 范围

分类：

- 新增、编辑名称、图标、颜色、排序、归档、恢复。
- 区分支出分类和收入分类。

标签：

- 新增、编辑名称、颜色、归档、恢复。
- 账单表单支持搜索和多选。

账户：

- 新增、编辑名称、类型、排序、归档、恢复。
- 暂不做余额校准。

### 4.3 规则

- 已使用项不可物理删除，只能归档。
- 归档项不出现在新增账单默认选择器。
- 历史账单仍显示归档项名称。
- 名称在同一账本内不可重复。

### 4.4 验收

- owner 可管理全部元数据。
- editor 是否可管理由 Foundation RBAC 配置决定，默认不可。
- viewer 不可管理。
- 归档分类后，历史账单仍正常显示。

## 5. 结算可解释性模块

当前实现状态：已完成结算余额解释字段、结算页 paid/share/raw_net/settlement/final_net 展示、影响明细入口、复制结算文案、剪贴板失败手动复制兜底、登记结算和当期历史记录真实 UI 验收；验收证据见 `docs/project_analysis/v1.1-settlement-acceptance-2026-07-07/`。

### 5.1 目标

让用户清楚理解当前结算金额怎么来的，减少对系统计算的怀疑。

### 5.2 范围

必须展示：

- paid：成员实际支付。
- share：成员应承担。
- raw_net：支付减承担。
- settlement：已结算影响。
- final_net：最终净额。
- 影响结算的共同支出明细入口。
- 复制结算文案。

暂不支持：

- 自动通知共同支付。
- 支付状态跟踪。
- 自动催付。

### 5.3 规则

- 结算记录不进入消费统计。
- 复制文案不改变任何状态。
- 已删除共同支出不参与结算。

### 5.4 验收

- A 支付 200、B 支付 80、双方平均分摊时，系统能解释最终谁欠谁。
- 复制文案包含月份、付款人、收款人、金额。
- 结算后生成 settlement 记录，不修改历史共同支出。

## 6. 移动端高频路径模块

### 6.1 目标

保证手机端完成核心工作流：看首页、记一笔、筛流水、看结算、进设置。

### 6.2 范围

- 375px 宽度无横向滚动。
- 流水筛选使用底部 Sheet。
- 表单使用分组和折叠低频字段。
- 危险操作使用明确确认。
- 底部导航突出记账入口。

### 6.3 验收

- 375px、390px、430px 三个宽度下页面不重叠。
- 移动端可完成普通支出、共同支出、筛选、复制结算文案。
- 危险操作按钮不与主操作紧贴。
