# PRD：v1.1-v1.2 验收样例矩阵

状态：建议采纳  
适用阶段：v1.1 / v1.2 冻结前  
目的：把产品需求转成可复用的业务样例，供开发、测试、验收和 AI 编码任务引用。

## 1. 使用规则

1. 本文中的金额单位写作元，系统实现和 API 必须使用整数分。
2. 每个任务进入开发前，应从本文选择相关样例作为测试 fixture 或手工验收用例。
3. 如代码实现与本文样例冲突，优先暂停并更新 PRD，不允许在代码中隐式改变业务口径。
4. 本文不替代自动化测试，但应成为自动化测试的事实来源。

## 2. 基础角色与账本

默认验收数据：

| 对象 | 值 |
|---|---|
| 用户 A | lynn |
| 用户 B | polar |
| 默认账本 | 两人生活账本 |
| A 角色 | owner |
| B 角色 | editor |
| viewer 用户 | guest |
| 币种 | CNY |

权限基线：

- owner 可管理账本、成员、元数据、备份、导出。
- editor 可新增普通账单、共同支出、结算记录，但默认不可管理成员、导出、备份。
- viewer 只可查看授权数据，不可新增、编辑、删除。

## 3. 普通账单验收样例

### Case T01：普通支出创建

前置：

- A 登录。
- 当前账本为“两人生活账本”。
- 分类“餐饮”和账户“微信”存在且未归档。

操作：

- A 创建支出：餐饮，金额 35.80，付款人 A，可见性 private。

期望：

- API 写入 `amount_cents=3580`。
- 流水列表 A 可见。
- B 不可见。
- Dashboard 只在 A 的私有统计口径中体现。
- 修改金额或删除时写审计日志。

### Case T02：partner_readable 账单

操作：

- A 创建支出：药品，金额 68.00，可见性 partner_readable。

期望：

- A 可查看和编辑。
- B 可查看但不可编辑。
- B 导出时可见该记录，但不得获得 A 的 private 记录。
- 该账单不进入共同结算。

### Case T03：删除普通账单

操作：

- A 删除自己创建的普通账单。

期望：

- 账单状态变为 deleted，不物理删除。
- 列表、统计、导出默认排除。
- 审计日志记录 before_json。

## 4. 共同支出与结算验收样例

### Case S01：双人平均分摊

操作：

- A 支付晚餐 200.00，类型 shared_expense，A/B equal。

期望：

- transaction amount_cents = 20000。
- A split = 10000。
- B split = 10000。
- 当前结算结果：B 应转给 A 100.00。

### Case S02：双人多笔净额

操作：

1. A 支付晚餐 200.00，双方 equal。
2. B 支付超市 80.00，双方 equal。

期望：

- A paid = 20000，share = 14000，raw_net = 6000。
- B paid = 8000，share = 14000，raw_net = -6000。
- 当前结算结果：B 应转给 A 60.00。

### Case S03：生成结算记录

操作：

- B 向 A 结算 60.00。

期望：

- 新增 settlement 记录。
- 新增 settlement 类型流水或等价结算流水。
- 历史 shared_expense 不被修改。
- final_net 双方为 0。
- 结算记录不进入消费统计。

### Case S04：修改历史共同支出

操作：

- A 将第一笔晚餐从 200.00 改为 180.00。

期望：

- split 重新计算。
- 结算页重新展示 paid/share/raw_net/settlement/final_net。
- 审计日志记录修改前后。
- 已生成 settlement 记录不被改写。

### Case S05：删除共同支出

操作：

- A 删除第一笔共同支出。

期望：

- 该支出不再参与结算。
- 已有 settlement 保持历史记录。
- 结算页可能出现新的 final_net，必须解释原因。

## 5. 分类、标签、账户验收样例

### Case M01：归档已使用分类

操作：

- 分类“餐饮”已被历史账单使用。
- A 归档“餐饮”。

期望：

- 新增账单选择器不再默认展示“餐饮”。
- 历史账单仍显示“餐饮”名称。
- 不允许物理删除该分类。

### Case M02：归档账户

操作：

- 账户“微信”已被历史账单使用。
- A 归档“微信”。

期望：

- 新增账单默认账户不再选择“微信”。
- 历史账单仍显示“微信”。
- 模板引用该账户时，生成账单前必须提示替换。

### Case M03：同名元数据

操作：

- A 新增分类“餐饮”，当前账本已存在未归档“餐饮”。

期望：

- 创建失败。
- 错误码为 VALIDATION_ERROR 或等价业务错误。

## 6. 模板与周期账单验收样例

### Case P01：复制一笔

操作：

- A 复制一笔外卖支出 42.50。

期望：

- 新账单默认保留分类、账户、标签、付款人、可见性、备注。
- 发生时间默认当前时间。
- 原账单不改变。
- 保存前金额必须可见。

### Case P02：模板生成账单

操作：

- A 从房租模板生成本月账单。

期望：

- 模板不进入统计。
- 生成后的真实账单进入统计和流水。
- 删除模板不影响历史账单。

### Case P03：周期账单待确认

操作：

- 水电费周期规则到期。

期望：

- 生成 pending instance。
- pending 不进入统计和结算。
- 用户确认后才生成真实账单。
- 用户跳过后本期不再提醒。

## 7. 导入验收样例

### Case I01：CSV 预览不写库

操作：

- A 上传微信 CSV，包含 10 行。

期望：

- 生成预览批次。
- transactions 不新增。
- 每行展示解析状态、金额分、收支方向、推荐分类。

### Case I02：重复导入

操作：

1. A 导入同一份 CSV 并确认。
2. A 再次上传同一份 CSV。

期望：

- 第二次预览显示 duplicate。
- 默认跳过重复行。
- 确认后不新增重复账单。

### Case I03：疑似重复

操作：

- CSV 中有一行与历史账单同金额、同商户、时间相近，但无外部订单号。

期望：

- 标记 suspicious。
- 用户必须确认导入或跳过。
- 系统不得自动提交 suspicious 行。

### Case I04：导入事务回滚

操作：

- 批次中一行必需字段无效，用户未标记跳过仍确认导入。

期望：

- 整批失败。
- 不写入半批 transactions。
- import batch 显示 failed。
- 错误提示包含行号和原因。

## 8. 权限与隐私验收样例

### Case R01：viewer 禁止写入

操作：

- guest 以 viewer 身份尝试新增账单。

期望：

- 前端不显示新增入口。
- 后端返回 403。
- 不写入 transactions。

### Case R02：非成员访问账本

操作：

- 用户 C 携带不属于自己的 `X-Ledger-Id` 访问账单列表。

期望：

- 返回 403 或 404。
- 不泄露账本名称、成员、账单摘要。

### Case R03：private 附件

操作：

- A 的 private 账单包含附件。
- B 访问附件 URL。

期望：

- 返回 403 或 404。
- 不允许通过裸 `/uploads` 路径绕过权限。

## 9. 移动端验收样例

### Case U01：375px 记账

设备：

- 375px 宽度。

期望：

- 无横向滚动。
- 金额输入框默认聚焦。
- 低频字段折叠。
- 保存按钮不被键盘或底部导航遮挡。

### Case U02：移动端筛选

操作：

- 用户在流水页打开筛选。

期望：

- 使用 bottom sheet。
- 可设置月份、类型、分类、标签、成员、关键词。
- 应用筛选后列表刷新。
