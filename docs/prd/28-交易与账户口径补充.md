# PRD：交易与账户口径补充

状态：建议采纳  
适用阶段：v1.1 / v1.2 冻结前  
目的：明确退款、报销、转账、账户余额和统计口径，避免后续实现时把“记账系统”误扩展成复杂财务系统。

## 1. 总原则

LedgerTwo 的短中期目标是共同生活记账，不是资产负债管理系统。

因此：

1. 账户在 v1.1 只表示支付来源，不表示真实余额。
2. 退款和报销优先作为收入或负向调整口径处理，不进入复杂应收应付。
3. 账户间转账暂不作为 v1.1 必做能力。
4. 结算记录只表达共同账本成员之间的补款，不等同于银行转账流水。
5. 所有统计必须明确是否包含 settlement、refund、income。

## 2. 交易类型口径

| 类型 | v1.1 状态 | 说明 |
|---|---|---|
| expense | 必做 | 普通个人支出 |
| income | 保留/可用 | 普通收入、报销到账、退款到账可先用 income 表达 |
| shared_expense | 必做 | 共同支出，必须生成 transaction_splits |
| settlement | 必做 | 成员之间结算记录，不进入消费统计 |
| refund | 延后 | v1.1 不单独建类型，先用 income + 备注/标签承载 |
| transfer | 延后 | 账户间转账不进入 v1.1 |

## 3. 退款口径

### 3.1 v1.1 处理方式

退款到账时，用户可创建 income：

- 分类：退款。
- 标签：原商户、原订单，可选。
- 账户：到账账户。
- 备注：关联原账单信息。

### 3.2 暂不支持

- 自动关联原账单。
- 自动冲减原支出。
- 半退、多次退款、跨月退款的复杂统计。

### 3.3 统计口径

默认统计：

- 支出统计仍显示原支出。
- 收入统计显示退款收入。
- 净支出视图如后续实现，可用 `expense - income(refund)` 计算，但 v1.1 不强制。

## 4. 报销口径

报销到账与退款类似，v1.1 使用 income 表达：

- 分类：报销。
- 标签：公司、差旅、医疗等。
- 备注：关联原支出。

暂不支持：

- 报销审批状态。
- 应收报销。
- 自动匹配原始支出。

## 5. 账户口径

### 5.1 v1.1 账户定义

账户只表示支付来源或收款来源，例如：

- 现金。
- 微信。
- 支付宝。
- 银行卡。
- 信用卡。
- 其他。

### 5.2 v1.1 不做

- 账户余额。
- 余额校准。
- 信用卡账单日和还款日。
- 账户间转账。
- 资产负债表。

### 5.3 账户归档

- 已使用账户不能物理删除。
- 归档账户不出现在新增账单默认选择器。
- 历史账单仍展示归档账户名称。
- 模板或周期规则引用归档账户时，生成账单前必须提示替换或确认。

## 6. 结算口径

settlement 表示成员之间对共同支出的补款。

必须遵守：

- settlement 不进入消费统计。
- settlement 会影响共同账本 final_net。
- settlement 不修改历史 shared_expense。
- settlement 可以有对应 transaction 流水用于展示，但类型必须清晰标记为 settlement。

不得把 settlement 当作：

- 普通收入。
- 普通支出。
- 账户间转账。
- 自动支付状态。

## 7. 统计口径

### 7.1 支出统计

默认包含：

- expense。
- shared_expense 中当前用户承担份额，若统计视图支持个人承担口径。

默认排除：

- settlement。
- deleted。
- template。
- pending recurring instance。

### 7.2 收入统计

默认包含：

- income。

默认排除：

- settlement。
- refund 独立类型，除非后续正式引入。

### 7.3 共同结算统计

共同结算只使用：

- shared_expense 的 paid/share。
- settlement 的 settled_out/settled_in。

不使用：

- ordinary expense。
- income。
- private 账单。

## 8. 导入口径

v1.2 导入时：

- 微信/支付宝支出默认映射为 expense。
- 收入类记录默认映射为 income。
- 退款类记录在 v1.2 默认映射为 income + 分类“退款”。
- 转账类记录默认标记为 unknown 或 skipped，不自动导入。

用户可在预览页手工改成：

- expense。
- income。
- shared_expense。
- skipped。

不得自动导入为：

- settlement。
- transfer。
- refund 独立类型。

## 9. Open Questions

以下问题不阻塞 v1.1/v1.2，但需要在 v1.3 前重新评审：

1. 是否引入正式 refund 类型。
2. 是否引入账户间 transfer。
3. 是否支持信用卡还款和账单日。
4. 是否支持净支出统计。
5. 是否允许 shared_expense 的退款参与结算重算。

## 10. 冻结标准

v1.1 冻结前：

- 文档、UI、API 不得暗示账户余额准确。
- 退款/报销用 income 处理的说明必须在 UI 或帮助文案中可被用户理解。
- settlement 从消费统计中排除。

v1.2 冻结前：

- 导入预览能识别或提示转账/退款/收入记录。
- 转账记录默认不自动入账。
- 退款记录默认按 income + 退款分类处理。
