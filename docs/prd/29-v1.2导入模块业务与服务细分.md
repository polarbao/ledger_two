## 1. 文档目的

本文把 v1.2 的账单导入能力拆成模块、业务对象、服务边界、API 切片、UI 工作台和验收任务。Task47-Task49 已完成 CSV 主链路；Task49X 在同一边界内增加微信 XLSX reader，并冻结支付宝官方 CSV 字段，专项规格见 `docs/prd/30-v1.2微信XLSX与支付宝CSV导入专项.md`。

v1.2 的核心原则：

1. 导入先预览，用户确认前不写正式账单。
2. 去重和规则只降低人工成本，不替用户做不可逆决策。
3. 所有正式落库仍复用 transaction service 的金额、权限、分类、账户、标签和可见性校验。
4. 导入模块是账本内能力，必须强制 `X-Ledger-Id`；Owner/Editor 只能操作本人创建的批次，Viewer 不可导入。
5. 微信支持 CSV/XLSX，支付宝和通用模板支持 CSV；支付宝 XLSX、OCR、银行同步、xls/xlsm、PDF 不进入 v1.2。

## 2. 版本范围

### 2.1 必做范围

| 模块 | 对应任务 | 业务目标 | 技术目标 | 用户可见结果 |
|---|---|---|---|---|
| CSV 导入预览 | Task47 | 上传文件后看见可调整的预览批次 | parser、normalizer、batch、row 基础落地 | 用户能看到行级状态和错误原因 |
| 去重与事务落库 | Task48 | 避免重复账单和半批写入 | import_hash、dedupe、commit 事务和审计 | 用户能确认导入并看到结果摘要 |
| 导入规则 | Task49 | 让常见商户自动给出分类建议 | rule service、命中解释和归档规则 | 用户能管理规则并看见命中来源 |

### 2.2 非目标

1. 不做 OCR、小票图片、PDF、xls/xlsm 和任意通用 XLSX 映射。
2. 不做银行同步或自动读取第三方账号。
3. 不自动提交 suspicious 行。
4. 不自动把转账、红包、理财、信用卡还款识别成正式账单。
5. 不实现后台异步任务队列；大文件先用单批行数限制解决。
6. 不开放 Viewer 导入；Owner/Editor 可导入本人账单，但账本级规则管理和商户学习仍仅 Owner 可用。

## 3. 业务对象

### 3.1 ImportBatch

导入批次，表示一次上传和预览结果。

| 字段类别 | 字段 | 说明 |
|---|---|---|
| 身份 | id、ledger_id、created_by_user_id | 归属账本和创建者 |
| 来源 | source_type、file_format、filename、file_sha256、parser_metadata_json | 来源类型、容器格式、文件指纹和非敏感解析元数据 |
| 状态 | status、expires_at、committed_at | previewing、ready、committed、failed、expired |
| 统计 | total_rows、new_rows、duplicate_rows、suspicious_rows、invalid_rows、imported_rows、skipped_rows、failed_rows | 预览和提交统计 |
| 时间 | created_at、updated_at | 批次生命周期 |

业务规则：

1. `ready` 之前不可提交。
2. `committed` 后不可修改行。
3. `expired` 后不可提交，只能重新上传。
4. 同一文件重复上传可以提示，但不得仅靠 `file_sha256` 判定正式账单重复。

### 3.2 ImportRow

导入行，表示 CSV 中一行标准化后的候选账单。

| 字段类别 | 字段 | 说明 |
|---|---|---|
| 定位 | id、batch_id、ledger_id、row_number | 行归属和原始行号 |
| 原文 | raw_json、normalized_json | 原始值和标准化值 |
| 标准字段 | occurred_at、title、merchant、description、amount_cents、direction、source_account | 预览展示和导入基础 |
| 目标字段 | target_transaction_type、visibility、selected_category_id、selected_account_id、selected_tag_ids_json | 用户确认后的写入目标 |
| 推荐字段 | suggested_category_id、suggested_account_id、suggested_tag_ids_json、matched_rule_id | 规则推荐，不等同最终选择 |
| 去重 | import_hash、duplicate_status | new、duplicate、suspicious、invalid |
| 状态 | row_status、error_code、error_message、generated_transaction_id | pending、adjusted、skipped、imported、failed |

业务规则：

1. `invalid` 行不得进入提交队列。
2. `duplicate` 行默认 `skipped`，用户可查看原因。
3. `suspicious` 行必须明确选择导入或跳过。
4. 用户手工调整后，规则不得再次覆盖 `selected_*` 字段。
5. `amount_cents` 必须为整数分，禁止使用 float。

### 3.3 ImportRule

导入规则，表示从商户、描述、来源账户或金额区间推导分类、标签、账户和可见性的推荐策略。

| 字段类别 | 字段 | 说明 |
|---|---|---|
| 身份 | id、ledger_id、created_by_user_id | 账本内规则 |
| 匹配 | match_type、pattern、amount_min_cents、amount_max_cents | merchant_contains、description_contains、source_account、amount_range |
| 结果 | result_json | category_id、account_id、tag_ids、visibility |
| 控制 | priority、status | active、archived |
| 时间 | created_at、updated_at | 规则生命周期 |

业务规则：

1. 规则只产生建议，不直接写正式账单。
2. archived 规则不参与匹配。
3. 引用已归档元数据时不得静默应用，必须给出提示或要求用户替换。
4. 规则优先级必须在 API 和 UI 中使用同一方向，建议数字越小优先级越高。

### 3.4 ImportSuggestion

规则命中结果，挂在预览行上。

| 字段 | 说明 |
|---|---|
| rule_id | 命中的规则 |
| reason | 用户可读命中原因 |
| suggested_category_id | 推荐分类 |
| suggested_account_id | 推荐账户 |
| suggested_tag_ids | 推荐标签 |
| blocked_reason | 元数据归档或权限问题 |

## 4. 服务边界

后端建议新建 `internal/importer`，旧 `/api/transactions/import/*` 只做 transitional 兼容委托。

```text
http handler
-> importer service facade
   -> parser/wechat | parser/alipay | parser/generic
   -> normalizer
   -> batch service
   -> row service
   -> dedupe service
   -> rule service
   -> commit service
      -> transaction service
      -> metadata repository
      -> audit log
```

### 4.1 Parser

职责：

1. 通过 CSV/XLSX tabular reader 读取表格行；XLSX 负责格式安全、工作表选择和表头定位。
2. 按 source_type 提取原始字段。
3. 返回带 `row_number` 的 raw row，不处理正式业务落库。

边界：

- Reader 和 Parser 都不访问数据库。
- Parser 不生成正式交易。
- Parser 不做权限判断。

验收：

- 微信 CSV/XLSX、支付宝 CSV 和通用 CSV 至少各 1 个 fixture；支付宝 XLSX 使用拒绝用例覆盖。
- 编码不支持、表头缺失、行列数不一致时返回可行动错误。

### 4.2 Normalizer

职责：

1. 把来源字段映射成标准字段。
2. 金额转整数分。
3. 时间转 ISO8601。
4. 将支出、收入、退款、转账、未知映射为标准 direction。

边界：

- Normalizer 不判断重复。
- Normalizer 不应用导入规则。
- 转账和未知类型默认不自动入账。

验收：

- 金额不能使用 float 作为最终计算依据。
- 时间无法解析时生成行级 `IMPORT_ROW_INVALID`。
- 退款口径符合 `docs/prd/28-交易与账户口径补充.md`。

### 4.3 Batch Service

职责：

1. 创建导入批次。
2. 维护批次状态和统计。
3. 查询批次详情。
4. 控制过期和不可提交状态。

边界：

- 必须强制 ledger_id。
- 必须校验 Owner/Editor 角色与 `created_by_user_id == actor_user_id`。
- 不直接写 transactions。

验收：

- `POST /api/imports/preview` 成功后生成 `ready` 批次。
- 用户确认前 transactions 数量不变。
- Viewer 请求返回 `FORBIDDEN`；其他成员访问非本人批次统一返回 `LEDGER_OBJECT_NOT_FOUND`。

### 4.4 Row Service

职责：

1. 写入预览行。
2. 更新用户调整字段。
3. 标记跳过、确认 suspicious 或修正 invalid。
4. 返回移动端友好的行 DTO。

边界：

- 不直接提交正式账单。
- 不允许修改 committed 批次。
- selected 字段必须重新校验分类、账户、标签和可见性。

验收：

- `PATCH /api/imports/{batch_id}/rows/{row_id}` 可调整分类、账户、标签、可见性和跳过状态。
- 手工调整后不会被规则覆盖。

### 4.5 Dedupe Service

职责：

1. 生成稳定 `import_hash`。
2. 判断 new、duplicate、suspicious、invalid。
3. 解释重复或疑似重复原因。

边界：

- 不决定是否写入 suspicious 行。
- 不只依赖文件 hash。
- 不跨账本比较重复。

验收：

- 同订单号或同 import_hash 识别为 duplicate。
- 同商户、同金额、近时间但无订单号识别为 suspicious。
- duplicate 默认跳过。

### 4.6 Rule Service

职责：

1. 管理规则 CRUD、归档、恢复。
2. 根据预览行产生建议。
3. 解释命中原因和阻断原因。

边界：

- 规则不直接提交账单。
- archived 规则不参与匹配。
- 引用归档元数据时只提示，不静默应用。

验收：

- 禁用或归档规则后不再命中。
- 用户手工修改优先级高于规则。
- 规则管理仅 owner 可用。

### 4.7 Commit Service

职责：

1. 校验批次状态、行状态和权限。
2. 在单个事务内创建正式账单。
3. 写入 generated_transaction_id 和批次统计。
4. 写审计日志。

边界：

- 创建账单必须调用 transaction service 或等价业务服务，不绕过 service 层。
- 任一必需行失败时整批回滚。
- 不修改历史 shared expense 或 settlement 记录。

验收：

- invalid 行未修复时不可提交。
- suspicious 行未确认时不可提交。
- duplicate 和 skipped 行不写 transactions。
- 成功提交后批次状态为 `committed`。

## 5. API 切片

### 5.1 Task47 API

```text
POST /api/imports/preview
GET  /api/imports/{batch_id}
PATCH /api/imports/{batch_id}/rows/{row_id}
```

要求：

1. 全部要求登录和 `X-Ledger-Id`；批次 API 允许 Owner/Editor 操作本人批次，规则 API 仅 Owner 可用。
2. preview 使用 `multipart/form-data`。
3. batch detail 返回行列表、统计、字段映射和可行动错误。
4. row update 支持 selected 字段和 row_status 调整。

### 5.2 Task48 API

```text
POST /api/imports/{batch_id}/commit
GET  /api/imports/{batch_id}
```

要求：

1. commit 前返回导入、跳过、疑似、错误数量。
2. commit 成功返回 generated_transaction_ids。
3. commit 失败不得写半批数据。
4. 已提交批次重复提交返回 `IMPORT_COMMIT_CONFLICT`。

### 5.3 Task49 API

```text
GET   /api/import-rules
POST  /api/import-rules
PATCH /api/import-rules/{id}
POST  /api/import-rules/{id}/archive
POST  /api/import-rules/{id}/restore
```

要求：

1. `DELETE /api/import-rules/{id}` 仅保留历史兼容，正式行为使用 archive。
2. 规则列表支持 status、keyword、match_type 筛选。
3. 规则变更写审计日志。
4. OpenAPI 和 API Inventory 必须同步更新。

## 6. UI 工作台拆分

### 6.1 页面与组件

| UI 单元 | 目标 | 关键状态 |
|---|---|---|
| ImportEntryPage | 设置页进入导入工作区 | Owner/Editor 可见，Viewer 显示只读无权限状态 |
| SourceStep | 选择微信、支付宝、通用 CSV | source_type |
| UploadStep | 上传账单文件 | 上传中、格式错误、来源不匹配、成功 |
| MappingStep | 字段映射 | 自动识别、手工修正、缺必填字段 |
| PreviewWorkbench | 预览批次和统计 | new、duplicate、suspicious、invalid |
| ImportRowCard / RowTable | 行级预览 | 移动端卡片，桌面表格 |
| RowEditorDrawer | 调整分类、账户、标签、可见性和跳过状态 | adjusted、skipped、blocked |
| CommitConfirmModal | 二次确认提交 | 导入/跳过/疑似/错误数量 |
| ImportResultSummary | 提交结果 | imported、skipped、failed |
| ImportRuleManagePage | 管理规则 | active、archived、priority |

### 6.2 移动端约束

1. 375px 不允许横向滚动。
2. 预览表格在移动端必须切换为卡片。
3. 行详情使用 drawer 或独立详情页，不在卡片中塞满所有字段。
4. invalid 和 suspicious 必须有明显状态，不只依赖颜色。
5. 确认导入按钮附近必须展示本次会写入多少行。

## 7. 权限矩阵

| 角色 | 进入导入页 | 预览 | 调整行 | 提交 | 管理规则 | 查看历史批次 |
|---|---:|---:|---:|---:|---:|---:|
| owner | yes | yes | yes | yes | yes | yes |
| editor | no | no | no | no | no | no |
| viewer | no | no | no | no | no | no |
| 非成员 | no | no | no | no | no | no |

说明：

1. v1.2 MVP 不开放 editor 导入，避免批量写入风险。
2. 前端隐藏入口只是体验，后端必须强制拒绝。
3. 批次存在性不得向非成员泄露。

## 8. 降级与开关

建议至少预留以下服务开关或配置常量：

| 开关 | 默认 | 作用 |
|---|---|---|
| import.preview.enabled | true | 关闭后禁止新建预览，历史批次可只读 |
| import.commit.enabled | false in early dev, true after Task48 | 控制正式落库入口 |
| import.rules.enabled | false before Task49 | 控制规则推荐和管理入口 |
| import.source.wechat.enabled | true | 单独关闭微信 parser |
| import.source.alipay.enabled | true | 单独关闭支付宝 parser |
| import.max_rows_per_batch | 2000 | 覆盖已验证的 1232 行支付宝官方 CSV，仍受 2 MiB 文件上限约束 |
| import.format.xlsx.enabled | false before Task49X acceptance | 独立关闭 XLSX，不影响 CSV |
| import.preview_ttl_days | 7 | 预览批次过期时间 |

降级规则：

1. parser 有问题时关闭对应 source_type，不影响 generic。
2. commit 有问题时关闭 commit，保留 preview 只读。
3. 规则误伤时关闭 rules，不影响人工调整。

## 10. Task47-Task49 冻结标准

### 10.1 Task47 可冻结

1. `/api/imports/preview`、`GET /api/imports/{batch_id}`、行更新接口可用。
2. 微信、支付宝、通用 CSV fixture 通过。
3. invalid 行展示行号、错误码和可行动原因。
4. preview 不写正式 transactions。
5. 前端导入工作台能展示预览批次和移动端卡片。

### 10.2 Task48 可冻结

1. `import_hash` 方案定案并有唯一约束或等价映射。
2. duplicate 默认跳过。
3. suspicious 未确认时阻止提交。
4. commit 单事务落库，失败回滚。
5. 提交写审计日志。
6. 结果页能回看 imported、skipped、failed。

### 10.3 Task49 可冻结

1. 规则 CRUD、归档、恢复完成。
2. 预览行显示规则命中来源。
3. 用户手工调整不被规则覆盖。
4. 规则引用归档元数据时阻止静默应用。
5. 规则管理权限和审计日志完成。

## 11. 开发前检查清单

Task47 开工前必须确认：

1. v1.1 已冻结，或剩余问题不影响导入能力。
2. OpenAPI 将随新增 API 更新。
3. 新 migration 只追加，不修改历史 migration。
4. fixture 文件不包含真实个人账单隐私。
5. 前端入口默认只对 owner 显示。
6. commit 开关在 Task47 阶段保持关闭或不可见。

Task48 开工前必须确认：

1. import_hash 存储方案已在 migration 和文档中定案。
2. commit 使用 transaction service 校验链路。
3. rollback 测试可自动化执行。

Task49 开工前必须确认：

1. 分类、标签、账户归档规则已经稳定。
2. import_rules 的优先级方向在后端、前端和文档一致。
3. 规则只推荐、不自动提交的原则没有被 UI 文案弱化。
