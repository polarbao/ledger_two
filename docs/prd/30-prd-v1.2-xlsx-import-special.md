## 1. 决策摘要

当前真实导出格式为：微信可交付 CSV 或 Excel `.xlsx`，支付宝仍交付 CSV。v1.2 增加 Task49X 专项，在不改变“预览、人工确认、事务提交”主链路的前提下，为微信增加 XLSX 支持，并冻结支付宝 CSV 的当前官方字段。

本专项不是新增一套导入系统，而是在现有 Task47-Task49 导入管线前增加受控的工作簿读取层：

```text
微信 CSV/XLSX 或支付宝/通用 CSV 文件
-> 文件格式与安全校验
-> 表格行读取和表头定位
-> 现有来源字段标准化
-> 现有规则推荐、去重、预览、人工调整
-> 现有事务提交与审计
```

本机 `1.2.0-rc/schema 19` 已支持受控 XLSX preview；NAS 当前部署仍不得宣称支持 XLSX。任何真实文件均不得绕过预览和用户确认直接提交到 production。

## 2. 用户问题

### 2.1 核心用户故事

作为账本 Owner，我希望直接选择微信导出的 `.xlsx` 或 CSV，以及支付宝导出的 CSV，先查看系统识别出的流水数量、异常行和重复状态；微信 XLSX 还应展示工作表与表头位置，再决定哪些账单写入正式账本。

### 2.2 当前阻力

1. 微信用户可能获得 XLSX，原页面只接受 `.csv`。
2. 手工转换可能破坏日期、金额小数、长订单号和前导零。
3. 微信账单可能在文件前部包含说明行，表头不一定是第一行。
4. 直接把 XLSX 当 CSV 上传会解析失败，且错误信息不能指导用户处理。
5. 真实账务数据不能通过未经测试的临时脚本直接写入数据库。

## 3. 目标与成功指标

### 3.1 必须达成

1. 微信来源支持 `.csv` 和 `.xlsx`；支付宝与通用模板 v1.2 只支持 `.csv`。
2. XLSX 上传后沿用现有预览批次，不直接创建 `transactions`。
3. 自动定位非首行表头，并展示识别到的工作表与表头行。
4. 金额仍以整数分进入业务层，禁止 float 作为最终金额依据。
5. 日期、订单号、退款、转账和收支方向与 CSV 口径一致。
6. 微信 XLSX 和对应 CSV 表达同一流水时，标准化结果与 `import_hash` 一致。
7. 重复、疑似重复、无效行、规则推荐和事务提交复用现有能力。
8. 文件扩展名、文件内容或工作簿结构不匹配时，在预览前拒绝并给出可行动原因。

### 3.2 发布指标

| 指标 | 冻结门槛 |
|---|---:|
| 微信 XLSX fixture 标准字段一致率 | 100% |
| 支付宝当前官方 CSV 样本字段一致率 | 100% |
| 微信 XLSX/CSV 等价 fixture 的 hash 一致率 | 100% |
| 金额分转换错误 | 0 |

| 超限、加密、伪装文件拒绝率 | 100% |

## 4. 范围

### 4.1 支持矩阵

| source_type | CSV | XLSX | 说明 |
|---|---:|---:|---|
| wechat | 支持 | 本专项新增 | 支持说明行后出现表头 |
| alipay | 支持 | 不支持 | 当前官方导出仍为 CSV；未来格式变化需重新评审 |
| generic | 支持 | 暂不支持 | 避免本专项扩成任意 Excel 映射器 |

### 4.2 微信 XLSX 支持边界

1. 只支持标准 Office Open XML `.xlsx`。
2. 单文件上传上限保持 2 MiB。
3. 单批最多 2000 条数据行。该值覆盖已验证的 1232 行支付宝官方 CSV，同时保留 2 MiB 文件上限。
4. 最多检查 10 个工作表、每个工作表前 50 行和前 64 列以定位表头。
5. 只从可见工作表中选择来源表头完整匹配的唯一工作表。
6. 原文件只在请求内解析，不长期保存，不写审计原文。

### 4.3 非目标

1. 不支持 `.xls`、`.xlsm`、`.xlsb`、Numbers、ODS、PDF 和图片。
2. 不支持带密码或加密工作簿。
3. 不执行宏、公式、外部链接或嵌入对象。
4. 不提供任意工作表、任意列的通用可视化映射器。
5. 不引入后台异步队列。
6. 不自动提交任何 XLSX 行。
7. 不在本专项中提高 owner-only 权限范围。
8. 不把真实微信或支付宝文件提交到 Git 仓库。
9. 不推测或预实现支付宝 XLSX；支付宝官方导出格式变化时重新打开 PRD、fixture 与真实样本评审。

## 5. 业务规则

### 5.1 文件识别

1. 浏览器扩展名只用于选择提示，后端必须独立校验。
2. `.xlsx` 必须同时满足 ZIP magic 和 OOXML 核心部件存在。
3. 扩展名与内容不一致返回 `IMPORT_FILE_FORMAT_MISMATCH`。
4. `.xls`、`.xlsm` 或含 VBA 部件的文件返回 `IMPORT_FILE_UNSUPPORTED`。
5. source_type 与识别到的表头不匹配时返回 `IMPORT_SOURCE_MISMATCH`。
6. `alipay + xlsx` 和 `generic + xlsx` 均返回 `IMPORT_FILE_UNSUPPORTED`，不得进入工作簿解析。

### 5.2 工作表与表头

1. 对用户选择的 source_type 使用固定的来源表头字典，不根据内容猜测来源。
2. 在可见工作表中扫描表头；完整匹配且只有一个候选时自动选择。
3. 没有候选时说明缺失字段；多个候选时拒绝并说明候选工作表名称。
4. 微信 XLSX 允许表头位于第 1-50 行；真实待导入样本的表头位于第 18 行。
5. `row_number` 使用工作表物理行号，例如表头为 18 时首条流水为 19。
6. 空白尾行不计入批次总数。

### 5.3 单元格处理

1. 金额读取为显示文本或精确十进制文本，再复用现有元转分逻辑。
2. 金额最多两位小数；千分位、人民币符号和“元”后缀按现有规则清理。
3. 日期支持来源导出的文本日期和合法 Excel 日期单元格，统一输出 `+08:00` 的 RFC3339。
4. 订单号按文本处理，保留前导零；科学计数法或无法保证精度的数字订单号标记为无效。
5. 必需字段中的公式单元格不采信缓存结果，返回行级 `IMPORT_ROW_FORMULA_UNSUPPORTED`。
6. 合并单元格或隐藏行进入已识别数据区时拒绝文件，避免遗漏或重复解释。

### 5.4 预览与提交

1. 预览页展示文件格式、工作表名称、表头行和数据行数。
2. XLSX 与 CSV 使用同一套 duplicate/suspicious/invalid 状态。
3. 用户手工调整继续优先于导入规则。
4. duplicate 默认跳过；suspicious 必须明确导入或跳过。
5. commit 仍在单个数据库事务内完成，XLSX 不增加绕过入口。

## 6. UI/UX 需求

### 6.1 上传区

1. 来源选项改为“微信账单”“支付宝账单”“通用模板”。
2. 微信说明为“支持 CSV、XLSX”；支付宝和通用模板说明为“仅 CSV”。
3. 微信文件选择器在运行门禁开启时接受 `.csv,.xlsx`；支付宝和通用模板始终只接受 `.csv`。
4. 当运行环境关闭 XLSX 时，微信文件选择器只接受 CSV，并展示“当前环境微信账单仅开放 CSV”。
5. 上传按钮统一使用“选择账单文件”，不再写死“选择 CSV 文件”。
6. 展示 2 MiB、2000 行限制和“不直接写入正式账单”的关键提示。

### 6.2 预览区

新增只读信息：

- `file_format`：CSV 或 XLSX。
- `source_sheet`：XLSX 工作表名称，CSV 不展示。
- `header_row_number`：XLSX 表头行，CSV 默认为 1 且不强调。
- `parsed_rows`：实际识别的数据行数量。

### 6.3 错误文案

| 场景 | 用户文案方向 |
|---|---|
| 扩展名与内容不符 | 文件后缀与实际格式不一致，请重新导出原始账单 |
| 工作簿加密 | 暂不支持加密 Excel，请导出未加密的 `.xlsx` |
| 找不到表头 | 未找到所选来源的完整账单表头，并列出缺失字段 |
| 多工作表匹配 | 检测到多个账单工作表，请只保留一个后重试 |
| 文件或解压后内容超限 | 文件内容过大，请按时间范围拆分后导入 |
| 公式或合并单元格 | 账单数据区包含不受支持的 Excel 结构，请重新导出原始账单 |

## 7. 权限、隐私与数据保留

1. 仅当前账本 owner 可预览、调整和提交。
2. 后端不得记录绝对路径、完整原始行或完整订单号到普通日志。
3. `file_sha256` 可保留用于同文件提示；不得保存原工作簿字节。
4. `parser_metadata_json` 只允许保存格式、工作表、表头行、行列统计和解析器版本，不保存账单明文。
5. 真实文件只用于 staging 受控验收，production 提交必须再次获得用户确认。

## 8. 验收场景

详细用例见 `docs/prd/27-acceptance-case-matrix.md` 的 I05-I10，最低覆盖：

1. 微信表头位于第 18 行仍生成正确预览。
2. 支付宝当前官方 CSV 的金额、日期、交易号和 GB18030 解码与真实样本一致。
3. 伪装、超限、加密、宏和多候选工作表被拒绝。
4. 日期、金额、订单号和物理行号保持正确。
5. 预览前后 transactions 数量不变。
6. 同一流水先通过 XLSX 提交，再上传 CSV 时识别为 duplicate。

## 9. 发布与降级

1. 新增 `import.format.xlsx.enabled`，仅控制微信 XLSX；默认在开发环境开启，staging 验收后再在 production 开启。
2. XLSX 解析故障时只关闭该开关，CSV、历史批次、规则和 commit 保持可用。
3. 数据库 migration 只追加；回退应用版本时新增字段由旧版本忽略。
4. Task49X 未完成前，RC05 保持“禁止真实 XLSX 提交”。
