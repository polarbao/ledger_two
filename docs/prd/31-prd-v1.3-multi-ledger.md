# PRD：v1.3 Task50 多账本正式化

版本：v1.0-frozen

状态：产品范围与全部开发前契约已冻结；Task50.1 获准在独立 development 环境启动

冻结日期：2026-07-15

对应任务：Task50P.2；后续实现 Task50.1-Task50.6
上游事实：`docs/project_analysis/2026-07-15-task50-current-state-gap-matrix.md`

下游技术事实：`docs/tech/25-v1.3-multi-ledger-implementation-contract.md`

## 1. 背景

LedgerTwo 已有账本、成员关系、角色、账本选择器和大部分 `ledger_id` 隔离基础，但这些能力仍带有单账本兼容逻辑，缺少生命周期、最后 Owner、成员离开、失效回退和全模块隔离闭环。

Task50 的目标不是把双人记账扩成聚会分账，而是让一个账号可以可信地管理多个相互独立的双人账本，例如“共同生活”“旅行专项”“装修专项”。

## 2. 产品目标

1. 用户可以创建、切换、重命名、归档和恢复多个独立账本。
2. 每个账本继续服务一人或两人，Owner/Editor/Viewer 权限清楚且后端强制。
3. 所有账单、结算、元数据、导入、报表、附件和导出均不会跨账本串读串写。
4. 当前账本被归档或用户失去成员资格时，界面有确定、可解释的回退，不静默落到任意账本。
5. 升级与回滚不丢失 v1.2 原有账本、成员和业务数据。

## 3. 非目标

Task50 明确不实现：

1. 第三名及以上活跃成员。
2. ratio/shares/custom 多人分摊和最小转账算法。
3. 公开邀请链接、邮件、短信、微信或应用内通知。
4. 永久删除账本和级联删除历史数据。
5. 单账本物理备份、账本合并、跨账本搬移或复制账单。
6. 银行同步、OCR、原生 App、企业多租户和复杂预算。
7. 修改现有金额、结算、导入 parser/hash/事务或消费统计口径。

## 4. 目标用户与核心场景

### 4.1 一个人维护多个用途账本

用户创建“共同生活”和“装修”两个账本，账单、分类、账户、导入规则和报表完全独立。切换后页面只能显示目标账本数据。

### 4.2 两人长期共用一个账本

Owner 将已存在账号加入为 Editor 或 Viewer。Editor 可参与日常记账和结算；Viewer 只查看。账本仍最多两名活跃成员。

### 4.3 项目结束后保留历史

旅行或装修结束后，Owner 归档账本。账本默认从日常选择列表隐藏，仍可查看历史和导出；恢复后才能继续写入。

### 4.4 关系或职责变化

Owner 先把所有权原子移交给另一名成员，再作为 Editor 留下或自行离开。系统始终保证至少一个 Owner。

## 5. 核心概念

| 概念 | 定义 |
|---|---|
| 账本 | 数据隔离和权限判断的第一边界 |
| 活跃账本 | 可读可写，写权限继续由成员角色和业务对象规则决定 |
| 归档账本 | 默认隐藏、全员只读、可查看/报表/按权限导出，Owner 可恢复 |
| 当前账本 | 用户当前界面和 API 请求显式选择的账本，不由后端猜测 |
| Owner | 账本生命周期和成员关系负责人 |
| Editor | 日常协作记账者，不管理账本生命周期、成员、元数据、导入和整库运维 |
| Viewer | 只读成员 |
| 实例管理员 | NAS/本机 LedgerTwo 实例的运维负责人，独立于任何账本角色，负责整库备份、恢复准备和诊断 |

## 6. 冻结产品规则

### 6.1 成员上限

1. 每个账本最多两名活跃成员，包含 Owner。
2. 新建账本初始只有创建者一人，创建者为 Owner。
3. 达到两人后不可再添加成员；API 返回稳定业务错误，不依赖前端隐藏。
4. 升级前若发现历史账本超过两名成员，不自动删除任何关系。该账本保持可读并阻止新增成员，发布门禁标记为需人工处理；处理过程必须有备份和审计。
5. 第三名及以上成员只可由 Task51 重新冻结可见性、分摊和结算规则后开放。

### 6.2 角色权限

| 能力 | Owner | Editor | Viewer |
|---|:---:|:---:|:---:|
| 查看账本与成员 | 是 | 是 | 是 |
| 创建/编辑/软删除自己的普通账单 | 是 | 是 | 否 |
| 创建共同支出与登记结算 | 是 | 是 | 否 |
| 查看报表与历史 | 是 | 是 | 是 |
| 导出当前角色可见 CSV/JSON | 是 | 是 | 否 |
| 管理分类、标签、账户 | 是 | 否 | 否 |
| 导入 preview/行调整/commit/规则 | 是 | 否 | 否 |
| 创建新账本 | 是，作为当前账号全局能力 | 是，作为当前账号全局能力 | 是，作为当前账号全局能力 |
| 重命名、归档、恢复账本 | 是 | 否 | 否 |
| 添加/移除成员、调整角色、移交 Owner | 是 | 否 | 否 |
| 整库备份、恢复准备、系统诊断 | 否，除非同一账号另具实例管理员身份 | 否 | 否 |

补充规则：

1. Task50 保留已验收的 Owner/Editor 可见数据导出行为，并要求 RolePolicy、OpenAPI、UI 和后端测试统一。
2. Task50 不改变交易对象级规则：普通账单仍只允许业务所有者/创建者按既有规则编辑，Editor 不是“可编辑所有人的账单”。
3. 后端是最终权限边界；UI 隐藏或禁用不能替代 403 校验。
4. 账本 Owner 不天然获得整库运维权限。Task50 必须把实例管理员与 ledger_members 角色分离；实例管理员的存储、初始化和部署配置由 P.3 冻结，但不得再以“任一账本 Owner”代替。

### 6.3 Owner 与成员不变量

1. 每个账本任何时刻至少一名且最多一名 Owner。
2. 通用“修改角色”接口不能把 Owner 降级，也不能把成员直接升级为 Owner。
3. Owner 移交必须使用独立原子操作：目标成员升为 Owner，原 Owner 同时降为 Editor；任一步失败则全部回滚。
4. Owner 不能通过通用移除接口移除自己。
5. Editor/Viewer 可以主动离开；离开后立即失去账本和历史数据访问权。
6. Owner 只有在完成所有权移交后才能主动离开。
7. Owner 可移除 Editor/Viewer。移除不删除该成员创建、拥有、支付或参与的历史账单，不重写结算和审计。
8. 所有添加、移除、角色变化、所有权移交和主动离开写审计日志。
9. Task50 不提供账号硬删除或停用 UI。未来账号停用/删除前必须先完成其全部 Owner 移交；存在唯一 Owner 关系时操作必须被阻止。

### 6.4 添加成员与历史可见性

1. Task50 只支持按精确 username 添加已存在的本地账号，不提供邀请注册或通知。
2. 添加前必须二次确认，并明确：新成员会按照交易可见性规则看到该账本的历史数据。
3. `private` 历史账单仅业务所有者及既有授权路径可见。
4. `partner_readable` 对当前另一名成员可读但不可由其编辑；成员替换后，新成员可读取历史 `partner_readable`，这是该可见性的冻结语义。
5. `shared` 对当前账本成员可见，编辑权限继续遵守创建者和既有共同账单规则。
6. 一人账本中的 `partner_readable` 暂时只有本人可见；第二名成员加入后可读取既有 `partner_readable`。

### 6.5 账本生命周期

状态只允许：

```text
active -> archived -> active
```

规则：

1. 只有 Owner 可归档和恢复。
2. 不提供永久删除。
3. 归档账本默认不出现在日常账本选择列表，可在“账本管理 > 已归档”查看。
4. 归档账本允许查看成员、历史流水、附件、结算和报表；Owner/Editor 可按各自可见范围导出。
5. 归档账本禁止任何业务写入，包括账单、共同支出、结算、元数据、模板、周期确认、附件上传、导入 preview/行调整/commit/规则和成员变化。
6. 归档账本不可重命名；需先恢复。
7. 存在未结清净额不阻止归档，但确认框必须展示未结清金额并说明不会自动生成结算。
8. 存在 `ready` 导入预览批次时阻止归档，用户需先完成或明确放弃该批次。Task50 技术契约必须提供可执行的放弃/过期规则后才能编码。
9. 周期提醒不阻止归档；归档期间不允许确认生成账单，恢复后按既有周期规则处理。
10. 归档和恢复均写审计，记录 actor、时间、归档前状态和确认时的未结清摘要。

### 6.6 当前账本与回退

1. 所有账本内业务请求必须显式携带账本 ID；后端不得用 `LIMIT 1` 猜测当前账本。
2. 登录、健康检查、账本列表和创建账本等全局请求可不携带账本 ID。
3. 前端可持久化最近使用账本，但它只是偏好，不是权限事实；每次启动都要用最新账本列表校验。
4. 当前账本仍 active 且用户仍是成员时，继续使用。
5. 当前账本被归档、用户被移除或偏好无效时，优先选择“最近使用且仍可访问的 active 账本”。
6. 没有可访问 active 账本时进入明确的“暂无活跃账本”状态，不自动选择归档账本，不发送无账本业务请求。
7. 无账本状态提供“创建账本”和“查看已归档账本”；创建成功后新账本成为当前账本。
8. 回退发生时显示一次可解释提示，例如“原账本已归档，已切换到共同生活”，不能静默切换。
9. 切换账本必须隔离进行中请求和缓存；旧账本晚返回的数据不得覆盖新账本页面。

### 6.7 归档态读取与错误语义

1. 非成员访问任何账本对象统一返回不泄漏数据的 403 或冻结后的 404 策略，OpenAPI 只能选定一种对外口径。
2. 对归档账本执行写操作返回稳定 `LEDGER_ARCHIVED` 冲突错误，建议 HTTP 409。
3. 未提供业务请求账本 ID 返回稳定 `LEDGER_REQUIRED`，不得 fallback。
4. 当前成员数已满返回 `LEDGER_MEMBER_LIMIT_REACHED`。
5. 最后 Owner、无效移交、ready 导入批次阻断、并发版本冲突均使用独立错误码，供 UI 给出可操作说明。

## 7. 模块隔离要求

| 模块 | Task50 冻结要求 |
|---|---|
| 流水/共同支出 | 所有读写按显式账本过滤；private/partner_readable/shared 语义不变 |
| 结算 | 仅结算当前账本成员与当前账本共同账单；不改历史账单 |
| 元数据 | 分类、标签、账户、默认值和排序不得跨账本复用 ID |
| 模板/周期 | 规则和生成结果都属于当前账本；归档态禁止确认和生成 |
| 导入 | 批次、行、规则、hash 和 transaction_import_refs 全部按账本隔离；重复只在同账本判定 |
| 附件 | 下载必须先验证父交易属于当前账本且当前用户可见；禁止裸静态路径绕过 |
| 报表 | settlement 不进入消费统计；所有聚合只读取当前账本和当前用户可见数据 |
| CSV/JSON 导出 | Owner/Editor 仅导出当前账本且当前角色可见数据，不夹带其他账本业务数据 |
| 备份/恢复 | 明确为整库运维能力，不伪装成单账本备份；只允许独立实例管理员 |
| 审计 | lifecycle/member 事件按账本记录；跨账本管理员数据另行定义，不混入普通账本导出 |

## 9. 数据升级与守恒

1. 只新增 migration，不修改 001-019 已应用 migration。
2. 所有既有账本补为 active，既有成员角色和业务数据原样保留。
3. 升级前输出账本数、每账本成员数/Owner 数、交易/结算/导入/附件数量和 SQLite `quick_check`。

6. development、staging、production 数据库继续物理隔离；先在副本和 NAS staging 验证，再进入 production 窗口。

## 10. 验收标准

### 10.1 产品主路径

1. 用户创建第二个账本后，两个账本数据和 UI 状态相互独立。
2. Owner 可重命名、归档和恢复；Editor/Viewer 请求均被后端拒绝。
3. 归档后所有写入口禁用，直接调用 API 也返回 `LEDGER_ARCHIVED`；历史读取和按权限导出仍可用。
4. Owner 原子移交成功后，新 Owner 获得管理权，原 Owner 成为 Editor；任何中途失败不产生双 Owner或零 Owner。
5. Editor/Viewer 离开或被移除后，当前页面立即退出该账本，旧缓存不再显示。
6. 最多两名成员的限制在 UI 和 API 同时生效。

### 10.2 隔离矩阵

对账本 A/B 和 Owner/Editor/Viewer/非成员组合，至少验证：

1. A 的 transaction/category/tag/account/template/recurring/import batch/import rule/settlement ID 不能在 B 读取或修改。
2. A 的附件不能通过 B 的 header、裸 URL 或同名文件访问。
3. A 的 private 账单不会出现在另一成员或 B 的列表、报表、CSV、JSON 和审计导出。
4. A 的 import hash 不会把 B 的同一流水误判为重复。
5. 切换 A -> B 时，A 的迟到响应不能覆盖 B；刷新后仍保持有效当前账本。
6. 无 active 账本时不发出交易、报表、导入等无 header 请求。

### 10.3 升级与回滚

1. v1.2 schema 19 副本升级后 `quick_check=ok`，行数和金额汇总守恒。
2. 升级前后现有账本均为 active，现有用户仍能访问原账本。
3. migration 中途失败不留下半应用状态。
4. 回退镜像后原有 v1.2 核心读写仍可用，或发布计划明确要求同步数据库回退并验证备份恢复。

## 11. 成功指标

1. 跨账本隔离自动化矩阵 100% 通过，P0 数据泄漏为 0。
2. lifecycle/member 不变量测试 100% 通过，无法构造零 Owner、双 Owner或第三名成员。
3. 375/390/430/1440 无横向溢出、操作遮挡和不可解释的静默切换。
4. 真实 v1.2 副本升级后账单、结算、导入引用和附件数量守恒。
5. Task50 发布后不增加单账本用户完成高频记账的步骤数。

## 12. 已冻结决策与后续延后

本 PRD 不保留影响数据模型或权限矩阵的 Open Question。已冻结：

1. Task50 最多两名成员，第三名留到 Task51。
2. archived 为默认隐藏且全员只读，Owner 可恢复，不支持删除。
3. Owner 唯一且必须存在；所有权通过独立原子操作移交。
4. Editor/Viewer 可离开；Owner 先移交再离开。
5. partner_readable 对当前另一名成员开放历史读取，添加成员前必须警告。
6. 所有账本内请求显式账本；无可用账本进入空状态，不 fallback 到第一账本。
7. Owner/Editor 可导出其可见数据；账本导入 Owner-only；整库运维仅实例管理员可用。
8. 未结清净额不阻止归档但必须提示；ready 导入批次阻止归档。

其中第 7 条的“整库运维”准确含义为实例管理员专属，不随账本 Owner 身份自动获得。

延后到技术/API/UI 评审的只是实现形式，例如字段名、ETag/version 方案、错误响应结构、索引和组件布局；这些不得反向改变上述产品规则。
