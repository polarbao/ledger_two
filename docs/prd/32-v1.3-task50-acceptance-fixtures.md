# Task50 v1.3 多账本 Fixture 与验收矩阵

状态：Task50P.4 冻结  
冻结日期：2026-07-15  
适用范围：Task50.1-Task50.6 自动化、Migration 副本演练、Fresh Light UI 验收和发布准入

## 1. 目标与边界

本文件把 `31-prd-v1.3-multi-ledger.md` 和 `25-v1.3-multi-ledger-implementation-contract.md` 转成可重复构造的匿名数据与验收用例。它不包含真实用户、账单、数据库、附件或密钥，也不代表 Task50 已进入开发。

冻结规则：

1. Fixture 只允许通过测试 builder、SQL test seed 或隔离数据库生成，不提交运行数据库。
2. 金额全部使用整数分；时间固定为 UTC ISO8601，测试不得依赖当前日期。
3. 正常 Fixture 始终满足“每账本 1 名 Owner、最多 2 名成员”；异常 Fixture 独立构造，只用于 Migration 预检拒绝。
4. 账本 A 的对象 ID 在账本 B 上读取、修改、导出和附件下载都必须失败。
5. archived 默认可按权限读取、不可写；restore 是唯一 archived lifecycle mutation。
6. Task50 不扩大到第三名成员、多人分摊、邀请链接、通知或单账本物理备份。

## 2. 确定性 Fixture

### 2.1 用户

| 代号 | 固定 ID | 用户名 | 实例管理员 | 用途 |
|---|---|---|:---:|---|
| U-A | `usr_task50_alice` | `fixture_alice` | 是 | A/C/D Owner，实例运维交叉权限 |
| U-B | `usr_task50_bob` | `fixture_bob` | 否 | A Editor、B Owner、C Viewer |
| U-C | `usr_task50_cara` | `fixture_cara` | 否 | B Viewer，非 A/C/D 成员 |
| U-D | `usr_task50_dan` | `fixture_dan` | 否 | 全部账本非成员 |

密码只在测试运行时由 builder 生成固定哈希；文档和仓库不得保存可登录真实环境的明文密码。

### 2.2 账本与成员

| 代号 | 固定 ID | 名称 | 状态/version | 成员 | 关键用途 |
|---|---|---|---|---|---|
| L-A | `ldg_task50_active_a` | 共同生活 | active/v3 | U-A Owner、U-B Editor | 双人主路径、跨账本源对象 |
| L-B | `ldg_task50_active_b` | 旅行计划 | active/v5 | U-B Owner、U-C Viewer | 隔离目标、Viewer 权限 |
| L-C | `ldg_task50_archived_c` | 旧居生活 | archived/v8 | U-A Owner、U-B Viewer | 归档读取与写入拒绝 |
| L-D | `ldg_task50_single_d` | 个人筹备 | active/v1 | U-A Owner | 新增第二成员、无伙伴状态 |

L-C 固定归档时间为 `2026-06-30T12:00:00Z`，归档人为 U-A。其他账本 `archived_at/archived_by_user_id` 必须为 null。

### 2.3 业务对象

所有对象都显式归属 L-A 或 L-B；子对象通过父对象验证账本，不能只比较裸 ID。

| 对象 | L-A 固定 ID | L-B 固定 ID | 数据要求 |
|---|---|---|---|
| transaction/private | `txn_a_private` | `txn_b_private` | 10001/20001 分，仅创建者可见 |
| transaction/partner_readable | `txn_a_partner` | `txn_b_partner` | 12002/22002 分，另一成员可读不可编辑 |
| shared expense | `txn_a_shared` | `txn_b_shared` | 30003/40003 分，均有 `transaction_splits` |
| settlement | `stl_a_001` | `stl_b_001` | 独立记录，不改历史共同账单 |
| category | `cat_a_food` | `cat_b_food` | 名称可同为“餐饮”，ID 和账本隔离 |
| tag | `tag_a_trip` | `tag_b_trip` | 名称可同为“旅行” |
| account | `acc_a_wechat` | `acc_b_wechat` | 名称可同为“微信” |
| template | `tpl_a_daily` | `tpl_b_daily` | active 模板 |
| recurring rule | `rec_a_rent` | `rec_b_hotel` | 不在归档态生成 |
| import rule | `imr_a_coffee` | `imr_b_coffee` | 同 pattern、不同账本 |
| import batch | `imb_a_ready` | `imb_b_ready` | L-A ready 且未过期；L-B ready 用于 hash 隔离 |
| attachment | `att_a_receipt` | `att_b_receipt` | 文件名可相同，下载先校验父交易 |

额外批次：

1. `imb_a_committed`：L-A 已提交，含 `transaction_import_refs`。
2. `imb_a_expired`：L-A 已过期，不阻止归档。
3. `imb_c_committed`：L-C 历史批次，只读。
4. L-A/L-B 各放一条相同来源、外部订单号、金额和日期的导入行，验证重复 hash 只在账本内生效。

### 2.4 金额守恒快照

测试 builder 必须输出下列基线，不允许在断言中读取随机顺序或浮点值：

```text
ledger_count=4
member_count=7
owner_count_by_ledger=A:1,B:1,C:1,D:1
transaction_amount_sum_cents_by_ledger=A:<builder-fixed>,B:<builder-fixed>,C:<builder-fixed>,D:0
settlement_amount_sum_cents_by_ledger=A:<builder-fixed>,B:<builder-fixed>,C:<builder-fixed>,D:0
import_batch_count_by_ledger=A:3,B:1,C:1,D:0
sqlite_quick_check=ok
```

`<builder-fixed>` 在实现 Task50 fixture builder 时由同一常量文件给出，并在 migration 前后复用；不能在两套 seed 中分别手写。

## 3. 异常升级 Fixture

异常数据每次只引入一种不变量破坏，预检必须停止升级且不修改 schema/data：

| 代号 | 异常 | 预期 |
|---|---|---|
| X-01 | L-X 有 3 名成员 | 拒绝 020，报告账本 ID 与成员数，不自动删除 |
| X-02 | L-X 没有 Owner | 拒绝 020，不猜测最早成员 |
| X-03 | L-X 有 2 名 Owner | 拒绝 020，不自动降级任一 Owner |
| X-04 | transaction 指向不存在 ledger | 拒绝升级，报告表与对象 ID |
| X-05 | member 指向不存在 user | 拒绝升级，报告孤立关系 |
| X-06 | schema 版本不是 19 | 拒绝 Task50 升级，保持原数据库 |
| X-07 | `quick_check` 非 ok | 拒绝升级并要求从可信备份恢复 |

## 5. 全模块隔离和归档 Guard

对下表每一行执行“L-A 对象 + `X-Ledger-Id: L-B`”以及“L-C archived 写入”两组用例：

| ID | 模块 | 跨账本请求预期 | archived 写入预期 | archived 读取 |
|---|---|---|---|---|
| T50-ISO-001 | transaction CRUD/批量标签 | 404 `LEDGER_OBJECT_NOT_FOUND` | 409 `LEDGER_ARCHIVED` | 按可见性允许 |
| T50-ISO-002 | shared expense/splits | 404 | 409 | 允许，金额不重算 |
| T50-ISO-003 | settlement | 404/空集合且不泄漏 | 409 | 历史与余额允许 |
| T50-ISO-004 | category/tag/account/default | 404 | 409 | 允许 |
| T50-ISO-005 | template/recurring/reminder | 404 | 409 | 允许；不生成新账 |
| T50-ISO-006 | import batch/row/rule/hash | 404 | 409 | 已有批次/规则允许 |
| T50-ISO-007 | attachment upload/download | 404，裸 `/uploads` 仍 404 | 上传 409 | 可见父交易附件可下载 |
| T50-ISO-008 | reports/dashboard | 只聚合请求账本 | 不适用写入 | 允许且 settlement 不进消费 |
| T50-ISO-009 | CSV/JSON export | 文件不得含 L-A 对象 | Owner/Editor 允许受控导出 | 允许 |
| T50-ISO-010 | audit | 不返回另一账本事件 | lifecycle 写入独立受控 | 按权限允许 |

附加可见性断言：

1. U-B 可读取 L-A 的 `partner_readable/shared`，不可读取 `txn_a_private`。
2. U-C 是 L-B Viewer，只能读取 L-B 可见数据，不能写入或导出。
3. 成员离开/被移除后，历史 actor 名称仍可展示，但该用户不能再访问账本。
4. 同名附件必须通过父交易与账本定位，不能只凭文件名下载。

## 9. 自动化分层与完成标准

### 9.1 最低自动化集合

1. Repository：生命周期条件更新、partial unique Owner、两人 trigger、跨账本查询。
2. Service：归档事务、ready 阻断、原子移交、离开/移除、instance policy。
3. Router：cookie auth、403/404/409、header/path mismatch、If-Match/ETag、统一 envelope。
4. Frontend：store 回退、query key/cancel、角色和 archived 状态、冲突保留输入。
5. Migration：schema19 副本、所有异常预检、守恒和备份恢复演练。
6. Browser：375/390/430/1440 的 Owner/Editor/Viewer、归档和无账本主路径。

### 9.2 Task50P.4 完成判定

P.4 只有同时满足以下条件才关闭：

1. `openapi-v1.3-ledger-draft.yaml` 可由标准 YAML parser 解析，path parameter 均有定义。
2. 错误码唯一且 UI 可据此提供下一步，不依赖 SQL 文案。
3. 正常与异常 Fixture、跨账本对象、实例管理员和 migration 守恒均已定义。
4. owner/editor/viewer/非成员、active/archived、A/B 账本组合没有权限空白。
5. 文档明确是开发前冻结契约，未声称当前 router 已实现。
