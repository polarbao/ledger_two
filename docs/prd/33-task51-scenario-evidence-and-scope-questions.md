# Task51 多人分摊场景证据与范围问题

状态：前置发现草案，不构成 PRD 冻结或开发准入
创建日期：2026-07-16
当前阶段：仅启动 Task51P.1 的非约束性证据准备；正式 Task51P 仍等待 Task50.6

## 1. Goal

在解除 schema 21 的最多两名成员约束、设计多人分摊算法或新增 UI 之前，先证明 LedgerTwo 的真实用户是否需要三人及以上账本，以及需求主要来自长期家庭/合租还是旅行/聚会等临时场景。

本阶段的产出是可评审的场景证据、问题清单和继续/缩减/延后建议，不是“多人功能一定开发”的既定结论。

## 2. Current facts

1. LedgerTwo 当前定位是情侣、夫妻、合租伙伴和小家庭的私有化共同生活账本，双人长期记账与阶段性结算仍是核心价值。
2. Task50 冻结每账本最多两名成员；Task51 不能修改已应用 migration，只能在后续通过新 migration 向前演进。
3. 当前 shared expense、split 和 settlement 已保留多人数据形态，但权限、历史可见性、成员生命周期和兼容 UI 仍按双人可信闭环验收。
4. 旅行、装修和小范围合租可能需要多人账本；一次性聚会 AA 使用频率低、留存弱，并容易把产品带向通知、邀请和支付状态机。
5. “直接通知共同支付”继续延后；复制结算文案是现阶段低成本替代，不因 Task51P 启动而自动进入范围。
6. 当前没有足以证明多人需求优先级的匿名遥测、访谈记录或稳定生产使用数据，不得把路线文档当作真实需求证据。

## 3. Candidate scenarios

| 场景 | 使用频率假设 | 与产品定位匹配 | 主要复杂度 | 初步判断 |
|---|---|---|---|---|
| 3-6 人长期家庭/合租 | 中高 | 高 | 成员更替、历史隐私、持续结算 | 优先验证 |
| 3-8 人旅行/装修项目 | 阶段性 | 中高 | 临时加入退出、结束归档、多人结算 | 次优先验证 |
| 单次聚餐/活动 AA | 低 | 低 | 快速邀请、通知、支付追踪 | 不作为默认主线 |
| 公开大群或组织报销 | 不适用 | 低 | 多租户、审批、合规 | 明确非目标 |

初步产品建议：Task51 若获得准入，应优先服务“有持续账务关系的小范围成员”，再验证旅行/装修项目；不为单次聚会先建立邀请通知与支付追踪系统。

## 4. Evidence plan

### 4.1 Evidence sources

1. 结构化访谈：目标为 3-5 个真实家庭、合租或旅行小组，记录成员数量、持续时间、账单频率、结算频率和现有替代方案。
2. 任务回放：至少记录“创建多人账本 -> 记共同支出 -> 修改分摊 -> 查看解释 -> 完成结算 -> 成员退出/账本归档”完整流程中的阻力。
3. 现有实例人工观察：只使用用户明确提供的匿名统计，不读取或提交真实账单内容、姓名、商户、备注、附件或支付文件。
4. 低成本原型：在不改 schema 和生产代码的前提下，用本地 Fixture/Figma 验证 3、5、8 人时的信息密度和错误修复路径。
5. 替代方案比较：记录微信群收款、表格、Splitwise 类工具或手工转账文案为何不足，避免只因为“可以做”而扩展范围。

### 4.2 Evidence record

每条证据至少包含：

```text
scenario_id
group_type: household | co_living | travel | renovation | gathering
member_count_range
duration_range
shared_expenses_per_period
settlement_frequency
current_workaround
top_failure_or_cost
privacy_expectation
join_leave_expectation
notification_need
source_and_consent
```

禁止记录真实姓名、账号、商户、订单号、金额明细、附件或可反向识别个人的数据。

## 5. Decision gate

P1 评审只能给出以下三种结论：

1. `continue`：存在重复、可复现的 3+ 成员长期或阶段性工作流，且核心问题确实需要账本内多人模型解决。
2. `narrow`：需求成立但只覆盖一种场景，例如 3-4 人合租；后续 PRD 必须按该边界冻结，不能顺带覆盖聚会通知。
3. `defer`：证据主要来自偶发聚会，或复制文案/表格已足够；Task51 继续延后，不解除两人约束。

建议的最低评审材料，而非未经验证的成功指标：

1. 至少 3 个彼此独立的真实目标小组记录。
2. 至少 2 次完整工作流回放，其中 1 次包含成员更替或结束归档。
3. 至少 1 组 3 人、1 组 5 人匿名 Fixture 的可解释分摊/结算演练。
4. 明确列出不做通知后，目标用户是否仍愿意使用。

材料不足时默认 `defer`，不能用开发者直觉补齐证据。

## 6. Open questions

以下问题在 P2 PRD 前必须回答，但当前不得预先冻结答案：

1. 首个支持上限是 4、6、8 人还是仅由性能/交互评审后决定？
2. 新成员能看到加入前哪些 partner_readable/shared 历史？private 永不扩大可见性的证明方式是什么？
3. 成员离开后，历史 split/settlement、姓名展示和再次加入如何处理？
4. Owner 是否仍唯一；多人账本是否需要协作者管理成员，还是继续 Owner-only？
5. equal 的整数分余数由谁承担，amount/ratio/shares 如何证明总额守恒和可解释？
6. “最小转账”是必须算法、可选建议还是应先沿用确定性贪心？如何保证建议稳定且不修改历史？
7. 临时账本结束是归档、结清后归档还是独立 completed 状态？
8. 邀请如何完成？若不做链接/通知，精确 username 是否足以验证首版价值？
9. 多人 UI 在 375px 下如何编辑、解释和修复错误，而不把每个成员塞进嵌套卡片？
10. 从 schema 21 升级后，旧双人账本和 API 客户端如何保持行为不变？

## 7. Non-goals

1. 本文不冻结版本号、成员上限、算法、API、migration 编号或发布日期。
2. 不修改 migration 020/021，不解除数据库成员 trigger，不实现 Task51 代码。
3. 不把通知、支付状态、邀请链接、OCR、银行同步或公开群组纳入 Task51P.1。
4. 不采集隐蔽遥测，不把真实账务数据复制到仓库、AI 对话或测试 Fixture。

## 8. Completion criteria

Task51P.1 只有在 Task50.6 完成正式入口检查后，使用匿名证据记录重新评审并形成 `continue/narrow/defer` 结论，才算正式完成。本草案当前只完成场景分类、证据格式、隐私边界和决策规则准备。
