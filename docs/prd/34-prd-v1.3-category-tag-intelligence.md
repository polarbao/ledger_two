# PRD：v1.3 分类、标签与导入智能归类专项

状态：产品基线已冻结；Task53.1-Task53.3 已完成，Task53.4 准备关闭<br>
任务编号：Task53<br>
创建日期：2026-07-16<br>
优先级：Task53.3 预览持久化、解释与重分类<br>
关联研究：`docs/project_analysis/2026-07-16-category-tag-competitive-research.md`

## 1. Goal

让用户导入微信、支付宝和通用模板账单后，大部分重复、明确的分类与标签整理可以自动完成，同时保证每一项自动结果可解释、可修改、可回退，不让错误分类静默进入正式账本。

专项同时补齐首次初始化、新建账本和既有账本的基础分类/标签能力，建立“默认元数据 -> 导入识别 -> 用户修正 -> 明确学习 -> 下次自动选择”的闭环。

## 2. Product principles

1. 分类是唯一主统计维度；标签是可选横向维度。
2. 自动化发生在导入预览，不绕过整批确认，不自动提交正式账单。
3. 用户明确创建或确认的规则优先于系统内置知识。
4. 低置信结果只能建议，不能假装为确定答案。
5. 学习必须显式，不能在用户不知情时根据真实账务行为建立规则。
6. 所有识别在本机/NAS 实例内完成，不向外部服务上传商户、标题、备注、金额或订单信息。
7. 分类、标签和规则全部按账本隔离，同名对象可以存在于不同账本。

## 3. Users and scenarios

### 3.1 Target users

1. 每月导入微信/支付宝账单并需要批量整理的 Owner。
2. 希望不同账本拥有合理基础分类和标签的双人/家庭用户。
3. 经常在相同商户消费，希望一次修正后后续自动识别的长期用户。

### 3.2 Core scenarios

1. 用户首次初始化后直接拥有可用的支出、收入分类与基础标签。
2. 用户新建账本时默认复制基础分类/标签包，也可明确选择空白账本。
3. 导入账单后，星巴克、滴滴、国家电网等稳定商户依据用户规则自动选择分类。
4. 用户把某商户改为特定分类和标签，并勾选“记住此商户”；下次导入自动选择。
5. 系统只对模糊商户给出建议，用户可以批量接受、逐条修改或忽略。
6. 规则引用的分类/标签归档后不再自动应用，并在规则管理中显示修复提示。

## 4. Scope

### 4.1 Default metadata profile

首期冻结两个 profile：

| Profile | 用途 | 行为 |
|---|---|---|
| `basic_cn_v1` | 默认 | 创建基础支出分类、收入分类和标签 |
| `empty` | 高级用户 | 只创建账本和 Owner，不创建分类/标签 |

系统首次初始化固定使用 `basic_cn_v1`。Task50 新建账本界面默认选中 `basic_cn_v1`，Owner 可明确切换为 `empty`。既有账本不得静默补齐，必须通过“预览基础包 -> 解决同名冲突 -> 确认应用”操作。

### 4.2 Graded automation

分类决策优先级：

```text
手工行调整
> 本批次批量调整
> 用户显式规则
> 用户确认生成的学习规则
> 本地内置建议
> 支出/收入兜底分类
```

| Level | 来源 | category 行为 | tag 行为 | 置信度 |
|---|---|---|---|---|
| L0 | 手工/批量调整 | 最终选择 | 最终集合 | manual |
| L1 | 用户显式规则 | 自动选择 | 自动添加 | high |
| L2 | “记住此商户”学习规则 | 自动选择 | 自动添加 | high |
| L3 | 内置商户/关键词知识 | 仅建议 | 仅建议 | medium/low |
| L4 | 系统兜底 | 自动选择其他支出/收入 | 不添加 | fallback |

约束：

1. L1/L2 只填充预览行，不自动提交。
2. L3 必须由用户逐条或批量接受。
3. L4 单独计数，不能计入“正确自动识别”。
4. 多个分类候选只选择最高优先级的一个；同级冲突转为建议并要求确认。
5. 多个高置信标签可取并集，但最多 8 个；用户手工编辑后标签集合为最终值。
6. duplicate、invalid 和 skipped 行不执行自动选择。

### 4.3 Explicit learning

用户在行编辑器中修改分类或标签时，可勾选：

```text
记住此商户的分类和标签
```

行为：

1. 仅对规范化后非空的商户生效。
2. 创建或更新 `origin=learned` 的精确商户规则。
3. 默认限制当前账本和当前来源；用户可在规则管理中扩展为所有来源。
4. 若已有用户显式规则，不静默覆盖，提示查看冲突。
5. 用户可编辑、归档和恢复学习规则。
6. 取消勾选只保存本行调整，不产生规则。

首期不做基于次数的隐式学习，也不从历史账单后台扫描生成规则。

### 4.4 Import preview experience

批次摘要新增：

1. 已自动选择。
2. 待接受建议。
3. 使用兜底分类。
4. 手工调整。
5. 批量调整。
6. 分类冲突。

每行展示：

1. 最终分类、标签。
2. 状态：`自动选择`、`建议`、`兜底`、`手工`、`批量`、`冲突`。
3. 来源：用户规则、学习规则、内置词典或系统兜底。
4. 可读原因，例如“精确匹配商户「滴滴出行」”。
5. “接受建议”“编辑”“应用到相同商户”“跳过”。

批量操作：

1. 接受当前筛选下全部无冲突建议。
2. 将分类/标签应用到同一规范化商户。
3. 只批量处理选中的行；操作前显示影响数量。
4. 不允许对 invalid、duplicate 默认跳过行批量应用。
5. “接受建议”使用每行已持久化的建议，不要求所有行拥有相同分类；“应用相同值”才统一写入指定分类和标签。
6. 批量调整不隐式创建学习规则；长期学习必须通过单条“记住此商户”或“应用到相同商户并记住”明确触发。

### 4.5 Category and tag management

1. 分类/标签显示 `系统基础`、`用户创建` 标识，但视觉不夸张。
2. 显示使用次数和被多少条启用规则引用。
3. 归档前提示受影响规则数量。
4. 兜底分类归档必须先指定同类型替代分类。
5. 既有账本可预览应用基础包，同名同类型默认复用，不重复创建。
6. 同名异类型分类可共存；标签同账本名称唯一。

分类合并、标签分组、二级分类和跨账本复制属于后续增强，不阻断 Task53 首期。

## 5. Roles and permissions

| 操作 | Owner | Editor | Viewer |
|---|:---:|:---:|:---:|
| 查看分类/标签 | 是 | 是 | 是 |
| 管理分类/标签 | 是 | 否 | 否 |
| 导入与调整预览 | 是 | 否 | 否 |
| 创建/管理规则 | 是 | 否 | 否 |
| 生成学习规则 | 是 | 否 | 否 |
| 查看规则命中解释 | 是 | 否 | 否 |

Task53 不扩大当前导入权限。

## 6. Business rules

1. 分类必须属于当前账本，且类型必须与目标交易类型兼容。
2. 支出和 shared_expense 只能使用 expense 分类；收入和退款映射使用 income 分类。
3. 已归档分类/标签不能用于新的自动选择或建议。
4. 预览创建后，即使规则发生变化，当前批次结果不自动重算；用户应重新生成预览或显式刷新分类结果。
5. commit 只使用 `selected_*` 最终值，不在事务提交阶段重新运行规则。
6. 用户手工清空自动标签后，commit 不得重新补回。
7. 所有金额继续使用整数分；分类引擎不得修改金额、时间、付款人、可见性、分摊或重复状态。
8. 分类结果不能扩大 private 账单可见范围。

## 7. Success metrics

不启用隐蔽遥测。系统可在当前批次和本地统计页计算：

1. `auto_selected_rate`：高置信自动选择行 / 可导入行。
2. `suggestion_accept_rate`：被接受建议 / 建议行。
3. `auto_correction_rate`：自动选择后被修改行 / 自动选择行。
4. `fallback_rate`：兜底分类行 / 可导入行。
5. `learned_rule_reuse_count`：学习规则后续命中次数。

首期产品目标：

1. 对已有稳定规则的账本，自动选择覆盖率不低于 60%。
2. 高置信自动选择修正率低于 5%。
3. 任一自动结果都可在两次操作内定位原因并修改。
4. 375px 下可完成建议筛选、接受、编辑和提交，无横向滚动。

指标不足不阻止基础功能发布，但 `auto_correction_rate >= 10%` 时必须默认关闭学习规则自动选择，降级为建议。

## 8. Error and empty states

| 场景 | 行为 |
|---|---|
| 无分类 | 引导应用基础包或创建分类，不允许静默提交无分类支出/收入 |
| 无标签 | 分类仍可工作，标签区域显示可选空状态 |
| 规则引用归档元数据 | 规则不应用，显示修复入口 |
| 同级分类规则冲突 | 不自动选择，进入冲突筛选 |
| 标签超过 8 个 | 阻断保存，显示需移除数量 |
| 商户为空 | 不允许创建学习规则，仍可手工分类 |
| 基础包同名冲突 | 预览显示复用、创建或跳过，不自动改名 |
| 批量操作影响 0 行 | 禁用确认按钮，不发空请求 |

## 9. Non-goals

1. 不做云端 AI、LLM、OCR、银行同步和外部分类 API。
2. 不自动修改正式历史账单。
3. 不让规则修改金额、时间、类型、可见性、付款人或分摊。
4. 不做通用 AND/OR/正则自动化平台。
5. 不做分类预算、标签分组、二级分类和跨账本元数据复制首期实现。
6. 不把自动通知共同支付带入本专项。

## 10. Acceptance

1. 首次初始化和默认新账本拥有 `basic_cn_v1` 完整分类/标签，写入同一事务。
2. 新账本选择 `empty` 时不创建分类/标签。
3. 既有账本应用基础包是可预览、幂等和显式确认的。
4. 用户规则/学习规则命中时自动填充最终分类和标签，但批次仍需确认提交。
5. 内置知识只产生建议，未接受时不能进入最终标签。
6. 行调整后勾选“记住此商户”，下次相同账本/来源/规范化商户自动选择。
7. 规则冲突、归档引用、标签上限和跨账本引用均被稳定拒绝或降级。
8. commit 不重新分类，失败时仍保持整批事务回滚。
9. 分类/标签解释、批量接受和错误定位在 375/390/430/1440 通过验收。
10. 关闭专项开关后退回现有 Task49 “仅建议”行为，不影响 CSV/XLSX 解析和去重。
