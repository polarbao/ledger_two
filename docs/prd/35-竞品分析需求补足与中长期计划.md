# PRD：竞品分析、需求补足与中长期产品计划

状态：建议采纳<br>
创建日期：2026-07-17<br>
角色视角：产品经理 + 全栈架构师<br>
适用阶段：v1.3 Task53 之后的中长期规划输入

## 1. 文档目的

本文把外部记账、预算、自托管财务、共同支出产品的能力拆解为 LedgerTwo 可吸收的需求池，并判断当前产品受众、短中长期方向和架构优化重点。

本文不是立即开发任务卡。进入开发前仍需拆成独立 PRD、技术契约、UI 流程、OpenAPI、migration 评审和验收样例。

## 2. 上次会话建议回放

上次全局产品评审给出的核心建议如下：

1. LedgerTwo 后续应坚持做“可信、私有、可解释的共同生活账本”，不要变成大而全财务软件。
2. 竞品能力可吸收，但需要按 LedgerTwo 定位过滤：
   - Splitwise / Tricount / Settle Up：吸收共同支出、分组、谁欠谁、结算体验。
   - Actual Budget / Firefly III：吸收自托管、数据控制、导入、规则、报表。
   - 钱迹 / 随手记 / 鲨鱼记账：吸收快速记账、本土支付场景、分类智能、移动端效率。
   - YNAB / Monarch / Copilot / MoneyWiz：吸收目标、周期交易、报告、家庭协作，但只采用轻量形态。
3. 当前最优路线不是扩张功能，而是把现有路线收成三个产品记忆点：
   - 两个人的钱，算得清楚。
   - 自己的账，自己掌控。
   - 越用越省事，而不是越用越复杂。
4. 短期保持 v1.2/NAS 发布线独立维护，继续完成 Task53 分类标签智能化；不做银行同步、OCR、原生 App、自动通知支付和复杂预算。
5. 中期重点是降低记账成本：默认分类/标签、导入分级自动化、批量调整、周期账单、模板、移动端细节和轻量报表。
6. 长期目标是轻家庭账本：多账本、多成员、多人分摊、最小转账建议、家庭/旅行/装修/合租场景模板、月报年报和恢复演练。

## 3. 当前竞品类型与能力分析

### 3.1 共同支出与 AA 产品

代表产品：Splitwise、Tricount、Settle Up。

| 产品 | 核心定位 | 主要优势 | LedgerTwo 可借鉴点 |
|---|---|---|---|
| Splitwise | 朋友、室友、旅行、家庭共享支出和结算 | 跟踪 shared expenses、balances、who owes who；按 group 管理；快速添加；settle up 记录现金或线上支付 | 强化“谁欠谁”和结算解释；后续多人分摊、组内结算、结清后隐藏但不删除历史 |
| Tricount | 旅行、活动、室友和情侣的多人分摊 | 链接邀请、每个人都能添加支出、自动同步、余额可见 | 后续多人场景要先做低摩擦加入和共享视图，但当前不急于公开邀请 |
| Settle Up | 旅行者、合租、情侣、群组共同费用 | 所有费用可备份和群组可见；每个成员理解谁欠什么 | 多人账本要保留群组可见性和备份，不把结算做成一次性黑盒 |

结论：

LedgerTwo 当前最该深挖的是共同支出的信任体验，而不是预算或投资。下一阶段多人分摊必须以真实场景证据为准，避免过早把双人主路径复杂化。

### 3.2 自托管与开源财务产品

代表产品：Actual Budget、Firefly III。

| 产品 | 核心定位 | 主要优势 | LedgerTwo 可借鉴点 |
|---|---|---|---|
| Actual Budget | local-first、隐私优先、信封预算、自托管同步 | 用户拥有数据；可自托管；多设备同步；端到端加密；文件导入导出；离线可用 | 继续强化 NAS 私有部署、备份恢复、导入、数据迁移和离线草稿 |
| Firefly III | 自托管个人财务管理 | 复式记账、多币种、预算、分类、标签、规则、导入和财务报表 | 吸收规则引擎、分类标签管理和报表，不引入复杂复式会计心智 |

结论：

LedgerTwo 的私有化路线成立。自托管产品的关键不是功能炫技，而是部署可解释、数据可迁移、升级可回滚、导入可修正。

### 3.3 预算与财富管理产品

代表产品：YNAB、Monarch、Copilot、MoneyWiz。

| 产品 | 核心定位 | 主要优势 | LedgerTwo 可借鉴点 |
|---|---|---|---|
| YNAB | 预算方法论与家庭协作 | 银行导入、跨设备同步、目标追踪、贷款工具、多人共享订阅 | 后续可做轻量目标和家庭协作；不照搬完整预算方法论 |
| Monarch | 综合家庭财务管理 | 多账户聚合、预算、报告、家庭成员协作 | 可借鉴家庭财务仪表盘，但不做银行聚合 |
| Copilot | 现代个人财务体验 | 自动分类、周期交易、订阅识别、精致移动体验 | 可借鉴周期账单、订阅识别和分类解释，但避免云端 AI 依赖 |
| MoneyWiz | 全面个人财务与多平台同步 | 账户、贷款、预算、目标、账单订阅、投资、30+ 报表、多币种、导入 | 可借鉴报表、日历、周期预测；投资、加密资产和复杂多币种延后 |

结论：

预算和资产管理是成熟财务产品的常见能力，但 LedgerTwo 近期只应做“轻预算”和“复盘报表”。复杂预算、投资、银行同步会把产品拉离共同生活账本主线。

### 3.4 国内个人记账与本土支付场景

代表产品：随手记、鲨鱼记账、钱迹。

| 产品 | 核心定位 | 主要优势 | LedgerTwo 可借鉴点 |
|---|---|---|---|
| 随手记 | 多场景账本、快捷/自动记账、多人协同 | AI 语音/图片记账、账本模板、多维报表、多人权限、云同步和恢复 | 账本模板、多人权限和多维报表值得借鉴；AI/OCR 不进近期 |
| 鲨鱼记账 | 极简快速生活记账 | 3 秒记账、预算、导出、定时记账、日历、提醒、搜索 | 高频入口要继续做轻；移动端记账速度是关键指标 |
| 钱迹 | 纯粹、无广告、极速记账 | 导入导出、自定义、搜索、日历、密码保护、家庭共享、周期记账 | LedgerTwo 应坚持纯净、可导出、可搜索、家庭共享和本土支付导入 |

结论：

国内产品最值得学习的是速度、本土支付导入和低干扰体验。LedgerTwo 不应变成广告、理财和金融产品入口。

## 4. 需求补足与深挖计划

### 4.1 P0：可信账本与发布安全

| 需求 | 用户价值 | 竞品启发 | LedgerTwo 计划 |
|---|---|---|---|
| 发布前备份和回滚 | 防止账务数据丢失 | 自托管产品重视数据可控 | 继续保持 staging/production 隔离、schema 与镜像成对升级 |
| 恢复演练向导 | 用户知道数据真的可恢复 | Actual / Firefly 的数据控制心智 | v1.5 前做恢复演练 Checklist 和 UI 引导 |
| private 附件权限 | 避免隐私泄露 | 共同账本隐私比功能更重要 | 附件必须通过受保护接口，不开放裸静态路径 |
| 操作审计 | 高风险动作可追踪 | 随手记强调操作留痕 | 金额变更、删除、导入提交、结算、恢复都写审计 |

产品原则：

可信优先于功能数量。任何导入、结算、恢复和多人协作功能都不能绕过权限、审计和备份。

### 4.2 P0/P1：快速记账与移动端效率

| 需求 | 用户价值 | 竞品启发 | LedgerTwo 计划 |
|---|---|---|---|
| 10 秒普通支出 | 降低每日记账阻力 | 鲨鱼记账 3 秒记账、钱迹秒开 | 保留金额优先、默认分类、最近账户、保存并继续 |
| 20 秒共同支出 | 让分摊不成为负担 | Splitwise 快速添加 shared expense | 付款人、参与人、分摊方式和预览同屏解释 |
| 复制一笔和模板 | 降低重复录入 | 钱迹/鲨鱼定时记账、模板心智 | 复制、模板、周期待确认继续作为核心效率功能 |
| 记账提醒与日历补账 | 防漏记、补历史 | 鲨鱼日历和提醒 | 可做轻量日历视图，不把提醒做成强推送系统 |

边界：

不以 AI 聊天记账替代核心表单。AI 可作为远期辅助，但核心路径必须稳定、可预测、离线友好。

### 4.3 P0/P1：导入与分类标签智能化

| 需求 | 用户价值 | 竞品启发 | LedgerTwo 计划 |
|---|---|---|---|
| 微信/支付宝/通用导入 | 降低历史补账成本 | 国内记账产品适配本土支付；Actual/Firefly 重视导入 | 保持 preview -> adjust -> commit，不直接写正式账单 |
| 重复与疑似重复 | 防止污染账本 | Firefly 规则和导入清洗 | 保留 import hash、状态解释和批次追踪 |
| 默认分类/标签包 | 新用户无需从空白开始 | 钱迹/随手记分类模板 | Task53 继续推进 profile、幂等应用和可预览 |
| 分级自动化 | 越用越省事 | MoneyWiz/Copilot 自动分类，Firefly 规则 | 人工调整 > 用户规则 > 显式学习 > 内置知识 > 其他 |
| 批量修正 | 导入后快速整理 | 桌面财务工具常见 bulk edit | Task53 后续做批量分类/标签/账户调整 |

边界：

不做隐式学习，不调用云端 LLM，不自动提交内置建议。所有自动化必须可解释、可回滚、可审计。

### 4.4 P1/P2：共同支出、多人分摊和结算解释

| 需求 | 用户价值 | 竞品启发 | LedgerTwo 计划 |
|---|---|---|---|
| paid/share/net 解释 | 用户知道为什么欠钱 | Splitwise 的 who owes who | 结算页显示支付、承担、已结算、最终净额 |
| 结算文案复制 | 降低沟通成本 | Settle up 心智 | 继续保留复制文案，自动通知延后 |
| 多人 equal/amount/ratio/shares | 覆盖旅行、装修、合租 | Splitwise / Tricount / Settle Up 多人分摊 | Task51 需要真实证据后冻结范围 |
| 最小转账建议 | 减少多笔转账 | AA 产品常见能力 | v1.4 评审；必须保留解释路径 |
| 结清后历史处理 | 已结算不扰民但可追溯 | Splitwise 会隐藏已结清交易但不删除 | 可做“已结算历史折叠”，不删除历史账单 |

边界：

第三成员和多人分摊不能提前压进双人主路径。Task51P 真实证据不足时，只做研究，不动 schema/router。

### 4.5 P2/P3：轻预算、报告和长期复盘

| 需求 | 用户价值 | 竞品启发 | LedgerTwo 计划 |
|---|---|---|---|
| 月度支出和分类趋势 | 看清钱花在哪里 | YNAB/Firefly/MoneyWiz 报表 | 先做消费复盘，不做完整预算方法论 |
| 分类预算 | 控制生活开销 | 鲨鱼月度/分类预算 | 可做轻预算：月度总额、分类额度、剩余可花 |
| 周期账单预测 | 预见固定支出 | MoneyWiz calendar forecast | 基于周期规则生成未来提醒，不自动入账 |
| 年报/月报 | 长期使用奖励 | 成熟财务产品都有复盘 | v1.5 建议做家庭月报、年度趋势和共同支出复盘 |
| 备份健康报告 | 私有化信任感 | 自托管产品数据控制 | 显示最近备份、恢复演练状态和导出状态 |

边界：

不做投资组合、复杂资产负债、银行聚合和复杂复式会计。

## 5. 当前受众与产品定位

### 5.1 第一目标用户

1. 情侣、夫妻：共同生活支出、阶段性结算、隐私账单和长期数据保存。
2. 合租伙伴：房租、水电、日用品和公共支出的分摊。
3. 小家庭：多账本、多成员、孩子/装修/旅行等小范围账本。
4. 重视数据自主的技术用户：愿意使用 NAS、Docker、备份和私有部署。
5. 有微信/支付宝历史账单的人：希望导入、去重、归类，而不是从零手记。

### 5.2 非第一目标用户

1. 只需要极简个人记账、不关心共同账本的人。
2. 需要银行自动同步、投资净值和全资产聚合的人。
3. 需要企业报销审批、员工权限和经营账本的人。
4. 不愿维护私有部署，也不关心数据自主的人。
5. 需要复杂预算方法论的人。

## 6. 短中长期产品计划

### 6.1 短期：收口可信与分类智能

目标：让当前 v1.3 候选成为可信、可迁移、可继续迭代的基础。

重点：

1. 保持 v1.2/NAS 发布线独立维护，不混用本机 v1.3/Task53 staging。
2. 完成 Task53 分类/标签智能化闭环：默认模板、候选读取、预览持久化、reclassify、显式学习、批量调整和解释。
3. 保持 Fresh Light 已完成基线，不借 UI 调整改变业务规则。
4. 继续保证 10 秒普通记账、20 秒共同支出。
5. 保持第三成员、多人分摊、通知、OCR、银行同步延后。

### 6.2 中期：省时间和轻家庭账本

目标：让用户连续 30 天使用，且越用越省事。

重点：

1. 批量整理：导入后批量分类、标签、账户、可见性。
2. 轻量日历：补账、周期提醒、未来固定支出查看。
3. 结算解释增强：结算影响、历史折叠、复制文案。
4. 多账本场景模板：家庭、旅行、装修、合租。
5. 轻报表：分类趋势、标签趋势、成员支付/承担、共同支出趋势。
6. 备份健康：最近备份、恢复演练、导出提醒。

### 6.3 长期：小范围成员共同账本

目标：从双人账本演进为小家庭/小团队的可信共同账本。

重点：

1. 多成员正式化：第三成员、角色权限、成员生命周期。
2. 多人分摊：equal、amount、ratio、shares。
3. 最小转账建议和解释。
4. 月报、年报和长期趋势。
5. 恢复演练向导和迁移工具。
6. 可选轻预算：月度总预算、分类预算、目标储蓄。

长期仍不建议默认进入：

1. 银行自动同步。
2. OCR 自动入账。
3. 原生 App。
4. 投资资产管理。
5. 企业多租户。

## 7. 架构优化判断

### 7.1 不需要推倒重写

当前 Go + SQLite + REST JSON + React/Vite + Docker/NAS 的架构仍适合 LedgerTwo 的私有化家庭账本定位。

不建议：

1. 拆微服务。
2. 引入复杂消息队列。
3. 把 SQLite 提前换成 PostgreSQL。
4. 引入银行聚合或云 AI 作为核心依赖。
5. 用原生 App 重写移动端。

### 7.2 需要渐进优化的架构点

| 方向 | 问题 | 优化建议 | 触发时机 |
|---|---|---|---|
| 后端领域模块 | transaction 模块承担过多业务 | 渐进拆出 template、recurring、importer、metadata、settlement explanation、audit | 每次触及对应功能时顺手拆，不做大爆炸重构 |
| 导入管线 | parser、normalizer、classifier、dedupe、commit 容易耦合 | 固化 TabularReader、Normalizer、Classifier、CommitService 分层 | Task53 后续继续推进 |
| 规则与学习 | 用户规则、显式学习、内置知识需要统一解释 | 建立分类来源、置信度、命中理由 DTO | Task53.4A 之后 |
| 前端 features | 页面和组件仍有混合 | 按 `features/transactions`、`features/imports`、`features/settlement`、`features/metadata` 渐进迁移 | 新页面或大改时 |
| API 契约 | OpenAPI 与实际 handler 需要持续同步 | 每个版本冻结 API diff，生成或校验类型 | v1.3 后持续执行 |
| 权限与审计 | 多账本、多成员后风险增加 | 将 LedgerContext、RBAC、审计作为所有 service 的强前置 | Task51 前必须复核 |
| 部署与数据 | 多条 staging 线容易混淆 | 固定 runtime root、端口、schema、channel 和发布记录 | 每个 schema 变更前 |
| 性能 | 前端包体和列表性能会随功能增长变差 | route-level lazy loading、表格虚拟化、查询缓存分层 | 报表/导入工作台继续扩大前 |

### 7.3 架构边界建议

1. 业务规则只在后端 service 层做最终判断，前端只做预览和交互辅助。
2. 金额、分摊、结算、去重、权限、导入提交不能在前端成为权威。
3. 任何 schema 变更必须有 migration 评审、回滚策略、staging 隔离和数据守恒检查。
4. 任何智能分类都必须可解释、可关闭、可回滚。
5. 多人分摊进入前，必须先有真实场景证据和冻结的权限矩阵。

## 8. 推荐优先级总表

| 优先级 | 方向 | 具体事项 | 原因 |
|---|---|---|---|
| P0 | 数据安全 | staging/production 隔离、备份、回滚、审计 | 私有账本信任根基 |
| P0 | Task53 | 默认分类/标签、分类器、显式学习、批量调整 | 降低导入和日常整理成本 |
| P1 | 快速记账 | 默认值、模板、保存并继续、移动端细节 | 决定日活和留存 |
| P1 | 导入体验 | preview 解释、批量修正、重复处理 | 决定历史补账能力 |
| P1 | 结算解释 | paid/share/net、结算文案、历史折叠 | 决定共同账本信任 |
| P2 | 轻报表 | 月报、分类趋势、成员趋势 | 决定长期复盘价值 |
| P2 | 场景模板 | 家庭、旅行、装修、合租 | 支撑 v1.4 多场景扩展 |
