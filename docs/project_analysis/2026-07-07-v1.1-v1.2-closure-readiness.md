# LedgerTwo v1.1 收口与 v1.2 文档就绪判断

日期：2026-07-07

## 1. 判断结论

### 1.1 v1.1 是否可以继续收口

可以继续收口，但不建议直接冻结。

理由：

1. 代码层面 Task41-Task46 的主体功能已有落点。
2. 本机 WSL2 部署可用，`/api/healthz` 返回 `version=1.1.0-rc`、`schema_version=12`、`db=ok`。
3. 移动端关键入口已完成 Chrome CDP 截图验收，覆盖 375px/390px/430px，横向溢出 0，React Router 错误页 0。
4. 本轮验收发现的设置页空备份列表崩溃和移动端顶栏挤压已修复并提交。

不能直接冻结的原因：

1. 普通支出、共同支出、保存并继续记、复制一笔、模板生成账单仍需 UI 实际提交闭环。
2. 附件上传与受控读取、手动备份创建/下载、恢复前二次确认仍需深度验收。
3. NAS 地址下需要复核版本、schema、登录和关键入口。
4. v1.1 PRD 的验收清单尚未全部改成带证据的完成状态。

## 3. v1.1 后续收口建议

下一步建议不再扩大 v1.1 功能，只做验收和阻断缺陷修复：

1. UI 实际提交普通支出和共同支出。
2. 验证保存并继续记、复制一笔、模板生成账单。
3. 验证登记结算、复制结算文案、结算历史。
4. 验证附件上传与受控读取。
5. 验证手动备份创建、下载、恢复前确认。
6. 在 NAS 地址复核 healthz、登录、Dashboard、设置页和关键操作入口。
7. 更新 `docs/prd/22-prd-v1.1-trust-and-daily-use.md` 的验收状态。

## 4. v1.2 进入开发前检查

Task47 启动前必须确认：

1. v1.1 已冻结或明确不阻塞 v1.2 的剩余问题。
2. `docs/tech/20-v1.2-import-implementation-contract.md` 中 open questions 至少对 Task47 不构成阻塞。
3. API Inventory 和 OpenAPI 准备随 Task47 更新。
4. migration 不修改历史文件，只新增。
5. 微信、支付宝、通用 CSV fixture 准备完成。
6. 导入提交入口默认可关闭，preview 可独立运行。

## 5. 阶段判断

当前项目处于：

```text
v1.1 收口验收后半段
-> 完成 UI 实际提交和 NAS 复核
```

预计：

- v1.1 若无阻断缺陷，仍需 1 次以 UI 实际提交和 NAS 复核为主的会话完成冻结。
- v1.2 文档已经可以支持开发前评审；正式开发时仍需每个 task 随代码补 OpenAPI、migration 和测试 fixture。
