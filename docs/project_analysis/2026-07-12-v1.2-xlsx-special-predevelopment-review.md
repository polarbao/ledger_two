日期：2026-07-12

实施更新（2026-07-14）：Task49X 已按微信 CSV/XLSX、支付宝 CSV、通用 CSV 完成开发和本机验收；支付宝 XLSX 不在当前范围。真实微信 XLSX 与支付宝 CSV 仅在本机 WSL staging 预览，未 commit、未访问 NAS。最新证据见 `docs/project_analysis/2026-07-12-local-wsl-xlsx-csv-preview-acceptance.md`。

## 1. 触发原因

当前微信真实账单文件为 `.xlsx`，已只读确认包含 276 条数据，表头位于第 18 行。当前已部署的 `1.2.0-rc/schema 18` 只使用 Go `encoding/csv`，前端也只接受 `.csv`，无法可信处理该文件。

因此 v1.2 RC 从“冻结”调整为“范围重新打开”，插入 Task49X。该变化只扩展输入格式，不改变现有导入的预览、规则、去重、人工确认、事务提交和 owner-only 边界。

## 2. 已冻结的产品决定

1. 微信支持 CSV/XLSX；支付宝和通用模板保持 CSV。
2. 不支持 xls、xlsm、加密工作簿、任意 Excel 映射、OCR、PDF 和银行同步。
3. 文件上限 2 MiB，单批 2000 行，表头扫描前 50 行，最多 10 张工作表、64 列。
4. XLSX 必须先 preview，不增加直接提交入口。
5. 工作表和表头自动识别必须唯一；歧义时拒绝，不猜测。
6. 金额、日期和订单号必须经过精度保护；公式、宏、合并或隐藏数据不得静默采信。
7. 相同流水的 CSV/XLSX 必须生成相同 import_hash。
8. 真实微信文件先在 staging 完成 276 行数量守恒和抽样对账，用户确认后才考虑 production commit。

## 3. 已冻结的技术决定

1. 采用 `github.com/xuri/excelize/v2`，实现时固定 v2.10.1。
2. 不采用维护投入较弱的 `tealeg/xlsx/v3`，不使用标准库自研 OOXML parser。
3. 增加 TabularReader，把 CSV/XLSX 容器读取与来源标准化分离。
4. 复用 `/api/imports/preview`，后端推断 file_format。
5. 新增 migration 019，为 import_batches 增加 file_format 和 parser_metadata_json。
6. file_format 和 parser metadata 不参与 import_hash。
7. 增加独立 XLSX 开关，发生问题时只关闭 XLSX，CSV 主链路继续工作。

依赖影响：Excelize 为 BSD-3-Clause 纯 Go 库，通过 Go modules 管理；本项目不使用 CMake/vcpkg，两者无变更；Docker 构建会增加 Go module 下载和编译缓存。
