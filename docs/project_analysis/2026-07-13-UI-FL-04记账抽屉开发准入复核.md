## 3. 现状与改造原因

当前抽屉已经闭环支持普通支出、收入、共同支出、默认值、复制、模板、附件、离线草稿和保存并继续，但 `TransactionFormDrawer.tsx` 已达到约 1640 行，查询、mutation、业务编排和大量内联视觉样式集中在同一组件。

主要体验与维护问题：

1. Dark Glass 局部颜色、渐变和内联布局较多，不能稳定复用 Fresh Light Token。
2. 高频和低频字段没有清楚分层，模板管理与提交动作竞争注意力。
3. `handleClose` 当前直接关闭并清理上下文，没有对用户修改后的未保存内容进行统一确认。
4. 桌面侧抽屉和移动底部 Sheet 的结构主要依赖旧 CSS，焦点返回和关闭路径需要统一。
5. 交互测试以纯函数和历史 E2E 为主，缺少对新分层、脏字段关闭和 Footer 状态的组件契约覆盖。

## 4. 冻结行为

以下行为在 UI-FL-04 中只能迁移呈现，不能改变：

- 金额在 API 和状态层继续使用整数分，表单只在输入/显示层使用元。
- owner/editor 可记账，viewer 不可提交。
- 默认值优先使用服务端 `transaction-defaults`，再回退到既有本地最近值。
- 普通账单可使用账户；共同支出不提交账户，默认两人参与并 equal 分摊。
- 复制、模板和草稿继续回填现有字段；附件上传路径和失败处理不变。
- 离线提交保存为本地草稿，不伪装成正式账单。
- 保存并继续沿用 `buildContinueTransactionFormValues`，清空金额、标题、备注和附件，保留既定分类、账户、标签、付款人、分摊方式、日期和可见性，并重新聚焦金额。
- 成功提交继续失效 Dashboard 与流水 query；模板、默认值和草稿的既有失效/清理行为保持不变。

## 5. 组件拆分边界

建议按展示责任拆分，不下放请求：

| 组件 | 责任 | 禁止承担 |
|---|---|---|
| `TransactionFormDrawer` | query、mutation、RHF、默认值、复制/模板/草稿 orchestration | 局部主题常量 |
| `TransactionAmountSection` | 金额、类型、金额错误和首焦点 | 金额持久化、API 转换 |
| `TransactionCoreFields` | 分类、账户、日期与最近分类 | 自行读取 query |
| `SharedExpenseSection` | 付款人、分摊方式、参与人和承担预览 | 自行计算服务端最终结算 |
| `TransactionAdvancedFields` | 标题、标签、可见性、备注、附件 | 模板 mutation |
| `TransactionTemplateActions` | 选择、保存和管理模板的展示入口 | query invalidation |
| `TransactionFormFooter` | 取消、保存并继续、保存/离线草稿状态 | 直接提交 API |

拆分不是一次性重写。先冻结纯函数与 props，再迁移区块，最后删除对应内联样式；每一步保持主组件可运行。

## 6. 交互状态矩阵

| 场景 | 必须验证 |
|---|---|
| 普通支出 | 默认值加载、只填金额、账户、保存、金额聚焦 |
| 共同支出 equal | 两人参与、承担预览、付款人变化 |
| 共同支出 payer_only | 只有付款人承担、预览随切换更新 |
| 保存并继续 | 连续三笔、一次性字段清空、保留字段正确、重新聚焦 |
| 复制与模板 | 低频区自动展开、来源提示、模板保存/编辑/归档/恢复 |
| 离线草稿 | 保存文案、创建/更新草稿、恢复后正式提交 |
| 权限 | viewer 无提交入口，角色变化后不泄露写能力 |
| 附件 | 上传中禁用相关动作、失败可重试、关闭不丢失未确认修改 |
| 脏字段关闭 | 关闭按钮、遮罩、Escape 三条路径共用一个确认框 |
| API 失败 | 保留表单值、显示可恢复错误、不误报成功 |
