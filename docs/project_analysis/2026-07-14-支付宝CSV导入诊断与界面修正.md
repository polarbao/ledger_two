# 支付宝 CSV 导入诊断与 UI 修正记录

状态：源码修正与自动化验证完成；固定后端镜像运行复验待执行<br>
日期：2026-07-14

## 1. 检查对象与数据边界

只读检查了 `data/` 下两份支付宝官方 CSV，并使用当前 parser 执行预览级验证。未把真实文件加入 Git，未把账单内容写入文档，未调用 production/NAS，未执行正式 commit。

本机 `data/ledger.db` 中可见一次用户触发的失败批次。该批次 `transaction_import_refs=0`，说明提交事务已回滚，没有形成该批次的正式导入记录；数据库 `PRAGMA quick_check=ok`。

## 2. 根因

两份文件都可被识别为 `alipay + CSV`，表头位于第 22 行，分别解析 1232 条和 617 条。失败不是文件编码、表头或支付宝格式不兼容，而是预览与提交校验不一致：

1. parser 将 `0.00` 识别为合法金额，预览显示为普通 `new` 行。
2. commit 契约要求待导入金额必须大于 0，因此在提交时才拒绝第一条零金额流水。
3. 批次随后只记录 `failed_rows=1`，对应预览行没有被改成 `failed/invalid`，重新打开页面后无法从卡片定位失败原因。
4. 前端将 617/1232 条全部放在同一滚动列表，摘要不可筛选；即使存在行级错误，也很容易被埋在长列表中。
5. 旧前端还禁止对 `row_status=failed` 的无效行执行“跳过”，与后端允许跳过 invalid 行的契约冲突。

## 3. 样本结论

修正后只读 parser 结果：

| 文件范围 | 解析行数 | 无效零金额行 | 疑似重复 | 业务跳过 |
|---|---:|---:|---:|---:|
| 2026-01-01 至 2026-07-01 | 1232 | 11 | 6 | 89 |
| 2026-04-14 至 2026-07-14 | 617 | 7 | 0 | 35 |

零金额行不应生成金额为 0 的正式账单。它们现在会在 preview 阶段显示为 `invalid + failed`，错误原因是“金额必须大于 0”，用户可逐条筛选并跳过。较长文件中的 6 条疑似重复仍需人工选择“确认导入”或“跳过”。

因此，两个文件都能进入预览；在跳过无效零金额行、处理疑似重复后，才满足正式 commit 前置条件。本轮未执行正式导入，不能把“预览和门禁通过”表述为“真实账单已导入”。

## 4. 修正内容

- 后端在 parser 阶段将待落库的非正金额标成 `IMPORT_ROW_AMOUNT_INVALID`，并补齐空标题的行级错误保护。
- 前端摘要卡可直接筛选全部、新增、疑似、无效和跳过行。
- preview 存在待处理行时默认显示“需要处理”，提交被阻断时自动切换并滚动到待处理列表。
- 无效行卡片展示行号、错误码和原因，并允许“跳过此错误行”。
- batch 级提交失败使用独立危险提示，不再暗示一定存在可定位的 failed 行。
