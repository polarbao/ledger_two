# 分类、标签与自动归类竞品研究

状态：Task53 产品设计输入，不构成竞品功能照搬或开发完成证明<br>
研究日期：2026-07-16<br>
研究范围：LedgerTwo 当前实现、钱迹、小星记账、Actual Budget、Firefly III

## 1. 研究目标

为 LedgerTwo 的导入自动分类、自动标签、默认元数据和用户反馈学习建立可追溯设计依据。研究只提取可验证的产品机制，不复制竞品文案、视觉资产、私有实现或商业限制。

## 2. LedgerTwo 当前事实

| 能力 | 当前状态 | 主要缺口 |
|---|---|---|
| 分类/标签管理 | Owner 可新增、编辑、排序、归档、恢复；历史引用保留 | 默认数据不完整；缺少系统兜底分类保护、合并/迁移语义 |
| 导入规则 | 支持商户包含、描述包含、来源账户、金额区间 | 只取第一条命中规则；缺少精确商户、规则来源和可解释优先级 |
| 分类/标签推荐 | 规则可写入 `suggested_category_id` 和 `suggested_tag_ids` | 推荐不等于自动选择；没有置信度、候选来源和批次覆盖率 |
| 用户修正 | 可逐行调整分类和标签 | 修正不会形成可复用学习规则；没有“记住此商户” |
| 初始化 | 首个账本创建 5 个支出分类和 1 个“系统默认”标签 | 无收入分类；标签不可用；Task50 新建账本不创建任何默认元数据 |
| 隐私 | 全部本地处理，无云端分类服务 | 应继续保持，不为自动分类上传账单文本 |

## 3. 闭源产品观察

### 3.1 钱迹

可验证机制：

1. 分类是账单统计的主维度，账单必须具有分类。
2. “其他”作为不可删除的默认分类，避免历史账单失去归属。
3. 分类支持名称、图标、排序和自定义。
4. 标签默认可关闭，以降低首次使用复杂度。
5. 标签支持分组、颜色、归档和历史引用保留；多标签用于成员、项目、购物属性等横向维度。
6. 标签支持导入导出。

LedgerTwo 可吸收：

- 保留支出/收入系统兜底分类，并阻止无替代方案的归档。
- 分类承担唯一主口径，标签承担可选横向语义，不能用标签替代分类。
- 默认标签包保持精简；允许用户归档，不把高级标签体系强塞给所有账本。

不直接照搬：

- 不引入会员数量限制。
- 首期不做标签分组和 emoji 专项。
- 不采用删除分类后静默迁移至“其他”的不可逆行为，LedgerTwo 优先归档或显式合并。

### 3.2 小星记账

可验证机制：

1. 首次识别新商家/商品时要求人工选择类别，后续同类信息会自动记忆。
2. 可配置首次识别时的默认类别和默认账户。
3. 标签、商家、角色和备注也可按商户记忆。
4. 导入时先建立来源账户和类别关联；相同来源类别可批量修改，后续导入自动关联。
5. 对缺失类别可自动创建，但官方同时提示第三方导入只是历史迁移近似值。

LedgerTwo 可吸收：

- 用户修正后显式提供“记住此商户”的低成本学习闭环。
- 同一批次支持按相同商户/来源类别批量应用，而不是要求逐行重复调整。
- 自动结果必须可见、可修改，并保留导入仍需确认的安全边界。

不直接照搬：

- 不允许导入文件中的任意文本静默创建大量分类或标签。
- 不依赖固定中文分类名称完成核心识别，避免用户重命名后失效。

## 4. 开源产品观察

### 4.1 Actual Budget

可验证机制：

1. 导入交易按有序规则执行，条件可基于原始导入商户、账户、分类、日期、备注和金额。
2. 字符串匹配不区分大小写，并区分原始导入商户与清洗后的商户。
3. 用户反复分类或重命名商户后，系统可以自动创建或更新规则。
4. 自动生成的规则仍归用户所有，可以查看、修改、禁用。
5. 规则按阶段和具体程度排序，后执行的更具体规则可覆盖前面的广泛规则。

LedgerTwo 可吸收：

- 学习结果落为普通、可审阅、可归档的规则，而不是不可解释模型状态。
- 保留原始导入字段用于匹配，不能让前一条规则改写后的文本破坏后续判断。
- 规则优先级要由明确来源、精确度和用户优先级共同决定。

首期不采用：

- 不做通用 AND/OR/正则表达式规则编辑器。
- 不做 pre/default/post 三阶段和任意字段动作。
- 不允许规则修改金额、日期或交易类型。

### 4.2 Firefly III

可验证机制：

1. 规则由 trigger 与 action 组成，可在创建、修改或导入时执行。
2. 规则可以设置分类、预算、标签、描述和账户，也能应用于既有交易。
3. 规则分组和严格/非严格匹配提供较高表达能力。

LedgerTwo 可吸收：

- 条件与结果在领域模型上分离。
- 规则必须提供命中解释，并允许对历史记录执行前先预览。

首期不采用：

- 不建设通用自动化平台。
- 不对正式历史账单自动批量回写。
- 不开放删除交易、改金额或转换交易类型等高风险动作。

## 5. 推荐产品模式

采用用户已确认的“分级自动化”：

```text
手工行调整 / 批量调整
  > 用户显式规则
  > 用户确认后生成的学习规则
  > 本地内置词典与来源映射
  > 系统兜底分类
```

1. 用户显式规则和精确学习规则可自动填入预览行，但不会自动提交批次。
2. 内置词典只提供建议，不直接覆盖用户选择。
3. 系统兜底分类保证每条可导入收支都有分类，但必须单独统计为“兜底”，不能伪装成识别成功。
4. 标签只有高置信规则可以自动添加；中低置信标签必须由用户批量或逐行接受。
5. 所有结果显示来源、原因和置信度，并允许用户恢复为空或改为其他值。
6. 首期使用本地确定性规则，不引入云端 AI、LLM、向量数据库或第三方账单分类 API。
