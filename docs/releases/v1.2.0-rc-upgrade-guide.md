# LedgerTwo v1.2.0-rc 升级与回滚说明

适用目标：从现有 v1.0/v1.1 或 schema 12 环境升级到 `v1.2.0-rc` / schema `18`

范围提示（2026-07-12）：本文仅适用于当前已部署的 schema 18 基线。Task49X 计划新增 migration 019；在代码实现、staging 验收和本文更新前，不要根据规划手工执行 migration 019 或让 production 接收 XLSX。

推荐方式：先本机验收，再在确认的 NAS 维护窗口升级

NAS 上必须先按 `docs/tech/23-v1.2-deployment-environment-isolation.md` 建立 staging/production 物理隔离，不直接用候选构建覆盖唯一正式实例。

## 1. 升级影响面

启动新版本时 Goose 会自动顺序执行 migration 13-18：

1. 扩展导入批次与预览行字段、统计和索引。
2. 增加行级建议值、用户选择值和可见性字段。
3. 新增 `transaction_import_refs` 去重映射表。
4. 增加导入批次失败、提交和过期字段。
5. 扩展导入规则名称、匹配方式、金额范围、优先级、结果和归档状态。
6. 增加预览行命中规则与推荐原因字段。

这些 migration 包含 SQLite 追加列操作，不能依赖简单降级脚本完整恢复旧结构。因此回滚必须恢复升级前数据库副本，并同时使用旧镜像。

## 2. 升级前检查

1. 确认当前服务可登录，现有账单、结算和附件可读。
2. 访问 `/api/healthz`，记录当前 `version`、`schema_version` 和 `db`。
3. 在设置页执行“手动备份”，下载备份文件并在 NAS 外保存一份。
4. 备份 `data/ledger.db`、`uploads/`、`.env` 和 `docker-compose.yml`。
5. 确认备份文件大小非 0，并保留升级前镜像或源代码提交号。
6. 记录当前容器和镜像信息：

```bash
docker compose ps
docker image inspect ledger-two:latest --format '{{.Id}} {{.Created}}'
```

应用启动时生成的 `ledger.db.pre_migrate_v<旧版本>.bak` 只能作为附加保护，不能替代上述手动备份和 NAS 外副本。

## 3. 本机候选版本验证

在仓库根目录执行：

```bash
cd backend && go test ./... -count=1
cd ../frontend && pnpm lint && pnpm test && pnpm build
cd .. && docker compose config
docker compose build
docker compose up -d
```

验证：

```bash
curl http://localhost:38088/api/healthz
docker compose logs --tail=100 ledger-two
```

预期健康检查包含：

```json
{"success":true,"data":{"db":"ok","schema_version":18,"status":"ok","version":"1.2.0-rc"}}
```

## 4. NAS 升级步骤

1. 将候选代码或已验证镜像同步到 NAS 部署目录，不复制本机 `data/`、`backups/`、`uploads/` 或 `.env`。
2. 在 NAS 部署目录再次确认升级前备份存在。
3. 使用源码构建部署时执行：

```bash
docker compose config
docker compose down
docker compose up -d --build
docker compose ps
docker compose logs --tail=100 ledger-two
```

4. 使用镜像仓库部署时先固定候选镜像标签，再执行 `docker compose pull` 和 `docker compose up -d`；不要用无法追溯的浮动标签作为唯一回滚依据。
5. 检查 `/api/healthz` 返回 `version=1.2.0-rc`、`schema_version=18`、`db=ok`。

从既有 Compose project 切换到隔离后的 production project 时，推荐使用带自动 health 校验和失败回滚的脚本：

```bash
sudo sh deploy/nas/promote-production.sh \
  /volume1/docker/ledger-two \
  /volume1/docker/ledger-two/backups/predeploy/<final-backup>/ledger.db
```

脚本会保留旧容器、验证候选镜像、等待 version/schema/channel/db 全部符合预期；候选启动失败时会保存失败数据库、恢复 schema 12 备份并重新启动旧容器。

## 5. 升级后业务验收

1. Owner 登录和退出正常，原有账单、结算、附件仍可读取。
2. 设置页系统诊断显示数据库、备份、上传和日志目录正常。
3. 上传一份小型 fixture 执行导入预览，确认预览阶段正式账单数量不变化。
4. 检查 duplicate、suspicious、invalid 状态文案和人工调整。
5. 创建一条临时导入规则，确认命中解释；归档后重新预览不再命中。
6. 使用专门的验收 CSV 提交一次导入，核对 imported/skipped/failed 数量与正式流水。
7. 再执行一次手动备份，确认升级后的 schema 18 数据可生成备份。
8. 重启容器后复查健康检查和数据持久化。

## 6. 回滚触发条件

以下任一情况应停止使用新版本并回滚：

1. Migration 失败或 health 不是 schema 18。
2. 原有账单、结算或附件不可读取。
3. 登录、备份或数据库写入出现阻断错误。
4. 导入提交出现部分写入或无法解释的数据变化。

## 7. 回滚步骤

1. 停止当前容器：`docker compose down`。
2. 保留故障现场，将当前 `data/ledger.db` 重命名为带时间戳的故障副本。
3. 将升级前备份恢复为 `data/ledger.db`，同时恢复匹配的 `uploads/`（如升级期间写入过附件）。
4. 将 compose 镜像或源码恢复到升级前提交。
5. 启动旧版本并确认 health、登录、账单、结算、附件和备份。
6. 不要让旧版本直接连接已经迁移到 schema 18 的数据库。

回滚属于生产数据操作，正式执行前必须由维护者确认目标备份、目标镜像和数据目录。
