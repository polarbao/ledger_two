# LedgerTwo v1.3.0-rc 升级与回滚指南

适用来源：`1.2.0-rc / schema 19`  
目标：`1.3.0-rc / schema 21`

## 升级前

1. 确认目标是独立 staging 或已批准的 production 维护窗口。
2. 停止写入并使用 SQLite `.backup` 创建一致性数据库副本。
3. 备份 `uploads/`、Compose 和脱离仓库保存的 `.env`。
4. 记录旧镜像 ID、Git revision、health、schema、核心数量与金额汇总。
5. 对数据库和附件包计算 SHA-256，并在副本执行 `PRAGMA quick_check`。
6. 确认源 schema 精确为 19；schema 18、未版本化业务库或高于 21 的数据库必须停止。

## 独立 staging

仓库提供：

```text
deploy/v13/docker-compose.staging.yml
deploy/v13/env.staging.example
deploy/v13/verify-staging.sh
deploy/v13/rollback-staging.sh
```

本机默认使用 38091，不能复用 v1.2 的 38088 容器、Compose project 或数据目录。候选镜像必须使用带提交号的固定标签，禁止仅使用 `latest`。

执行：

```bash
ENV_FILE=/private/path/.env sh deploy/v13/verify-staging.sh
```

成功条件：

```text
version=1.3.0-rc
schema_version=21
deployment_channel=staging
db=ok
import_xlsx_enabled=true
PRAGMA quick_check=ok
```

## 数据守恒

至少比较升级前后：

```text
users / ledgers / ledger_members
transactions count + SUM(amount)
settlements count + SUM(amount)
import_batches / import_items / transaction_import_refs
非空 attachment_paths 引用数
```

migration 020/021 只新增生命周期、版本、权限和审计结构，上述业务数量与金额必须完全一致。

## 成对回滚

旧 v1.2 镜像不得连接 schema 21。回滚必须同时：

1. 停止 v1.3 容器。
2. 将失败的 schema 21 数据库、WAL 和 SHM 移到独立故障目录。
3. 校验升级前 schema 19 备份的 SHA-256 和 quick_check。
4. 恢复 schema 19 完整数据库。
5. 启动记录过 ID 的固定 v1.2 镜像。
6. 核验 `1.2.0-rc / schema 19 / db ok` 和历史核心数据。

示例：

```bash
ENV_FILE=/private/path/.env \
ROLLBACK_DATABASE=/private/path/ledger-schema19.db \
ROLLBACK_IMAGE=ledger-two:1.2.0-rc-<fixed-tag> \
sh deploy/v13/rollback-staging.sh
```

production 禁止执行 `goose down`。若需要保留回滚后产生的新写入，必须另行设计向前修复，不能把 schema 21 文件交给旧镜像。

## NAS

本指南不构成 NAS 部署授权。NAS production 升级必须另行确认维护窗口、外部备份位置、回滚负责人和公网访问策略。
