## 1. 结论

v1.2 的产品方向已经闭环：目标是可控 CSV 导入、预览、去重、人工确认和规则推荐。

但 v1.2 在进入开发前必须补齐实施契约，避免沿用早期 `/api/transactions/import/*` 的临时接口继续扩散。本文作为 Task47-Task49 的技术冻结补充。

## 2. 范围

v1.2 必做：

1. 微信、支付宝、通用 CSV 的基础解析。
2. 上传后生成预览批次，不写正式 `transactions`。
3. 行级字段映射、状态、错误和调整。
4. 去重判断：`new`、`duplicate`、`suspicious`、`invalid`。
5. 用户确认后事务落库。
6. 导入规则只做推荐，不自动提交。
7. 导入批次、行、提交结果可追踪。
8. 导入提交写审计日志。

非目标：

1. OCR、PDF、xlsx、银行同步。
2. 后台异步任务系统。
3. 自动读取微信或支付宝账号。
4. 自动提交 suspicious 行。
5. 自动导入 settlement、transfer 或独立 refund 类型。

## 3. API 迁移策略

当前代码和 API 清单中已有早期接口：

```text
POST /api/transactions/import/parse
POST /api/transactions/import/analyze
POST /api/transactions/import/commit
GET  /api/import-rules/
POST /api/import-rules/
DELETE /api/import-rules/{id}
```

v1.2 新开发应以导入资源为中心，目标接口为：

```text
POST  /api/imports/preview
GET   /api/imports/{batch_id}
PATCH /api/imports/{batch_id}/rows/{row_id}
POST  /api/imports/{batch_id}/commit

GET   /api/import-rules
POST  /api/import-rules
PATCH /api/import-rules/{id}
POST  /api/import-rules/{id}/archive
POST  /api/import-rules/{id}/restore
```

兼容策略：

1. Task47 可保留早期 `/api/transactions/import/*` 作为兼容入口，但新前端优先调用 `/api/imports/*`。
2. 旧接口若继续存在，必须标记为 transitional，并委托新 `internal/importer` service。
3. 不允许在 `transaction` handler 内新增导入业务判断；handler 只做参数解析和响应。
4. Task49 将 `DELETE /api/import-rules/{id}` 视为历史兼容，正式行为使用 archive/restore。

## 4. 权限与账本上下文

导入是批量写入能力，v1.2 默认权限：

| 角色 | 预览 | 调整行 | 提交 | 管理规则 |
|---|---:|---:|---:|---:|
| owner | yes | yes | yes | yes |
| editor | no | no | no | no |
| viewer | no | no | no | no |

规则：

1. 所有 `/api/imports/*` 和 `/api/import-rules*` 必须要求 `X-Ledger-Id`。
2. 非账本成员返回 `FORBIDDEN` 或 `NOT_FOUND`，不得泄露批次存在性。
3. 导入批次只属于当前账本，不允许跨账本读取。
4. 创建正式账单时复用 transaction service 的权限、金额、可见性、分类、账户和 split 校验。

3. result_json 中引用的分类、账户、标签在应用前必须重新校验未归档或给出替换提示。

## 6. 状态机

### 6.1 批次状态

```text
previewing -> ready -> committed
previewing -> failed
ready -> failed
ready -> expired
```

说明：

- `previewing`：文件已上传或正在解析，尚不可提交。
- `ready`：预览行已生成，可调整和提交。
- `committed`：已完成事务落库。
- `failed`：解析或提交失败。
- `expired`：预览超过保留时间，不再允许提交。

### 6.2 行状态

```text
pending -> adjusted
pending -> skipped
pending -> imported
adjusted -> skipped
adjusted -> imported
pending/adjusted -> failed
```

提交规则：

1. `invalid` 行不能提交，除非用户修正为合法状态。
2. `duplicate` 行默认 skipped。
3. `suspicious` 行必须用户明确选择导入或跳过。
4. `skipped` 行不写入 `transactions`。
5. 任一待导入行写入失败，整个提交事务回滚。

## 8. Parser 与标准化

Parser 分层：

```text
parser/wechat
parser/alipay
parser/generic
normalizer
dedupe
rule
commit
```

标准化规则：

1. CSV 编码至少兼容 UTF-8 和常见中文导出编码；无法识别时返回 `IMPORT_FILE_INVALID`。
2. 金额解析必须转整数分，禁止 float 参与最终金额计算。
3. 微信/支付宝的支出映射为 `expense`，收入映射为 `income`。
4. 退款映射为 `income + 退款分类建议`。
5. 转账映射为 `unknown` 或默认 `skipped`，不得自动入账。
6. 时间统一转换为 ISO8601，无法识别则行级 `IMPORT_ROW_INVALID`。

## 9. 去重规则

`import_hash` 建议输入：

```text
ledger_id
source_type
occurred_at 精确到分钟或来源原始精度
amount_cents
merchant/title 规范化文本
external_order_id 如存在
```

状态判断：

| 状态 | 条件 | 默认行为 |
|---|---|---|
| new | 未发现同 hash 或高相似记录 | 可导入 |
| duplicate | 同账本已有相同 import_hash 或外部订单号 | 默认跳过 |
| suspicious | 同金额、同商户、时间相近但缺少订单号 | 用户确认 |
| invalid | 必需字段缺失或不合法 | 不可导入 |

Task48 必须补后端测试覆盖：

- 同文件重复导入。
- 同订单号重复。
- 疑似重复不自动提交。
- invalid 行导致事务回滚。

## 10. 审计与安全

必须写审计日志：

1. 创建导入批次。
2. 提交导入批次。
3. 导入规则创建、更新、归档、恢复。

审计日志不得包含完整 CSV 原文、绝对路径、token、secret。

导入原始文件：

- 默认不长期保存原文件。
- 若后续支持保存原文件，必须进入备份策略和权限保护范围。

## 11. 前端实现约束

页面入口：

```text
设置 -> 导入导出 -> CSV 导入工作区
```

建议步骤：

1. 选择来源。
2. 上传文件。
3. 字段映射。
4. 预览列表。
5. 行详情调整。
6. 二次确认提交。
7. 结果页。

移动端：

- 预览表格必须切换为卡片。
- 375px 不允许横向滚动。
- invalid/suspicious 行必须有明显状态标记。
- “确认导入”必须展示导入、跳过、疑似、错误数量。

Query key：

- 所有 imports/import-rules query key 必须包含 ledgerId。

## 12. Task47-Task49 完成标准

### Task47：CSV 导入预览

完成标准：

- `/api/imports/preview` 可生成 `ready` 批次。
- 预览不写正式 `transactions`。
- 行级错误包含行号、错误码和可行动原因。
- 微信、支付宝、通用 CSV 至少各有一个 fixture 测试。
- 前端可展示预览列表和 invalid 行。

### Task48：去重与事务落库

完成标准：

- `import_hash` 稳定生成。
- duplicate 默认跳过。
- suspicious 必须人工确认。
- commit 在单个数据库事务中完成。
- 任一未跳过的 invalid 行导致整批失败且不写半批数据。
- 成功提交后可追踪 generated transaction。

### Task49：导入规则

完成标准：

- 支持规则 CRUD、归档和恢复。
- 规则命中只填充建议值。
- 用户手工调整优先于规则。
- 禁用/归档规则不再命中。
- 引用已归档元数据时不得静默写入。

## 14. 回滚策略

1. 若 parser 有问题：关闭对应 source_type，保留 generic。
2. 若 commit 有问题：关闭 commit 入口，保留 preview 只读。
3. 若规则推荐误伤：关闭规则应用，仅保留人工选择。
4. 若 migration 有问题：新增修正 migration，不修改历史 migration。
5. 若旧接口兼容有风险：旧接口返回明确错误，引导前端使用 `/api/imports/*`。

## 15. Open Questions

以下问题不阻塞 Task47 启动，但必须在 Task48 冻结前定案：

1. `transactions` 是否新增 `import_hash`、`import_batch_id`、`import_row_id` 字段，或使用独立映射表。
2. 预览批次过期时间默认值，建议 7 天。
3. 单批最大行数，建议 MVP 限制 1000 行以内。
4. editor 是否允许导入，当前默认不允许。
5. 是否保存原始 CSV 文件，当前默认不保存。
