# v1.2 部署环境与数据库隔离方案

状态：v1.2 Release Hardening 冻结方案

## 1. 决策

LedgerTwo 需要显式区分开发、验收和正式数据环境。环境隔离不通过数据库内增加“环境字段”实现，而通过部署实例、物理目录、容器、端口、密钥和数据库文件共同隔离。

采用两个维度：

1. `APP_ENV` 表示运行安全模式。NAS 的验收和正式实例都使用 `production`，确保强密钥、Cookie 和目录校验不会被绕过。
2. `DEPLOYMENT_CHANNEL` 表示数据用途，只允许 `development`、`staging`、`production`。

不把 `staging` 直接作为 `APP_ENV`，因为现有认证和安全校验以 `APP_ENV=production` 为边界，混用会让验收环境意外降低安全要求。

## 2. 环境矩阵

| 环境 | 部署位置 | Channel | 端口 | 数据目录 | 数据用途 |
|---|---|---|---|---|---|
| 本地开发/验收 | WSL2/本机 Docker | `staging` 或 `development` | `38088` | 本机仓库卷 | 测试数据 |
| NAS 验收 | `/volume1/docker/ledger-two-staging` | `staging` | `38089` | 该目录独立 `data/` | 可丢弃验收快照 |
| NAS 正式 | `/volume1/docker/ledger-two` | `production` | `38088` | 该目录独立 `data/` | 原有正式账务数据 |

强制规则：

1. 两个 NAS 实例不得挂载同一个 `data/`、`backups/`、`uploads/` 或 `.env`。
2. staging 可以从 production 的一致性备份复制数据库，但复制后不再同步，staging 写入不得回灌 production。
3. production 升级只能在 staging 迁移和业务验收通过后执行。
4. 本机 QA 数据不得复制到 NAS production。
5. 数据库 schema 与应用镜像必须成对回滚；旧镜像不得连接高于其支持版本的 schema 18/19 数据库。

## 3. 用户可见标识

`GET /api/healthz` 返回 `deployment_channel`，系统诊断同步返回该字段。前端登录、初始化、桌面侧栏和移动端顶部显示：

1. `开发环境`：仅测试数据。
2. `验收环境`：NAS staging 或本机候选验证。
3. `正式数据`：NAS production。

标识同时展示应用版本，避免只看端口却无法判断镜像版本。

Task49X 增加 `IMPORT_XLSX_ENABLED`：未显式配置时 development/staging 为 `true`、production 为 `false`；非法值拒绝启动。`GET /api/healthz` 同步返回 `import_xlsx_enabled`，production 只有在 schema 19 staging 验收后才允许显式开启。

## 4. 镜像与 Compose 约束

Compose 必须显式提供：

```text
DEPLOYMENT_CHANNEL
IMPORT_XLSX_ENABLED
COMPOSE_PROJECT_NAME
CONTAINER_NAME
IMAGE_TAG
APP_PORT
APP_BASE_URL
```

推荐实例：

```text
staging:    ledger-two-staging / ledger-two:1.2.0-rc / 38089
production: ledger-two         / ledger-two:1.2.0-rc / 38088
```

候选镜像应按版本或提交固定标签，不以无法追溯的 `latest` 作为唯一发布与回滚依据。

## 5. 原有数据库如何保存

production 的 `/volume1/docker/ledger-two/data/ledger.db` 原地保留。部署 staging 时只读取该库并使用 SQLite `.backup` 生成一致性快照，再把快照复制到 staging 的独立数据库路径。

production 正式升级前必须再次执行：

```bash
sh deploy/nas/backup-predeploy.sh /volume1/docker/ledger-two v1.2.0-rc
```

备份包包含：

1. 一致性 SQLite 数据库副本。
2. 上传附件压缩包。
3. Compose 和脱离仓库保存的环境配置副本。
4. schema、用户/账本/账单/结算数量、`quick_check` 和 SHA-256。

Migration 13-18 只在目标实例自己的数据库中执行；Task49X migration 019 同样先在隔离 staging 执行。升级过程不会清空用户、账本、历史账单或结算数据，但会新增 v1.2 导入字段、索引、映射表和 XLSX 解析元数据。

## 6. v1.2 任务归属

该能力属于 `v1.2 Release Hardening / RC02`，位于 Task49 完成后、NAS production 升级前。它不是 Task47-Task49 导入业务的一部分，也不推迟到 v1.3，因为环境混用会直接威胁当前真实账务数据。

## 7. 放行顺序

```text
本地质量门禁
-> production 一致性备份
-> staging 使用备份副本迁移到 schema 18
-> staging health/登录/历史数据/导入/备份验收
-> production 再次备份
-> production 原库 migration 13-18
-> production 回归与升级后备份
```
