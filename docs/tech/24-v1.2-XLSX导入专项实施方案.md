## 1. 目标

在不复制导入业务逻辑、不改变现有预览与事务提交语义的前提下，为微信来源增加安全、可测试的 XLSX 输入适配器；支付宝继续使用已按真实官方样本冻结的 CSV reader。

目标依赖方向：

```text
HTTP multipart handler
-> file detector / limits
-> tabular reader
   -> CSV reader
   -> XLSX reader
-> source parser / normalizer
-> existing rule + dedupe + batch service
-> existing commit transaction
```

## 2. 当前实现与改造边界

实施前代码基线：

1. `handler.go` 使用 2 MiB `MaxBytesReader` 并一次性读取上传内容。
2. `Service.PreviewCSV` 直接调用 `ParseCSV`。
3. `parser.go` 同时承担 CSV 解码、来源字段映射和标准化。
4. `import_batches` 记录 source_type、filename 和 file_sha256，但未记录文件格式和解析元数据。
5. 前端只允许 `.csv`，页面文案也写死 CSV。

改造原则：

1. 不新增 XLSX 专用 preview/commit API。
2. 不在 handler 中实现工作簿业务判断。
3. 不复制微信、支付宝的标准化函数。
4. 不修改 migration 001-018。
5. 不改变既有 CSV fixture 预期结果和 commit 事务边界。

## 3. 依赖选型

### 3.1 候选比较

| 方案 | 优点 | 风险 | 结论 |
|---|---|---|---|
| `github.com/xuri/excelize/v2` | 纯 Go；支持 reader、流式行读取、格式化单元格与可配置解压限制；BSD-3-Clause；当前维护活跃 | 功能面较大，需显式收紧默认解压上限；会增加若干间接依赖 | 采用 |
| `github.com/tealeg/xlsx/v3` | API 直接；BSD-3-Clause；支持 RowLimit/ColLimit | 维护者明确不计划主动新增功能，流式 builder 已因维护成本移除；长期维护确定性较弱 | 不采用 |
| 标准库 `archive/zip` + `encoding/xml` 自研 | 无第三方依赖，读取范围可完全控制 | OOXML shared strings、样式、日期系统、关系文件、稀疏单元格和安全校验复杂，容易造成账务精度缺陷 | 不采用 |

### 3.2 冻结版本与工程影响

1. 实现时固定评审通过的 Excelize `v2.10.1`，通过 `go.mod/go.sum` 管理。
2. Excelize 要求 Go 1.24+；本项目当前 `go 1.26.4` 满足要求。
3. 该库为纯 Go，不引入 CGO，不影响现有 SQLite CGO 之外的编译模式。
4. 本项目不使用 CMake/vcpkg，因此两者影响均为 N/A；Docker 构建只增加 Go module 下载与编译缓存。
5. 许可证为 BSD-3-Clause，与当前私有部署兼容；发布材料需保留依赖许可证清单。

选型证据：

- Excelize 官方介绍与版本要求：`https://xuri.me/excelize/en/`
- Excelize workbook 解压限制：`https://xuri.me/excelize/en/workbook.html`
- tealeg/xlsx v3 维护说明与 API：`https://pkg.go.dev/github.com/tealeg/xlsx/v3`

## 4. 文件格式检测

新增内部枚举：

```text
FileFormatCSV  = csv
FileFormatXLSX = xlsx
```

检测顺序：

1. 规范化文件名并提取小写扩展名。
2. `.csv` 不得具有 ZIP magic。
3. `.xlsx` 必须具有 ZIP magic，并包含 `[Content_Types].xml`、`_rels/.rels`、`xl/workbook.xml`。
4. 检查 content type 和 ZIP entries，拒绝 VBA、OLE、外部嵌入对象和非 XLSX 主文档类型。
5. 扩展名和内容不一致时返回专用错误，不进入 parser。
6. 只有 `wechat + xlsx` 可进入 XLSX reader；`alipay + xlsx` 和 `generic + xlsx` 返回 `IMPORT_FILE_UNSUPPORTED`。

错误码新增：

```text
IMPORT_FILE_FORMAT_MISMATCH
IMPORT_FILE_UNSUPPORTED
IMPORT_FILE_ENCRYPTED
IMPORT_FILE_STRUCTURE_UNSUPPORTED
IMPORT_SOURCE_MISMATCH
IMPORT_WORKBOOK_AMBIGUOUS
IMPORT_BATCH_TOO_LARGE
IMPORT_ROW_FORMULA_UNSUPPORTED
IMPORT_ROW_ORDER_ID_UNSAFE
```

## 5. 解析抽象

### 5.1 TabularReader

建议接口：

```go
type FileFormat string

type TabularDocument struct {
    Format   FileFormat
    Metadata ParserMetadata
    Rows     []TabularRow
}

type TabularRow struct {
    Number int
    Cells  []TabularCell
}

type TabularCell struct {
    DisplayValue string
    RawValue     string
    Kind         CellKind
    Formula      string
}
```

说明：

1. CSV reader 返回 header=1、sheet 为空。
2. XLSX reader 返回物理行号和工作表元数据。
3. source parser 只接收表格行，不关心 ZIP、CSV encoding 或 Excelize。
4. `ParseCSV` 可暂时保留为兼容 wrapper，内部委托新的 `ParseTabularRows`。

### 5.2 ParserMetadata

```json
{
  "parser_version": "xlsx-v1",
  "sheet_name": "Sheet1",
  "header_row_number": 18,
  "scanned_sheets": 1,
  "parsed_rows": 276,
  "max_columns": 11
}
```

元数据不得包含单元格值、订单号或绝对路径。

## 6. XLSX Reader 契约

### 6.1 资源限制

打开工作簿时显式配置，不依赖库默认值：

| 限制 | 冻结值 |
|---|---:|
| HTTP 压缩文件大小 | 2 MiB |
| 解压后总大小 | 32 MiB |
| 单 XML 内存阈值 | 8 MiB |
| 工作表扫描数 | 10 |
| 表头扫描行 | 每表前 50 行 |
| 最大列数 | 64 |
| 数据行数 | 2000 |

超过任一限制返回 `IMPORT_BATCH_TOO_LARGE`。临时文件目录使用应用受控临时目录，并确保解析结束后关闭 workbook。

### 6.2 工作表选择

1. 获取 workbook 顺序中的可见工作表。
2. 对每张表前 50 行构建规范化 header index。
3. 使用 source_type 对应的必需字段集合判断候选。
4. 恰好一个候选时选中；零个候选返回缺失字段；多个候选返回 `IMPORT_WORKBOOK_AMBIGUOUS`。
5. 不自动选择 active sheet 或文件中的第一张表，避免错误导入说明页。

### 6.3 来源表头

微信至少识别：

```text
交易时间 交易类型 交易对方 商品 收/支 金额(元)
支付方式 当前状态 交易单号 商户单号 备注
```

支付宝 CSV 至少识别：

```text
交易创建时间/付款时间 类型 交易对方 商品名称 收/支
金额(元) 交易号 备注
```

支付宝别名字典已由当前真实官方 CSV 样本冻结，不允许仅凭人工记忆扩展；该字典供 CSV reader 使用，不代表开放支付宝 XLSX。

### 6.4 单元格语义

1. header 使用 display value 规范化，兼容全角括号和首尾空白。
2. 金额使用不经过二进制浮点计算的十进制字符串进入现有 `parseYuanToCents`。
3. 日期文本支持 `YYYY-MM-DD HH:mm:ss` 和 `YYYY/MM/DD HH:mm:ss`；日期型数字使用 workbook 日期系统转换后再格式化。
4. 订单号优先 display value；若为数值、科学计数法或精度无法证明安全，行标记 invalid。
5. 必需字段或数据区出现公式时不读取缓存结果。
6. 数据区出现合并单元格、隐藏行或隐藏列时拒绝工作簿，不做静默跳过。

## 7. Service 与 API 改造

### 7.1 Service

建议重命名：

```text
PreviewCSV -> PreviewFile
ParseCSV   -> ParseImportFile / ParseTabularRows
```

`PreviewFile` 顺序：

1. owner 和 source_type 校验。
2. 文件非空与 2 MiB 限制。
3. 格式检测和 source/format 支持矩阵校验。
4. reader 读取 TabularDocument。
5. 来源 parser 标准化。
6. 2000 行限制。
7. 规则、去重、批次统计和持久化。

CSV 与 XLSX 必须从第 5 步开始共享代码。

### 7.2 API

继续使用：

```text
POST /api/imports/preview
```

请求字段不新增强制参数：

- `file`：CSV 或 XLSX binary。
- `source_type`：wechat、alipay、generic。

后端推断 `file_format`，不信任前端声明。响应增加：

```json
{
  "file_format": "xlsx",
  "parser_metadata": {
    "sheet_name": "Sheet1",
    "header_row_number": 18,
    "parsed_rows": 276
  }
}
```

## 8. 数据迁移

新增 migration，不修改 001-018：

```text
019_add_import_batch_file_metadata.sql
```

建议字段：

```sql
ALTER TABLE import_batches
ADD COLUMN file_format TEXT NOT NULL DEFAULT 'csv';

ALTER TABLE import_batches
ADD COLUMN parser_metadata_json TEXT NOT NULL DEFAULT '{}';
```

兼容性：

1. 历史批次默认 `csv`，符合当前事实。
2. 旧应用忽略新增字段；回退应用不需要降级 schema。
3. parser metadata 只在 service 层校验 JSON 结构。
4. 不给 `transactions`、`transaction_import_refs` 或 `import_items` 增加格式字段；正式去重与文件容器无关。

## 9. 去重一致性

`import_hash` 不得包含 file_format、sheet_name 或 header_row_number。相同来源流水无论由 CSV 还是 XLSX 进入，都应生成同一 hash。

必须新增等价测试：

```text
wechat-equivalent.csv  == wechat-equivalent.xlsx
```

比较字段：occurred_at、amount_cents、direction、merchant、title、external_order_id、target_transaction_type 和 import_hash。

## 10. 前端改造

1. 微信 `accept` 在门禁开启时为 `.csv,.xlsx`；支付宝和通用模板固定为 `.csv`。
2. 校验 source_type + extension 支持矩阵，支付宝 XLSX 在前端提前拒绝，后端仍是最终边界。
3. 页面所有“CSV 文件”通用动作文案改为“账单文件”。
4. 批次 DTO 增加 `file_format` 和只读 `parser_metadata`。
5. XLSX 预览摘要展示工作表、表头行和识别行数。
6. 不新增 XLSX 专用提交按钮或状态机。

## 11. Fixture 与测试计划

### 11.1 Fixture

新增匿名文件：


二进制 fixture 必须由匿名数据生成，评审时记录 SHA-256；真实用户账单不得进入仓库。

### 11.2 后端测试

1. 格式 magic、扩展名和 OOXML 部件检测。
2. 解压上限、行列上限和工作表上限。
3. 表头第 18 行、零候选和多候选。
4. 日期文本、Excel 日期、金额、订单号和公式拒绝。
5. 微信 XLSX/CSV 标准化与 hash 等价。
6. owner/editor/viewer 权限。
7. preview 不写 transactions。
8. XLSX commit 仍满足 I02-I04。

### 11.3 前端测试

1. 微信接受 CSV、XLSX，支付宝和 generic 只接受 CSV。
2. 支付宝或 generic 选择 XLSX 时前端拒绝。
3. 批次格式与工作表摘要展示。
4. 后端结构错误映射为可行动中文提示。
5. 375px/390px 上传和预览无横向溢出。

## 13. 风险与回滚

| 风险 | 控制 | 回滚 |
|---|---|---|
| ZIP bomb 或超大 shared strings | 双重大小限制、行列/工作表限制 | 关闭 xlsx 开关 |
| 日期或金额解释错误 | CSV/XLSX 等价测试、真实样本抽查 | 禁止 commit，保留 preview |
| 订单号精度丢失 | 数字/科学计数法拒绝策略 | 要求重新导出 |
| 第三方库回归 | 固定版本、fixture 回归、依赖许可证记录 | 回退代码和开关，不回退 schema |
| 支付宝未来改变导出格式 | 不推测支持；重新评审 PRD、fixture 和真实样本 | 继续只开放支付宝 CSV |
