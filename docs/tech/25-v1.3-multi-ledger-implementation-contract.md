## 1. 目标与约束

本契约把 Task50 冻结为可实现的技术切片，但不创建 migration、不修改 OpenAPI、不进入业务代码。目标是让多账本成为可信的数据与权限边界，而不是只增加一个选择器。

硬约束：

1. Go + SQLite + REST JSON + React/TypeScript 技术栈保持不变，不引入新依赖。
2. 每账本最多两名成员，且任何稳定状态恰好一名 Owner。
3. 所有账本内请求显式指定账本；Task50 删除业务 service 的首账本 fallback。
4. archived 账本允许读取和按权限导出，但所有业务写入统一拒绝。
5. 整库备份、恢复准备和系统诊断属于实例管理员，不属于账本 Owner。
6. migration 只新增 `020/021`，不修改 001-019；金额、分摊、结算、导入 parser/hash/commit 和统计口径不变。

## 2. 架构决策摘要

| 编号 | 决策 | 结论 |
|---|---|---|
| D1 | 生命周期存储 | 在 `ledgers` 增加状态与版本字段，不建平行 lifecycle 表 |
| D2 | 写入阻断 | `LedgerContext + LifecyclePolicy` 统一 Guard，禁止各 service 自行判断 |
| D3 | 当前账本 | 仅存前端最近使用偏好；服务端不新增 current ledger 偏好 |
| D4 | 并发控制 | 账本管理 mutation 使用 `version + If-Match` 乐观并发 |
| D5 | Owner 不变量 | partial unique index 防双 Owner；service 事务防零 Owner并执行原子移交 |
| D6 | 成员上限 | service 事务校验 + SQLite INSERT trigger 双层限制为 2 人 |
| D7 | 实例管理员 | 新建 `instance_admins` 与 `instance_audit_logs`，不复用账本角色或 `users.role` |
| D8 | API 非成员语义 | 冻结为 403；可访问账本内不存在对象仍返回 404 |
| D9 | 数据库回滚 | migration 前向兼容、生产禁止 goose down；必要时恢复升级前完整备份 |
| D10 | ready 导入批次 | 未过期 ready 阻止归档；用户先显式 discard，复用 `expired` 终态 |

## 3. 备选方案与取舍

### 3.1 生命周期字段与独立表

选择：在 `ledgers` 增加 `status/archived_at/archived_by_user_id/version`。

未选择独立 `ledger_lifecycle` 表，原因是当前只有 active/archived 两态，独立表会增加每次 LedgerContext 解析的 join 和一致性风险。完整变更历史继续进入 `audit_logs`，不把当前状态和事件历史混在同一表。

### 3.2 实例管理员存储

选择：新建 `instance_admins` 能力表。

未选择：

1. 复用 `ledger_members.role=owner`：会继续允许任一账本 Owner 下载整库，构成跨账本越权。
2. 复用 `users.role`：该字段当前统一为 `user` 且缺少能力历史、唯一性和授权审计，直接改语义会形成隐式全局 RBAC。
3. 使用 `app_settings.instance_admin_user_id`：无法建立外键与授权历史，也不利于未来安全移交。

`instance_admins` 初期只开放一个有效管理员的产品路径，但数据模型允许未来受审计地扩展；Task50 不提供普通用户自行授予实例管理员的 Web API。

### 3.3 Owner 约束

选择：partial unique index 保证同一账本最多一个 Owner，service 事务保证至少一个 Owner。

未选择强制“禁止 Owner 降级/删除”的数据库 trigger，因为 SQLite 的非延迟唯一约束会阻碍同一事务内的 Owner 角色交换。Owner 移交只能调用独立 service，通用角色和移除接口不得触碰 Owner。测试与 repository API 必须使零 Owner 路径不可达。

### 3.4 最近账本偏好

选择：继续由前端持久化，但只保存偏好，不保存权限事实。

未选择服务端用户偏好表，原因是当前单设备/NAS 场景不需要跨设备同步，且服务端偏好不能替代每次启动时的成员与状态校验。后续若出现多终端需求再独立评审。

## 4. 目标模块边界

### 4.1 后端依赖方向

```text
handler / middleware
        |
        v
ledger Service + LedgerContext + RolePolicy + LifecyclePolicy
        |
        v
ledger Repository / module Repository
        |
        v
SQLite
```

规则：

1. handler 只解析 HTTP DTO、Path/Header/ETag 并映射错误，不编排成员事务。
2. ledger service 拥有创建、重命名、归档、恢复、添加/调整/移除/离开/移交事务。
3. `LedgerContext` 是所有账本内 service 的输入事实；业务 service 不再自行查询第一个账本或复制角色 SQL。
4. `LifecyclePolicy` 只判断 active/archived 与操作类型；`RolePolicy` 只判断角色与能力，二者组合而不互相包含。
5. repository 不接受只有对象 ID 的跨账本方法；读取或修改必须同时带 `ledgerID`，子表通过父对象 join 校验。
6. safety 模块拆成“账本可见数据导出”和“实例级运维”两条权限路径。

### 4.2 目标上下文

```go
type LedgerStatus string

const (
    LedgerStatusActive   LedgerStatus = "active"
    LedgerStatusArchived LedgerStatus = "archived"
)

type LedgerContext struct {
    UserID     string
    LedgerID   string
    Role       Role
    Status     LedgerStatus
    Version    int64
    IsExplicit bool
}
```

`IsExplicit` 在 Task50 的账本内路由必须恒为 `true`。该字段只用于兼容测试和迁移审计，不能继续授权 fallback。

### 4.3 路由分组

全局路由无需账本上下文：登录、退出、`/me`、初始化、health、账本列表、创建账本、实例管理员运维。

账本内路由必须通过以下链路：

```text
Authenticate -> ResolveExplicitLedgerContext -> RequireLedgerContext
             -> RequireReadableLedger / RequireWritableLedger
             -> Role/对象级策略 -> handler
```

账本 ID 可来自 `X-Ledger-Id` 或规范中明确包含 `{ledgerId}` 的路径。若两者同时存在但不同，返回 `LEDGER_CONTEXT_MISMATCH`，不得任选其一。

## 5. 数据模型冻结

### 5.1 Migration 020：账本生命周期与成员约束

冻结文件名：`backend/migrations/020_add_ledger_lifecycle.sql`

新增字段：

```sql
ALTER TABLE ledgers ADD COLUMN status TEXT NOT NULL DEFAULT 'active'
    CHECK (status IN ('active', 'archived'));
ALTER TABLE ledgers ADD COLUMN archived_at TEXT;
ALTER TABLE ledgers ADD COLUMN archived_by_user_id TEXT REFERENCES users(id);
ALTER TABLE ledgers ADD COLUMN version INTEGER NOT NULL DEFAULT 1;
ALTER TABLE audit_logs ADD COLUMN actor_role TEXT;
```

状态约束由 service 同时校验：

- active：`archived_at IS NULL AND archived_by_user_id IS NULL`
- archived：两个归档字段非空，`archived_by_user_id` 在归档时必须是该账本 Owner
- `version >= 1`，每次账本生命周期或成员管理 mutation 加 1

新增索引/trigger：

```text
idx_ledgers_status_created             (status, created_at, id)
idx_ledger_members_user_ledger         (user_id, ledger_id)
idx_ledger_members_one_owner           UNIQUE (ledger_id) WHERE role='owner'
idx_audit_logs_ledger_created          (ledger_id, created_at, id)
trg_ledger_members_max_two_before_insert
```

`trg_ledger_members_max_two_before_insert` 仅阻止第三条成员关系；更新现有角色不计为新增。应用层把 SQLite constraint 错误稳定映射为 `LEDGER_MEMBER_LIMIT_REACHED`，不能把底层 SQL 文案暴露给 UI。

### 5.2 Migration 021：实例管理员

冻结文件名：`backend/migrations/021_add_instance_administration.sql`

```sql
CREATE TABLE instance_admins (
    user_id TEXT PRIMARY KEY,
    granted_at TEXT NOT NULL,
    granted_by_user_id TEXT,
    FOREIGN KEY (user_id) REFERENCES users(id),
    FOREIGN KEY (granted_by_user_id) REFERENCES users(id)
);

CREATE TABLE instance_audit_logs (
    id TEXT PRIMARY KEY,
    actor_user_id TEXT NOT NULL,
    action TEXT NOT NULL,
    entity_type TEXT NOT NULL,
    entity_id TEXT NOT NULL,
    before_json TEXT,
    after_json TEXT,
    created_at TEXT NOT NULL,
    FOREIGN KEY (actor_user_id) REFERENCES users(id)
);

CREATE INDEX idx_instance_audit_created
    ON instance_audit_logs(created_at, id);
```

升级已有实例时，021 选择“最早创建账本的当前 Owner”作为首个实例管理员；排序固定为 `ledgers.created_at, ledgers.id, ledger_members.created_at, user_id`。P.6 升级预检必须先证明每账本恰好一名 Owner，否则不执行 020/021。

空数据库在执行 migration 时没有用户，021 不回填；初始化事务必须把首位初始化用户写入 `instance_admins`。第二位用户及后续新账本 Owner不会自动获得该能力。

### 5.3 不新增的字段

1. 不新增 `current_ledger_id` 服务端偏好。
2. 不新增单账本物理备份表或账本删除字段。
3. 不重命名 `partner_readable`，Task50 继续使用冻结双人语义。
4. 不为 splits/tags/attachments 重复增加 ledger_id；访问时必须 join 父交易并校验父交易账本。

## 6. 生命周期与并发事务

### 6.1 通用并发规则

账本管理资源返回 `version` 与 ETag，格式在 P.4 冻结为等价于：

```text
ETag: "ledger:<ledger-id>:v<version>"
If-Match: "ledger:<ledger-id>:v<version>"
```

rename/archive/restore/add/update/remove/leave/transfer 都必须携带 `If-Match`。service 首个写语句使用 `WHERE id=? AND version=?` 竞争版本并获得 SQLite 写锁；`RowsAffected=0` 返回 `LEDGER_VERSION_CONFLICT`，客户端刷新后重试，不能自动覆盖。

### 6.2 归档

归档事务顺序：

1. 验证 active、Owner、If-Match 和当前成员不变量。
2. 将 `expires_at <= now` 的 ready 批次按既有状态机收敛为 expired。
3. 若仍有未过期 ready 批次，返回 `LEDGER_READY_IMPORT_EXISTS`，不改变任何数据。
4. 读取当前未结清净额，只用于确认快照与审计，不创建 settlement。
5. 条件更新 ledger 为 archived、写归档人/时间、version + 1。
6. 写 `ledger_archive` 审计，after_json 包含 status、version、未结清金额和 ready batch count。
7. 提交后失效该账本全部写入口及前端缓存。

用户显式放弃 ready 批次使用独立 Owner-only import mutation，将批次置为 `expired` 并记录 `discard_reason=user_requested`；不删除预览行，不改 hash，不写 transaction。

### 6.3 恢复与重命名

恢复仅 Owner 可执行；校验 archived + If-Match 后清空归档字段、status=active、version + 1 并写审计。归档账本不可直接重命名。

重命名仅 active Owner 可执行；名称 trim 后 1-60 个 Unicode 字符，同一用户可拥有同名账本，不以名称作为身份或幂等键。

### 6.4 成员事务

添加：

1. active Owner + If-Match。
2. username 精确匹配、用户 active、不是现有成员。
3. 成员数小于 2，否则返回 `LEDGER_MEMBER_LIMIT_REACHED`。
4. 只接受 editor/viewer；插入成员、version + 1、写历史可见性确认审计。

角色调整：只允许 editor 与 viewer 互转；通用接口不得接受 owner，也不得操作当前 Owner。

Owner 移交：

1. active 当前 Owner 发起，目标必须是同账本 editor/viewer。
2. 在同一事务内使用受控 repository 方法完成目标 -> owner、原 Owner -> editor、version + 1。
3. 事务提交前查询 `COUNT(role='owner')=1`；失败则整体回滚并记录错误，不产生审计半记录。
4. 移交成功写一个 `ledger_owner_transfer` 审计事件，不拆成两个可独立失败的角色事件。

移除/离开：Owner 可移除非 Owner；Editor/Viewer 可主动离开；Owner 必须先移交。删除关系不删除或改写其历史交易、split、settlement、附件和审计。

## 7. 权限冻结

### 7.1 RolePolicy

新增操作：

```text
archive_ledger
restore_ledger
leave_ledger
transfer_ledger_owner
manage_imports
discard_import_batch
```

修正：

1. Editor 获得 `export_data`，与已验收后端/UI 和 Task50 PRD 一致。
2. `manual_backup/restore_backup/system_diagnostics` 从 RolePolicy 删除，不再由 Owner 授权。
3. import preview/update/commit/rules/discard 统一 Owner-only。
4. archived 状态先经过 LifecyclePolicy；即使角色允许，写操作也返回 `LEDGER_ARCHIVED`。

### 7.2 InstancePolicy

`instance_admins` 是唯一实例级能力事实源。以下路由不读取 `X-Ledger-Id`，也不向 ledger audit 写伪造账本 ID：

```text
manual_database_backup
list/download_database_backups
prepare_database_restore
system_diagnostics
future instance-admin transfer
```

实例操作写入 `instance_audit_logs`。登录会话不缓存永久管理员权限；每次高风险请求从数据库读取，避免撤权后旧 cookie 继续生效。

## 8. 显式账本与 fallback 清理

Task50.2 必须删除以下生产路径的首账本查询：

1. dashboard service。
2. transaction service 与 shared expense repository。
3. settlement service。
4. reports service。
5. metadata repository/service。
6. safety 的账本数据导出路径。
7. auth `/me` 中注入的单一 `ledger_id`。

`/me` 改为返回用户资料、`instance_admin` 布尔值和可选能力摘要；当前账本由 `/ledgers` 列表与前端偏好决定。无 LedgerContext 的账本内 service 直接返回 `LEDGER_REQUIRED`，不接受 userID 再查首账本。

## 9. 归档 Guard 覆盖矩阵

| 模块 | archived 可读 | archived 可写 | 实现要求 |
|---|:---:|:---:|---|
| 流水/共同支出 | 是，按可见性 | 否 | 创建、编辑、软删除、批量标签全部经过 Guard |
| 结算 | 历史与余额是 | 否 | 不登记 settlement，不改历史账单 |
| 元数据 | 是 | 否 | 分类/标签/账户新增、编辑、归档、恢复、排序均拒绝 |
| 模板/周期 | 是 | 否 | 禁止模板 mutation、规则 mutation、确认/跳过生成 |
| 导入 | 已有批次/规则是 | 否 | preview、行调整、commit、规则 mutation、discard 均拒绝 |
| 附件 | 可见交易附件可下载 | 否 | 禁止上传/替换/删除；下载校验父交易账本与可见性 |
| 报表 | 是 | 不适用 | settlement 仍不进入消费统计 |
| CSV/JSON 导出 | Owner/Editor 是 | 生成导出审计是受控例外 | 只包含当前账本和当前角色可见数据 |
| 成员 | 名单是 | 否 | 归档期间不添加、调角、移除、离开或移交 |
| 生命周期 | 是 | Owner 只可恢复 | restore 是 archived 状态唯一账本级 mutation |

导出属于生成只读快照的受控动作，不改变账本业务数据；仍需写审计，但不能因此开放其他写操作。

## 10. 导出、备份与审计

### 10.1 账本导出

CSV/JSON 继续允许 Owner/Editor，且必须按对象可见性过滤。Task50 JSON 数据包移除全局 `users` 与 `app_settings`；只允许包含：

1. 当前账本及当前成员的必要展示资料。
2. 当前角色可见交易及其 splits/tags/附件引用。
3. 当前账本元数据、模板、周期、结算、导入引用和可见审计。

导出不是可直接恢复的物理备份，UI 和文件 manifest 必须明确格式用途。

### 10.2 整库运维

SQLite `VACUUM INTO`、备份列表/下载、恢复准备和诊断改由 InstancePolicy 管理。迁移到新权限模型后，账本 Owner 对 `/api/admin/*` 不再天然有权；P.4 fixture 必须包含“非管理员 Owner 拒绝”和“非 Owner 实例管理员允许”的交叉用例。

### 10.3 审计事件

ledger audit 新增：

```text
ledger_create
ledger_rename
ledger_archive
ledger_restore
ledger_member_add
ledger_member_role_update
ledger_member_remove
ledger_member_leave
ledger_owner_transfer
import_batch_discard
```

新事件必须写 actor_role；before/after JSON 不存密码、cookie、JWT、原始账单内容或附件二进制。实例级 backup/restore/diagnostics 只写 instance audit。

## 11. 前端状态与缓存契约

1. `/ledgers` 默认只返回 active，可显式请求 archived/all；每项包含 role/status/version。
2. Zustand 只持久化 `activeLedgerId` 与按账本最近使用时间，不把 role/status 作为长期可信值。
3. 启动时以最新 active 列表校验偏好；无效时选最近使用的可访问 active 账本，不按 `created_at ASC` 猜测。
4. 没有 active 账本时进入 no-active-ledger shell，只请求全局路由；不挂载 Dashboard/Transaction/Report/Import query。
5. 切换前调用 TanStack Query `cancelQueries` 取消旧账本请求，再更新 active ID；所有账本查询键必须包含 ledger ID。
6. API 请求传递 AbortSignal；页面渲染同时校验响应对应的 ledger ID，旧请求晚返回不得覆盖新页面。
7. 归档、恢复、移除和离开成功后先更新 ledger list，再执行确定性回退，并显示一次原因提示。
8. Task50 UI 复用 UI-FL 的 AppShell、Dialog、BottomSheet、Button、StatePanel 和断点，不新建平行组件。

## 12. 错误码冻结

P.4 OpenAPI 必须使用以下稳定码：

| HTTP | code | 语义 |
|---:|---|---|
| 400 | `LEDGER_REQUIRED` | 账本内请求未显式指定账本 |
| 400 | `LEDGER_CONTEXT_MISMATCH` | Path 与 Header 账本不同 |
| 400 | `LEDGER_INVALID_STATE` | 字段组合或状态转换无效 |
| 403 | `LEDGER_ACCESS_DENIED` | 当前用户不是成员或角色不允许 |
| 403 | `INSTANCE_ADMIN_REQUIRED` | 缺少实例管理员能力 |
| 404 | `LEDGER_OBJECT_NOT_FOUND` | 可访问账本内对象不存在 |
| 409 | `LEDGER_ARCHIVED` | 对 archived 账本执行禁止的写操作 |
| 409 | `LEDGER_VERSION_CONFLICT` | If-Match 与当前 version 不同 |
| 409 | `LEDGER_MEMBER_LIMIT_REACHED` | 已有两名成员 |
| 409 | `LEDGER_OWNER_TRANSFER_REQUIRED` | Owner 尝试离开/被移除/降级 |
| 409 | `LEDGER_OWNER_INVARIANT_VIOLATION` | 操作会产生零或多 Owner |
| 409 | `LEDGER_READY_IMPORT_EXISTS` | 未过期 ready 批次阻止归档 |

错误响应继续使用统一 envelope；不得暴露 SQL、文件系统路径或对象是否存在于其他账本。

## 13. Migration 升级评审

### 13.1 升级前阻断检查

P.6 发布脚本必须在数据库副本和目标环境执行并保存结果：

```sql
PRAGMA quick_check;
PRAGMA foreign_key_check;

SELECT ledger_id, COUNT(*) member_count
FROM ledger_members GROUP BY ledger_id HAVING COUNT(*) > 2;

SELECT l.id,
       SUM(CASE WHEN lm.role='owner' THEN 1 ELSE 0 END) owner_count
FROM ledgers l LEFT JOIN ledger_members lm ON lm.ledger_id=l.id
GROUP BY l.id HAVING owner_count <> 1;

SELECT ledger_id, user_id, role
FROM ledger_members
WHERE role NOT IN ('owner','editor','viewer');
```

还必须逐表检查孤立 ledger_id，并记录每账本 transactions/settlements/import_batches/audit_logs 数量、交易金额总和、结算金额总和、附件引用数。任一异常均阻止 migration，禁止自动删除或猜测修复。

### 13.2 空库、v1.2 副本与异常 fixture

Migration 测试至少覆盖：

1. 空数据库 001 -> 021 完整升级。
2. schema 19 正常双人账本副本升级，全部 ledger 为 active、version=1。
3. schema 19 一人账本升级。
4. 超过两成员、零 Owner、多 Owner、非法角色、孤立 ledger_id 的预检均失败。
5. partial unique owner index 与第三成员 trigger 生效。

7. 新旧镜像兼容窗口、health schema 21、数据计数和金额守恒。

### 13.3 Down 与回滚

020/021 的 goose Down 不删除新增列、实例管理员或实例审计数据，只保留显式 no-op/兼容说明。生产环境禁止依赖 `goose down` 回滚。

回滚分两种：

1. migration 后、Task50 功能尚未启用且没有 archived 账本：可回退应用镜像，旧应用忽略新增字段；成员上限和唯一 Owner 约束继续生效。
2. 已发生归档、Owner 移交或实例权限变更：禁止直接回退到 v1.2 镜像。应优先修复前滚；必须数据库回滚时，停机恢复 migration 前完整备份，并单独处置升级后产生的数据。

原因：v1.2 不认识 archived Guard，直接回退会重新允许对归档账本写入；删除 021 也会恢复“账本 Owner 可取整库”的安全缺陷。

### 13.4 发布门禁

1. development、staging、production 使用不同数据库、上传、备份、日志、端口和密钥。
2. migration 前由实例管理员生成并校验整库备份；记录 SHA-256、quick_check、schema 和核心计数。
3. staging 先升级到 schema 21，跑完 P.4 隔离矩阵和 P.5 全视口验收后才能进入 production。
4. health 必须回读应用版本、deployment channel、db ok 和 `schema_version=21`。
5. Task50 migration 不与 v1.2 schema 19 NAS 发布操作混在同一维护窗口。
