## 1. Goal

在不复制 CSV/XLSX parser、不改变去重与整批事务提交、不引入云端分类服务的前提下，将现有 Task49 “规则建议”升级为可解释的分级自动化，并让初始化、新账本和既有账本共享同一套幂等默认元数据机制。

## 2. Current state

当前可复用基础：

1. `import_rules` 已支持 CRUD、归档、优先级和分类/账户/多标签结果。
2. preview 行已区分 `suggested_*` 和 `selected_*`。
3. commit 只消费 `selected_*`，并在单个事务中写入 transaction/tags/import refs。
4. metadata 已支持分类、标签、账户按账本管理、排序、归档和恢复。
5. `categories.parent_id` 和 `is_system` 已存在，但当前服务未实现二级分类与系统分类保护。
6. 首次初始化和 Task50 新账本使用两条不同写入路径；后者当前不创建默认元数据。

主要缺口：

1. `applyImportRules` 只取第一条命中并只产生建议。
2. 匹配大小写、Unicode、商户噪声和来源范围没有统一规范。
3. 没有规则来源、置信度、自动选择开关和冲突状态。
4. 用户修正不能显式生成学习规则。
5. 默认分类/标签定义硬编码在 `InitRepo`，无法复用于新账本或既有账本。
6. 预览行没有持久化分类解释和规则候选。

## 3. Architecture

目标数据流：

```text
tabular reader
-> source parser / normalizer
-> duplicate pre-check eligibility
-> classifier context
   -> user rule provider
   -> learned rule provider
   -> built-in knowledge provider
   -> fallback provider
-> deterministic resolver
-> preview selected/suggested/explanation persistence
-> row/bulk adjustment
-> optional explicit learning
-> existing commit transaction
```

建议模块边界：

```text
backend/internal/importer/classifier/
  model.go          候选、置信度、来源和最终决策
  normalize.go      商户/标题规范化
  rule_provider.go  用户与学习规则候选
  builtin.go        本地内置知识候选
  resolver.go       纯函数优先级、冲突、标签并集

backend/internal/metadata/defaults/
  profile.go        basic_cn_v1 常量、图标、颜色、顺序
  apply.go          caller transaction-aware 幂等写入
```

约束：

1. classifier 不访问 HTTP，不写数据库，不知道 React DTO。
2. repository 负责加载规则/元数据和持久化预览解释。
3. service 负责权限、状态机、事务和稳定错误映射。
4. built-in provider 只能读取仓库内版本化常量，不访问网络。
5. parser 不承担业务分类，CSV/XLSX 等价性保持不变。

## 4. Classification model

```go
type ClassificationSource string

const (
    SourceManual   ClassificationSource = "manual"
    SourceBulk     ClassificationSource = "bulk"
    SourceUserRule ClassificationSource = "user_rule"
    SourceLearned  ClassificationSource = "learned_rule"
    SourceBuiltin  ClassificationSource = "builtin"
    SourceFallback ClassificationSource = "fallback"
)

type Confidence string

const (
    ConfidenceHigh   Confidence = "high"
    ConfidenceMedium Confidence = "medium"
    ConfidenceLow    Confidence = "low"
    ConfidenceNone   Confidence = "none"
)

type ClassificationStatus string

const (
    StatusAutoSelected ClassificationStatus = "auto_selected"
    StatusSuggested    ClassificationStatus = "suggested"
    StatusFallback     ClassificationStatus = "fallback"
    StatusManual       ClassificationStatus = "manual"
    StatusConflict     ClassificationStatus = "conflict"
    StatusUnresolved   ClassificationStatus = "unresolved"
)
```

候选至少包含：

```go
type Candidate struct {
    CategoryID string
    TagIDs     []string
    Source     ClassificationSource
    Confidence Confidence
    Priority   int
    Specificity int
    RuleID     string
    ReasonCode string
    ReasonText string
}
```

resolver 是纯函数，输入同序候选必须得到同结果。禁止依赖 map 遍历顺序、当前时间或随机值。

## 5. Normalization

商户匹配使用独立的原始值和规范化值：

1. 保留 parser 输出的原始 `merchant/title/description` 用于展示和审计。
2. 对匹配副本执行 trim、Unicode NFKC、Unicode 小写、连续空白折叠和全角标点统一。
3. 不删除中文主体，不用模糊分词猜测人名或商户。
4. 首期只移除已验证的来源噪声，例如首尾订单号片段；每条规则必须有测试 Fixture。
5. `golang.org/x/text` 已是直接依赖，使用 `unicode/norm` 不新增第三方依赖。

方案比较：

| 方案 | 优点 | 风险 | 结论 |
|---|---|---|---|
| 标准化 + 确定性规则 | 本地、可解释、可测试、可回滚 | 覆盖率随规则积累增长 | 采用 |
| 朴素贝叶斯/本地 ML | 可从历史学习 | 训练样本少、难解释、跨账本隐私和模型迁移复杂 | 首期不采用 |
| 云端 LLM/分类 API | 泛化能力可能更强 | 上传敏感账单、成本、网络、不可重复 | 明确不采用 |

## 6. Rule semantics

### 6.1 Existing rule compatibility

1. 历史规则升级后保持 `apply_mode=suggest`，不因 migration 自动改变行为。
2. 新建用户规则若包含分类或标签，UI 默认建议 `apply_mode=auto`，用户可改为“仅建议”。
3. 学习规则固定为精确商户匹配和 `apply_mode=auto`。
4. 归档规则不参与候选。
5. 引用归档/跨账本元数据的规则整体不可应用，不允许只应用剩余部分。

### 6.2 Match types

首期保留：

```text
merchant_contains
description_contains
source_account
amount_range
```

新增：

```text
merchant_equals
```

不新增正则、任意 AND/OR 或脚本表达式。

### 6.3 Resolution order

用户/学习规则排序键：

```text
priority ASC
origin: manual before learned
specificity: equals before contains before amount_range
created_at DESC
id ASC
```

分类：

1. 最高排序组只有一个分类时自动选择。
2. 同一最高排序组出现多个不同分类时返回 `conflict`，不自动选择。
3. 高置信分类存在时，built-in 分类只保留为诊断候选，不展示为第二选择。

标签：

1. 所有高置信自动规则标签按排序取并集。
2. 最多 8 个，超限返回冲突，不静默截断。
3. built-in 标签只进入建议集合。
4. 用户手工保存后以 `selected_tag_ids` 为最终全集，不在 commit 阶段补回。

账户维持 Task49 既有规则语义，本专项不扩展自动账户学习。

## 7. Built-in knowledge

内置知识使用版本化静态表：

```go
type BuiltinRule struct {
    Key              string
    SourceTypes      []string
    MerchantContains []string
    TitleContains    []string
    CategorySystemKey string
    TagSystemKeys    []string
    Confidence       Confidence
    ReasonCode       string
}
```

规则：

1. 只能引用 `system_key`，运行时解析为当前账本 ID。
2. 任一系统分类/标签不存在或已归档时跳过该候选，不自动创建。
3. built-in 默认 medium/low，只建议。
4. 词典更新必须伴随 Fixture、版本号和回归测试。
5. 不从真实账单自动生成仓库词典。

## 8. Default metadata

### 8.1 Transaction boundary

以下操作必须原子完成：

1. 首次初始化：账本、用户、成员、账户、`basic_cn_v1` 和 initialized lock。
2. 新建账本：账本、Owner、默认 profile、ledger_create audit。
3. 既有账本应用 profile：冲突确认、分类/标签写入和 metadata_profile_apply audit。

任何一项失败均不得留下半套分类/标签。

### 8.2 Idempotency

1. profile 项使用稳定 `system_key`，数据库 ID 仍为 UUID。
2. 同账本 `system_key` 唯一。
3. 既有同名同类型对象可由用户在预览中选择“复用”，复用不写入 system_key，避免夺取用户对象所有权。
4. 再次应用相同 profile 只补缺失项，不覆盖名称、颜色、图标和排序。
5. `metadata_profile_version` 只在成功事务后更新。

## 10. API and DTO

现有 `PreviewRow` 保持字段兼容，新增：

```json
{
  "classification": {
    "status": "auto_selected",
    "confidence": "high",
    "source": "learned_rule",
    "reason_code": "MERCHANT_EXACT_LEARNED",
    "reason_text": "精确匹配已记住的商户「滴滴出行」",
    "matched_rule_ids": ["rule-id"],
    "suggested_category_id": null,
    "suggested_tag_ids": []
  }
}
```

兼容规则：

1. `suggested_*` 和 `selected_*` 继续保留，旧前端仍可读取。
2. high auto 结果同时写 `selected_*`；medium/low 只写 `suggested_*`。
3. fallback 写入 `selected_category_id` 和 `status=fallback`。
4. 前端提交行调整后，后端更新 `status=manual` 或 `bulk`。
5. commit 不接收客户端 classification explanation，只信任服务端持久化结果。

新增 API 详见 OpenAPI 草案：

```text
GET  /api/metadata/default-profile
POST /api/metadata/default-profile/preview
POST /api/metadata/default-profile/apply
POST /api/imports/{batchId}/rows/bulk-adjust
POST /api/imports/{batchId}/rows/{rowId}/learn
POST /api/imports/{batchId}/reclassify
```

`reclassify` 只允许 ready 批次，使用当前规则重新生成未手工调整行；必须返回变化摘要，不自动执行。

`bulk-adjust` 使用显式 action：

1. `accept_suggestions`：逐行把已持久化的 `suggested_*` 转为 `selected_*`，不同预览行可以接受不同分类和标签。
2. `apply_values`：把请求中的同一分类和标签集合写入选中行。
3. `accept_suggestions` 跳过无建议、conflict、invalid、duplicate 和 skipped 行；`apply_values` 可显式解决 conflict，但仍不得修改 invalid、duplicate 或 skipped 行。
4. 两种 action 都把成功更新的行标记为 `bulk`，不创建学习规则；用户明确选中的 manual/bulk 行可被新的 `apply_values` 再次覆盖。
5. 学习规则只能经独立 `rows/{rowId}/learn` 调用创建或更新，避免批量接口对多个不同商户产生隐式长期副作用。

## 11. Errors

新增稳定错误码建议：

| Code | HTTP | 场景 |
|---|---:|---|
| `CATEGORY_FALLBACK_REQUIRED` | 409 | 归档兜底分类但未指定替代 |
| `CATEGORY_TYPE_MISMATCH` | 400 | 分类类型与交易类型不兼容 |
| `TAG_LIMIT_EXCEEDED` | 400 | 最终标签超过 8 个 |
| `CLASSIFICATION_CONFLICT` | 409 | 同级高置信分类冲突 |
| `CLASSIFICATION_RULE_STALE` | 409 | 规则引用归档或失效元数据 |
| `CLASSIFICATION_MERCHANT_REQUIRED` | 400 | 空商户尝试生成学习规则 |
| `METADATA_PROFILE_CONFLICT` | 409 | 默认包同名冲突未决 |
| `IMPORT_RECLASSIFY_CONFLICT` | 409 | 批次状态或手工调整不允许重算 |

行级冲突应进入 preview row error/details，不得把整个文件解析判为失败；基础包应用冲突则阻断整次 apply。

## 14. Feature flags and rollout

环境开关：

```text
IMPORT_CLASSIFICATION_MODE=off|suggest|graded
```

语义：

1. `off`：保持 Task49 规则建议之前的最低行为，不运行新 classifier。
2. `suggest`：所有规则和 built-in 只建议，不自动写 selected。
3. `graded`：按 PRD 执行高置信自动选择。

建议顺序：development `graded` -> 独立 staging `suggest` -> 验收后 staging `graded` -> production 初始 `suggest` -> 修正率门禁通过后 `graded`。

WSL staging/NAS production 在 Task53 发布评审前继续停留现有 schema/镜像，不执行 migration 022。

## 15. Rollback

1. 行为异常：切换 `IMPORT_CLASSIFICATION_MODE=suggest`，保留解释和规则数据。
2. 内置词典异常：关闭 built-in provider，不影响用户规则。
3. 学习规则异常：批量归档 `origin=learned`，不删除历史预览。
4. 默认包异常：停止新建/apply 入口；已创建分类/标签按普通元数据保留，不自动删除。
5. 应用回退：旧应用忽略 migration 022 新字段；不得 goose down 生产账务数据库。
