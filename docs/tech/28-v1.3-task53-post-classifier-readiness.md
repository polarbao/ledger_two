## 2. Feature flag semantics

`IMPORT_CLASSIFICATION_MODE` 只接受以下值，未知值启动失败：

| Mode | Preview behavior | Commit behavior | Intended use |
|---|---|---|---|
| `off` | 保留 Task49 既有单规则建议，不运行 Task53 classifier | 使用已持久化 selected/suggested，不重分类 | 默认值、代码回退 |
| `suggest` | 运行 Task53 classifier 并持久化解释；所有 classifier 结果只写 suggested，fallback 也不自动写 selected | 不重分类 | staging 观察与误命中评审 |
| `graded` | high auto 写 selected；medium/low 写 suggested；fallback 写兜底 selected | 不重分类 | 通过验收后的候选行为 |

共同约束：manual/bulk 结果始终受保护；duplicate/invalid/skipped 不进入 classifier；切换开关不会反向改写已提交 transaction。

## 3. Task53.3 contract freeze

### 3.1 Persistence mapping

| Classifier decision | import_items writes |
|---|---|
| `auto_selected` | `selected_*`；兼容 `suggested_*`；status/confidence/source/reason/matched rules |
| `suggested` | 仅 `suggested_*`；status/confidence/source/reason/matched rules |
| `fallback` | graded 写 `selected_category_id`；suggest 只保留建议与解释 |
| `conflict` | 不写分类选择；持久化冲突原因和 matched rules |
| `manual` / `bulk` | 保留当前 selected，不由 reclassify 覆盖 |
| `unresolved` | 不伪造分类或命中规则 |

`classification` DTO 只承载状态、来源、置信度、解释、命中规则和 suggested 投影。最终值继续使用 PreviewRow 已有 `selected_category_id`、`selected_account_id`、`selected_tag_ids`；禁止在嵌套 DTO 再复制一套 selected 真值。

### 3.2 Repository transaction boundary

1. preview 在一个 batch 事务中读取一次 classification context，逐行计算并持久化，不为每行重复查询规则。
2. parser、`file_sha256`、`import_hash` 和 duplicate 判定顺序不变；分类在 eligible 判定之后执行。
3. commit 只读取 preview 持久化结果，规则在 preview 后变化不得影响同一批次提交。
4. reclassify dry-run 不写库；执行模式只更新 ready batch 的 eligible 且非 manual/bulk 行。
5. summary 由持久化行状态聚合，不使用前端传入计数。

### 3.3 Reclassify response

返回 `dry_run`、eligible/changed/unchanged、protected manual/bulk、conflict 和各 classification status 计数。执行前后的差异只包含行 ID、旧/新 status 与旧/新最终分类/标签；不返回真实账单原文作为审计 payload。

## 4. Task53.4 contract freeze

### 4.1 Bulk adjustment

路径：`POST /api/imports/{batchId}/rows/bulk-adjust`，仅 active ledger Owner、ready 且未过期批次。

1. `row_ids` 非空、去重且最多 500；服务端按请求顺序返回行 ID。
2. `accept_suggestions` 逐行接受各自持久化 `suggested_*`；客户端不得提交临时建议值。无建议进入 skipped，classification conflict 进入 conflict。
3. `apply_values` 要求 `category_id`、可空 `account_id` 和完整 `tag_ids`；分类/账户/标签属于当前账本且未归档，标签最多 8 个。分类与行类型不兼容时该行进入 errors。
4. invalid/duplicate/skipped 永不修改；`apply_values` 可覆盖用户再次明确选择的 manual/bulk 行并重新标记为 `bulk`。
5. 成功行写 `selected_*`、`row_status=adjusted`、`classification_status/source=bulk`、high confidence 和固定可读原因；保留原 suggested/explanation 作为来源参考。
6. 响应按 `updated_row_ids`、`skipped_row_ids`、`conflict_row_ids` 和行级 `errors` 分类，并返回重算后的 classification summary；部分成功使用 HTTP 200。
7. 顶层参数、权限、账本对象或批次状态错误在事务前失败；进入事务后任一 SQL/审计失败整次回滚，不留下半行更新。
8. 一次请求只写一个 `import_bulk_adjust` 审计，payload 只含 action、行 ID 和计数；不含商户、标题、备注或账单原文，不创建 transaction 或 learned rule。

### 4.2 Explicit learning

路径：`POST /api/imports/{batchId}/rows/{rowId}/learn`，请求只含 `source_scope=current_source|all_sources`。

1. 行必须已通过单行保存或 bulk-adjust 形成最终 `selected_category_id` 与完整 `selected_tag_ids`；API 不接受客户端分类、账户、标签或可见性值。
2. 规范化 merchant 后生成 `merchant_equals`；规则 result 只复制最终分类与标签，不学习账户、可见性、金额或备注。
3. `current_source` 固定 batch source_type；`all_sources` 写 null source_type，必须由 Owner 显式选择。
4. learned rule 初始固定 `origin=learned`、`confidence=high`、`apply_mode=auto`。身份键是 `(ledger_id, source_scope/source_type, normalized_merchant)`；rule ID 使用 namespace `d96c24c9-4ba8-5e56-9f30-f52ddcf8a94a`，以 `ledger_id + U+001F + (all_sources|source:<source_type>) + U+001F + normalized_merchant` 生成 UUIDv5。重复 learn 更新分类/标签并保留用户已编辑的名称、优先级和 apply_mode；已归档 learned rule 返回 `restored` 并恢复。
5. 同范围存在 active manual `merchant_equals` 时返回 `CLASSIFICATION_CONFLICT`，不得覆盖或降级手工规则；响应 details 给 rule ID，不返回其他账本信息。
6. learned rule 的 origin、match_type、规范化 pattern 和 source_type 不可编辑；名称、优先级、apply_mode、分类和标签可编辑。手工创建接口始终生成 `origin=manual`。
7. 行保存与 learn 是两个请求和两个事务；learn 失败不回滚已完成的行保存。learn 审计不保存 merchant 原文或 normalized merchant。
8. 空商户、非 eligible/未保存行、归档元数据、分类类型错配和标签超过 8 个返回冻结错误码。

### 4.3 Metadata safeguards

1. metadata 列表新增 `rule_reference_count`，只统计当前账本 active import rule 的 result 引用。
2. rule DTO 新增 `origin/source_type/apply_mode/confidence/is_stale/stale_reference_ids`。stale rule 不参与候选；修复 update 可保存，restore 在单一写事务内复核 stale/manual conflict、更新状态并写审计，仍 stale 时全部回滚并返回 `CLASSIFICATION_RULE_STALE`。
3. 规则命中指标不新增计数列：`committed_hit_count` 只统计 committed batch 中 status=imported 且 `matched_rule_ids_json` 包含该规则的行；`last_committed_hit_at` 取相应 batch `committed_at`。ready/expired/failed preview 不计入，避免预览刷新放大指标。
4. 归档普通 metadata 保持现有 API。归档 `expense_other` 或 `income_other` 时，同一路径请求必须给 `replacement_category_id`。
5. 替代分类必须属于当前账本、active、同类型、不是自身且当前没有其他 `system_key`。事务内先把兜底 system_key 转移到替代项，再归档旧项并写审计；失败全部回滚。
6. system_key 转移就是 schema 22 的 fallback/profile 指向更新，不修改 `metadata_profile_version`；历史 transaction、transaction_tags 和 import rule result 不改写，因旧对象归档而 stale 的规则由用户显式修复。

### 4.4 Error and response freeze

| Code | HTTP | Boundary |
|---|---:|---|
| `IMPORT_BULK_ADJUST_CONFLICT` | 409 | 批次非 ready、已过期或并发变化 |
| `CATEGORY_FALLBACK_REQUIRED` | 409 | 归档兜底分类未提供替代 |
| `CATEGORY_FALLBACK_REPLACEMENT_INVALID` | 409 | 替代分类状态、类型或 system_key 不合法 |
| `CATEGORY_TYPE_MISMATCH` | 400/row error | 分类与目标交易类型不兼容 |
| `TAG_LIMIT_EXCEEDED` | 400/row error | 最终标签超过 8 个 |
| `CLASSIFICATION_CONFLICT` | 409/row conflict | 显式规则冲突或批量接受冲突行 |
| `CLASSIFICATION_RULE_STALE` | 409/row error | 建议或规则引用不可用 metadata |
| `CLASSIFICATION_MERCHANT_REQUIRED` | 400 | 学习行规范化商户为空 |
| `LEDGER_OBJECT_NOT_FOUND` | 404 | batch/row/metadata 不属于当前账本或不存在 |

错误 details 只能给当前账本对象 ID、行 ID 和原因码，不回显真实账单文本。`openapi-v1.3-category-tag-draft.yaml` 已冻结以上字段；Task53.4 实现不得临时改 DTO 而不联动前端类型、Fixture 和本文。
