# UI：设置页信息架构重组

状态：供审核  
目标：将当前 SettingsPage 从“所有功能堆在一个页面”调整为可长期扩展的信息架构。

## 1. 页面目标

设置页用于管理与“记账数据本身”不同的配置和安全能力，包括账号、账本、成员、分类、标签、支付账户、模板、周期账单、导入导出、备份恢复、系统诊断等。

## 2. 信息架构

```text
/settings
  ├── /profile                账号与登录
  ├── /ledger                 当前账本信息
  ├── /members                成员与权限
  ├── /categories             分类管理
  ├── /tags                   标签管理
  ├── /accounts               支付账户管理
  ├── /templates              账单模板
  ├── /recurring-rules        周期账单
  ├── /import                 CSV 导入
  ├── /export                 数据导出
  ├── /backup                 备份与恢复
  └── /diagnostics            系统诊断
```

Foundation 阶段不要求一次性路由全部拆完，但 SettingsPage 至少要以卡片分区方式呈现，避免继续堆叠。

## 3. 桌面端布局

```text
SettingsPage
  ├── Header：系统设置 / 当前账本 / 当前角色
  ├── Left nav 或 card grid
  └── Content area
```

## 4. 移动端布局

移动端使用分组列表：

```text
账号与安全
账本与成员
分类标签账户
导入导出
备份恢复
系统诊断
```

点击进入二级页面，减少长页面滚动。

## 5. 系统诊断页面

诊断页面展示：

- 服务版本。
- schema version。
- 当前 APP_ENV。
- 数据库连接状态。
- data/backups/uploads/logs 目录状态。
- 当前访问方式 HTTP/HTTPS。
- Cookie Secure 当前策略。
- 最近一次备份时间。
- 最近一次错误日志摘要，可选。

不得展示：

- JWT_SECRET。
- Cookie 值。
- 用户密码 hash。
- 数据库绝对路径，可显示脱敏路径或状态。

## 6. 备份恢复页面

要求：

1. 备份列表清晰展示文件名、大小、时间。
2. 恢复前必须二次确认。
3. 恢复前必须自动创建当前数据备份。
4. 恢复结果应说明下一步手动操作或自动重启策略。
5. 备份文件下载提醒用户妥善保管。

## 7. 成员页面

Foundation 阶段保留现有成员管理能力，但 UI 要明确：

- 当前是直接添加已有用户，不是 v1.1 审核中的邀请机制。
- owner 才能管理成员。
- editor/viewer 只能查看成员列表。
- 移除成员前必须确认。

## 8. 验收标准

1. SettingsPage 不再是单个大页面承载所有内容。
2. 各分区有清晰标题、说明和入口。
3. 数据安全相关入口带高风险提示。
4. 系统诊断不泄露敏感信息。
5. 移动端设置页可读、可操作、无横向滚动。
