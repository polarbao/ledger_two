## 1. 目标

LedgerTwo 后续 UI/UX 优化不做一次性“换皮”，而是建立长期专项：

1. 让日常记账更快。
2. 让结算和共同账本更可信。
3. 让列表、筛选、分类、导入和报表更像成熟记账工具。
4. 保持移动端 375px 可用，不牺牲桌面端效率。
5. 不引入银行同步、OCR、资产负债等短中期非目标。

## 2. 外部产品观察

本节只吸收通用交互经验，不复制品牌、视觉或商业模式。

参考来源：

- YNAB Features：强调跨端同步、共同使用、目标、报表和移动组件。`https://www.ynab.com/features`
- Monarch：强调一站式资金视图、可搜索交易列表、伴侣协作、周期订阅、报告和目标。`https://www.monarch.com/`
- Money Manager：强调周/月总览、票据照片、强化筛选、日历视图、图表和预算。`https://www.realbyteapps.com/`
- Wallet Categories Help：强调分类/子分类、常用分类、标签、自动规则、批量编辑和已使用分类不可直接删除。`https://support.budgetbakers.com/hc/en-us/articles/7077082048146-All-about-Categories-and-Subcategories`

可迁移经验：

| 经验 | 对 LedgerTwo 的启发 |
|---|---|
| 交易列表必须可搜索、可筛选、可快速审阅 | 流水页应从“列表”升级为“可处理工作台” |
| 分类和标签是长期记账的核心信息架构 | 分类名、图标、颜色、归档状态必须在卡片和表单中稳定展示 |
| 周期账单/订阅能降低漏记焦虑 | v1.1 周期待确认应在 Dashboard 有清晰入口 |
| 报表要回答“钱去哪了” | 分析页要优先展示分类、成员、趋势和异常，而不是堆图 |
| 伴侣/家庭协作需要共同视图 | 双人账本要突出“我/对方/共同”三种视角 |
| 移动端负责快速录入和查看，桌面端负责整理和复盘 | 手机优先记账、筛选、结算；桌面优先批量整理、导入、报表 |

## 3. LedgerTwo 体验原则

### 3.1 记账优先于配置

- 首页和底部导航必须让“记一笔”始终容易触达。
- 表单默认值应减少输入，而不是替用户做不可见决策。
- 保存并继续记必须清空金额、标题、备注和附件，保留分类、账户、付款人、可见性。

### 3.2 金额和结算必须可解释

- 金额展示永远用元，API 和代码永远用分。
- 共同支出必须可看到付款人、参与人、分摊方式。
- 结算页必须解释 paid/share/raw_net/settlement/final_net。

### 3.3 列表是工作台

流水列表不只是展示，应支持：

- 搜索标题、备注、标签。
- 按类型、分类、账户、付款人、可见性筛选。
- 移动端卡片，桌面端表格。
- 批量打标签。
- 快速复制一笔、存为模板、删除确认。

### 3.4 元数据要有生命周期

- 分类、标签、账户要有新增、编辑、排序、归档、恢复。
- 已归档项不出现在新增表单默认选择器。
- 历史账单必须显示归档项名称，不能退化成 UUID。
- 已使用项不可物理删除。

### 3.5 数据安全可见但不吓人

- 备份、导出、恢复和导入必须有明确说明。
- 高风险操作用二次确认，不用轻飘的 toast。
- 系统诊断要脱敏，只展示用户可行动的信息。

## 4. 当前界面改进路线

### 4.1 v1.1 收口内

目标：只修复影响高频使用和信任的问题，不大改视觉体系。

必须完成：

1. 移动端流水卡片显示分类名，不显示分类 UUID。
2. 保存并继续记实际 UI 验收通过。
3. 普通支出、共同支出实际 UI 提交成功。
4. 结算页复制文案和登记结算完成一次真实验收。
5. 附件上传和受控读取完成一次真实验收。

可选但推荐：

- Dashboard 最近流水展示分类名、付款人和共同/个人标记。
- 设置页高风险入口增加统一的危险操作文案。

### 4.2 v1.2 导入期间

目标：把 CSV/XLSX 导入做成“可控工作台”，而不是一次性上传按钮。

必须完成：

1. 导入预览移动端卡片化。
2. invalid、duplicate、suspicious、new 四种状态颜色和说明统一。
3. 提交前二次确认展示待导入、跳过、疑似、错误数量。
4. 行详情可调整分类、标签、账户、可见性。
5. 规则命中要有“推荐来源”提示，但不得覆盖用户手工修改。

### 4.3 v1.2 收口后的 Fresh Light 专项

目标：在不改变已冻结业务、API 和验收事实的前提下，将全应用从 Dark Glass 分阶段迁移到 Fresh Light。

执行方式：

1. UI-FL-01/02 先冻结 Token、基础组件、AppShell 和共享断点。
2. UI-FL-03/04/05 再迁移 Dashboard、记账和流水高频路径。
3. UI-FL-06/07 迁移结算、设置与元数据可信交互。
4. UI-FL-08/09 复用已完成导入与统计契约，迁移复杂工作台。
5. UI-FL-10 完成 375/390/430/1440、可访问性、真实业务和回退验收。

每个 UI-FL 任务必须关联既有业务 Task、声明共享组件归属和 API 不变范围。详细协同与并行规则见 `docs/codex_tasks/13-fresh-light-ui-interaction-plan.md`。

### 4.4 v1.3+ 长期

目标：从“能用”升级为“长期复盘和维护舒服”。

候选方向：

1. Dashboard 可定制信息块：结算、周期、最近流水、分类趋势。
2. 月历视图：按天看支出密度和漏记日。
3. 报表钻取：分类图表点击后进入对应流水筛选。
4. 账单审阅状态：导入或补账后标记已核对。
5. 大屏整理模式：桌面端批量编辑分类、标签、账户。

## 5. 页面级专项清单

### 5.1 Dashboard

优化目标：

- 第一屏回答：本月花了多少、谁垫付、谁应付、是否有待确认。
- 最近流水不显示技术 ID。
- 共同支出和个人支出视觉上可快速区分。

验收：

- 375px 第一屏不需要横向滚动。
- 用户能在 5 秒内找到“记一笔”和“当前应结算金额”。

### 5.2 记账抽屉

优化目标：

- 金额输入优先。
- 高频字段靠前，低频字段靠后。
- 保存并继续记可连续录入 3 笔，不产生脏字段继承。

验收：

- 普通支出 10 秒内完成。
- 共同支出 20 秒内完成。
- 保存并继续后金额为空、标题为空、备注为空、附件为空。

### 5.3 流水列表

优化目标：

- 移动端卡片展示：标题、金额、分类名、日期、个人/共同、付款人。
- 桌面端表格展示：日期、类型、分类、标题、付款人、分摊、金额、操作。
- 不展示 UUID，除非是调试或诊断页面。

验收：

- 375px 卡片长标题不撑破布局。
- 未分类账单显示“未分类”。
- 已设但未能解析名称时显示“已设分类”，不显示 ID。

### 5.4 设置与元数据

优化目标：

- 设置页按账号、账本、元数据、模板/周期、导入导出、备份诊断分区。
- 分类、标签、账户管理页支持筛选、归档、恢复和历史引用提示。

验收：

- owner 能完成新增、编辑、归档、恢复。
- viewer 不显示管理入口，后端也拒绝。

### 5.5 导入工作台

优化目标：

- 预览先于提交。
- 状态解释清晰。
- 用户能批量处理重复、疑似和错误行。

验收：

- 用户确认前不写 transactions。
- 提交确认文案必须写清楚导入影响。
- 移动端不展示宽表格。

## 6. 设计系统约束

LedgerTwo 当前已验收实现是 Dark Glass，Fresh Light 是分阶段迁移目标。两者并存期间应遵守：

1. 不引入营销页式 hero。
2. 不把所有页面都做成嵌套卡片。
3. 工具型页面优先信息密度和可扫描性。
4. 按钮文案表达动作，不写模糊“确定”。
5. 图标用于辅助识别，不能替代关键文字。
6. 移动端主按钮全宽，危险按钮和主按钮保持视觉间隔。
7. 页面不得新增局部颜色、圆角或同名组件，优先复用 UI-FL-01/02 冻结的 Token 和基础组件。
8. Dark Glass 在 Fresh Light 全局验收前保持可回退，不以删除旧主题作为页面迁移前提。
9. 全局验收前允许用户通过太阳/月亮按钮显式切换并持久化主题；该入口用于逐页验收，不等同于 Fresh Light 已成为默认生产主题。

## 8. 与业务任务的同步规则

1. 业务 Task 触及 UI 时必须登记关联 UI-FL 编号、路由、共享组件归属、并行冲突和验收视口。
2. UI-FL 任务必须反向列出复用的业务 Task，不能只依据截图修改实现。
3. 任务启动、组件契约冻结、实现完成、验收完成四个节点同步更新任务文档、必要 handoff 和验收记录。
4. 共享组件由 UI-FL-01/02/04/05 分别负责基础控件、AppShell、表单和响应式列表；其他任务不得平行创建近似实现。
5. 业务缺陷和发布阻断项优先于视觉迁移；发生文件所有权冲突时暂停 UI 页面任务，先完成基础或业务归属任务。

## 9. 验收记录要求

每次 UI/UX 专项提交必须包含：

1. 修改前后的目标页面或组件说明。
2. 移动端 375px 截图或 CDP 指标。
3. 至少一个真实业务操作或可解释的未验证原因。
4. 如果是纯文档，说明未运行前后端测试。
5. 如果影响表单、金额、权限、导入或备份，必须补业务验收记录。
