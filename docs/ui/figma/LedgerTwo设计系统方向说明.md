# LedgerTwo 设计系统方向说明

状态：方向规范已冻结，持续维护；2026-07-13 完成 Fresh Light 方向更新<br>
来源：`docs/prd/22-v1.1可信赖与高频记账.md`、`docs/prd/29-v1.2导入模块业务与服务细分.md`、`docs/ui/14-v1.1至v1.2模块流程.md`、`docs/ui/15-LedgerTwo长期体验优化专项.md`、当前 React 前端、`docs/ui/Fresh-Light视觉参考原型.html`

## 1. 产品语气

LedgerTwo 是双人长期记账工具，不是营销型财务 App。界面应让用户觉得：

1. 日常记账很快。
2. 共同支出和结算可信、可解释。
3. 历史数据、安全操作和导入动作有明确边界。
4. 移动端适合随手记录，桌面端适合整理和复盘。

设计表达应克制、清晰、可扫描，避免营销式 Hero、复杂动效、强发光和卡片套卡片。

## 2. 当前前端与阶段事实

当前前端主要是 Dark Glass：

| 项目 | 当前情况 |
|---|---|
| 主背景 | 深色 `#0A0C10`、`#12151E` |
| 卡片 | 半透明玻璃、较大圆角、发光边界 |
| 品牌色 | 紫色 `#9333EA` + 绿色 `#10B981` |
| 布局 | 桌面侧栏、顶部月份、移动底部 Tab |
| 组件 | 页面内联样式较多，Token 与组件复用仍不充分 |

优点：

1. v1.1 和 v1.2 已围绕该体系完成多轮移动端和真实业务验收。
2. 设置、结算、交易和导入工作台已经有可复用结构。
3. 在版本冻结阶段继续使用的回归风险较低。

问题：

1. 深色 + 紫绿渐变容易呈现开发者 Demo 感，不利于长期家庭记账。
2. 大量内联样式让 Figma、Token 和代码难以一一对应。
3. 高频记账、流水审阅和导入预览需要更高扫描效率，强玻璃效果会造成干扰。
4. Dashboard、流水、记账、结算、分析和设置尚未统一在一个成熟的设计语言下。

阶段结论：

- Dark Glass 是已经验收的历史/生产基线，不应被文档抹除。
- Fresh Light 是 v1.2 收口后的全应用目标设计基线。
- 迁移必须通过 Token、基础组件和页面原子任务逐步实施，不一次性重写。
- Dark Glass 暂时保留为可回滚模式。

## 3. `Fresh-Light视觉参考原型.html` 可吸收的语言

| 蓝本元素 | LedgerTwo 使用方式 |
|---|---|
| 浅色背景 `#F5F9FC` | 全应用目标页面底色 |
| 绿色主色 `#10B981` | 财务确认、同步正常和主要动作 |
| 深青文字 `#0B3B3B` | 浅色主题主文本 |
| 月度摘要卡 | Dashboard 首屏和导入批次摘要 |
| 分段 Tab | 流水、结算范围、导入状态和统计视角 |
| 交易卡片 | 移动流水、Dashboard 最近流水和导入行 |
| 轻量状态 Chip | 付款人、共同/个人、可见性和导入状态 |
| 轻边框与阴影 | 提升扫描性，不做强玻璃和发光 |

不直接照搬：

1. 手机模型式大圆角主壳；真实 App 使用全屏工作台。
2. Emoji 和装饰符；生产 UI 使用 Lucide 图标和可读文字。
3. 纯移动单列结构；桌面端保留表格、批量整理和报表效率。
4. 过软的信息层级；结算、导入、恢复等高风险动作需要清晰边界。
5. 把每个区块都做成嵌套卡片。

## 4. 体验原则

### 4.1 记账优先于配置

- “记一笔”在首页、移动端和流水空状态始终容易触达。
- 金额优先，高频字段靠前，低频字段折叠。
- 默认值减少输入，但不替用户做不可见决策。
- 保存并继续必须清空金额、标题、备注和附件，并按现有规则保留分类、账户、付款人和可见性。

### 4.2 付款、承担与结算必须区分

- 不把创建人、付款人和承担人混为一谈。
- 共同支出必须看到付款人、参与人、分摊方式和承担预览。
- 结算页以中文解释 paid/share/raw_net/settlement/final_net。
- settlement 是独立记录，不进入消费统计，不修改历史共同支出。

### 4.3 列表是工作台

- 桌面流水使用搜索、筛选、表格和详情抽屉。
- 移动流水使用卡片和 Bottom Sheet 筛选。
- 支持复制一笔、存为模板、编辑和软删除确认。
- 不向普通用户显示 UUID。

### 4.4 元数据有生命周期

- 分类、标签和账户支持新增、编辑、排序、归档和恢复。
- 归档项不进入新增表单默认选择器。
- 历史账单继续显示名称，并标注已归档。
- 已使用项不物理删除。

### 4.5 安全可见但不恐吓

- 导入、导出、备份和恢复必须说明数据影响。
- 高风险动作使用二次确认和明确动词。
- 系统诊断脱敏，只展示用户可行动的信息。

## 5. 信息架构

一级导航：

```text
首页 / 流水 / 结算 / 分析 / 设置
```

二级工具：

```text
账单导入 / 周期规则 / 模板 / 草稿箱 / 系统诊断
```

移动端保留五个一级 Tab，“记一笔”作为导航上方的主要动作，不作为第六个等权 Tab。

## 6. 主题与 Token 策略

1. Fresh Light 使用 `ledger-two.design-tokens.json` 和 `ledger-two.figma-variables.json`。
2. 新组件只能引用语义 Token，不随意新增颜色、圆角和阴影。
3. 先映射现有 CSS 变量，再拆除页面内联色值。
4. 紫色降级为辅助色，绿色和深青承担主要财务语义。
5. Dark Glass 保留为模式，待 Fresh Light 页面完成真实验收后再决定默认主题。
6. 任何主题迁移均需验证 375/390/430/1440 和离线、权限、高风险状态。

## 7. 页面目标

### Dashboard

第一屏回答：本月花了多少、双方支付/承担、谁应给谁、是否有待确认账单。收入降为次级摘要，最近流水显示分类、付款人和共同/个人。

### 记账抽屉

金额、类型、分类、账户、日期优先；共同支出展开付款人、分摊和承担预览；标题、标签、可见性、备注、附件和模板管理折叠。

### 流水

桌面为可筛选表格和详情抽屉，移动为卡片和筛选 Bottom Sheet。分类不得显示 UUID，归档项显示历史名称。

### 结算

展示“谁应转给谁”、范围、影响账单、实际支付、实际承担、净额和历史结算。复制文案不改变状态，登记结算生成独立记录。

### 分析

按趋势、分类、成员和标签回答问题。成员统计展示支付、承担和净垫付，不使用误导性的“记账人排行”。

### 设置

按账号、账本、元数据、模板/周期、导入导出、备份恢复和诊断组织。危险操作独立分区。

### 导入

继续执行 Preview 先于 Commit、状态可解释、疑似/错误阻断、移动卡片化和用户手工调整优先。
