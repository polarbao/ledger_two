# LedgerTwo 设计系统方向说明

状态：建议采纳  
来源：`docs/prd/22-prd-v1.1-trust-and-daily-use.md`、`docs/prd/29-prd-v1.2-module-business-service-breakdown.md`、`docs/ui/14-v1.1-v1.2-module-flows.md`、`docs/ui/15-ledgertwo-ux-optimization-program.md`、当前 React 前端、`docs/ui/lynntest(1).html`

## 1. 产品语气

LedgerTwo 是双人长期记账工具，不是营销型财务 App。界面要让用户觉得：

1. 日常记账很快。
2. 共同支出和结算很可信。
3. 历史数据、安全操作和导入动作可解释。
4. 移动端适合随手记录，桌面端适合整理和复盘。

设计表达应克制、清晰、可扫描，避免大面积装饰、营销式 hero、复杂动效和嵌套卡片。

## 2. 现有前端风格判断

当前前端主要是深色玻璃体系：

| 项目 | 当前情况 |
|---|---|
| 主背景 | 深色 `#0a0c10`、`#12151e` |
| 卡片 | 半透明玻璃、16px 圆角、发光边框 |
| 品牌色 | 紫色 `#9333ea` + 绿色 `#10b981` 渐变 |
| 布局 | 桌面侧边栏、顶部月份、移动端底部 Tab |
| 组件 | 大量页面内联样式，统一 token 仍不充分 |

优点：

1. v1.1 已围绕该风格完成多轮移动端和真实业务验收。
2. 深色视觉和玻璃卡片在设置、结算、交易工作台中已有组件沉淀。
3. 当前切换成本低，适合继续做 v1.1 收口。

问题：

1. 深色 + 紫绿渐变容易显得偏开发者 Demo，而不是长期家庭记账工具。
2. 页面中内联样式较多，Figma 与代码 token 难以一一对应。
3. 高频记账和导入预览需要更高信息密度，过强的玻璃效果会干扰扫描。
4. `lynntest(1).html` 所体现的轻量财务感与当前视觉差异较大，不能直接局部混用。

## 3. `lynntest(1).html` 可吸收的设计语言

蓝本中的可迁移部分：

| 蓝本元素 | LedgerTwo 使用方式 |
|---|---|
| 浅色背景 `#f5f9fc` | 作为 v1.2 工作台和未来浅色主题的页面底色 |
| 绿色主色 `#10b981` | 作为财务确认、同步正常、正向操作主色 |
| 深青文字 `#0b3b3b` | 作为浅色主题主文本 |
| 月度摘要卡 | Dashboard 首屏和导入批次摘要可复用结构 |
| 分段 Tab | 流水筛选、导入状态筛选、统计视角切换 |
| 交易卡片 | 移动端流水和导入预览行的基础模式 |
| 轻量状态 Chip | 付款人、共同/个人、duplicate/suspicious/invalid 状态 |

不建议直接照搬：

1. 大圆角容器作为真实产品主壳。真实 App 需要全屏工作台，不应像单个手机壳展示。
2. 过多 emoji 或装饰符。生产 UI 使用 lucide 图标和文本状态即可。
3. 纯移动端单列结构。LedgerTwo 桌面端需要批量整理、导入和报表效率。
4. 过软的信息层级。结算、导入和恢复等高风险操作需要更强边界。

## 4. v1.1 是否需要进一步调整

结论：v1.1 不建议做大规模 UI/UX 风格迁移，只做收口级优化。

理由：

1. v1.1 当前目标是可信赖与高频记账冻结，不是视觉重构。
2. 已有本地 WSL2 UI 验收覆盖 Dashboard、流水、结算、复制、模板、设置和数据安全路径。
3. 视觉迁移会影响大量页面内联样式、移动端断点和表单抽屉，冻结前风险高。
4. v1.1 剩余问题更偏证据和验收：NAS 复核、历史归档项展示、耗时目标证据。

v1.1 可做的轻量调整：

1. 继续修复横向溢出、长文本挤压、状态不可读等问题。
2. 将高风险按钮文案和状态提示统一。
3. 将分类、标签、账户、付款人、共同/个人标记展示清楚。
4. 不引入新的浅色主题，不重写全局布局。

## 5. v1.2 和长期建议

v1.2 建议以“浅色工作台试点”承接 `lynntest(1).html`：

1. 导入模块是新工作台，适合建立更清爽的表格/卡片/状态视觉。
2. 浅色主题更适合大批量行预览、错误标注、重复判断和规则解释。
3. 通过 token 先统一颜色、圆角、间距、状态，不直接推翻现有 AppShell。

v1.3+ 再评估是否全局演进：

1. 保留深色主题作为可选模式。
2. 将绿色/青色作为财务主色，紫色降级为辅助强调。
3. 减少玻璃态和强发光，提升长时间使用舒适度。
4. 把 Dashboard、流水、结算、设置逐步迁移到同一 token 体系。
