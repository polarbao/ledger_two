# v1.1-v1.2 UI 设计稿规格

状态：建议采纳  
蓝本：`docs/ui/lynntest(1).html`  
说明：本文是 Figma 逐屏设计稿的文字规格。每完成一个真实 Figma 页面或截图设计稿后，应在本文或同目录新增对应稿件记录。

## 1. v1.1 设计稿

### 1.1 Dashboard Mobile 390

目标：5 秒内看到本月总支出、结算方向和记一笔入口。

结构：

1. 顶部：账本名、同步/离线状态、月份选择。
2. 月度摘要：本月总支出、我支付、对方支付、待结算。
3. 快捷动作：记一笔、查看结算。
4. 周期账单提醒：待确认、跳过、确认记账。
5. 最近流水：最多 5 条，展示分类、金额、付款人、共同/个人、日期。

视觉：

1. v1.1 保持 Dark Glass。
2. 若出 Fresh Light 探索稿，不进入 v1.1 冻结代码。
3. 状态 chip 使用绿色、黄色、红色，不只依赖图标。

验收：

1. 375px 无横向滚动。
2. 金额不被长标题挤压。
3. 周期提醒为空时不展示异常空白块。

### 1.2 Transactions Mobile 390

目标：流水列表从展示页升级为可处理工作台。

结构：

1. 顶部筛选摘要：月份、关键词、活跃筛选数量。
2. 分段筛选：全部、个人、共同、收入。
3. 列表卡片：分类、标题、金额、付款人、标签、日期、可见性。
4. 卡片动作：详情、复制一笔、存为模板、删除。
5. 底部筛选抽屉：分类、标签、金额区间、付款人、可见性。

验收：

1. 分类不得显示 UUID。
2. 已归档分类展示历史名称并带“已归档”状态。
3. 批量模式不破坏单卡片布局。

### 1.3 Transaction Drawer Mobile

目标：普通支出 10 秒内完成，共同支出 20 秒内完成。

结构：

1. 金额输入置顶。
2. 类型使用分段控件。
3. 分类、账户、付款人使用最近选择优先。
4. 共同支出展开参与人和分摊方式。
5. 低频字段折叠：标签、时间、备注、附件、可见性。
6. 底部固定动作：保存、保存并继续。

验收：

1. 保存并继续不关闭抽屉。
2. 保存并继续后金额、标题、备注、附件清空。
3. 共同支出保存前能看见每人承担金额。

### 1.4 Settlement Mobile 390

目标：让用户理解谁该给谁多少钱，以及复制文案不会改变账本。

结构：

1. 顶部结论：`lynn 应转给 polar ¥128.50`。
2. 解释区：paid、share、raw_net、settlement、final_net。
3. 动作区：复制结算文案、登记结算。
4. 历史记录：本月结算记录。

验收：

1. 复制失败有兜底。
2. 登记结算前明确不会修改历史共同支出。
3. 空结算状态有解释文案。

### 1.5 Settings Desktop

目标：把设置页从杂项入口变成可信赖控制台。

结构：

1. 账号与登录。
2. 账本与成员。
3. 分类、标签、支付账户。
4. 模板与周期账单。
5. 导入导出。
6. 备份恢复。
7. 系统诊断。

验收：

1. owner-only 操作在前端隐藏或禁用，后端仍拒绝非 owner。
2. 恢复备份需要二次确认。
3. 诊断信息脱敏。

## 2. v1.2 设计稿

### 2.1 Import Entry Desktop

目标：用户知道自己在导入什么来源，以及导入前不会写正式账单。

结构：

1. 页面标题：导入工作台。
2. 来源选择：微信、支付宝、通用 CSV。
3. 上传区：文件选择、编码/格式错误提示。
4. 说明区：预览先于提交，owner-only。
5. 历史批次入口。

验收：

1. 非 owner 不展示上传入口或展示只读无权限提示。
2. 上传失败给出可行动错误。
3. preview 阶段明确不写 transactions。

### 2.2 Import Preview Desktop

目标：桌面端高效审阅和批量处理导入行。

结构：

1. 批次摘要：总行数、新增、重复、疑似、错误。
2. 状态筛选：全部、新增、重复、疑似、错误、已调整、已跳过。
3. 行表格：行号、时间、商户、金额、状态、推荐、目标分类、操作。
4. 右侧或抽屉：行编辑。
5. 底部提交栏：阻断原因和确认导入。

验收：

1. invalid 和 suspicious 未处理时提交按钮禁用。
2. duplicate 默认跳过。
3. 规则命中来源可见。

### 2.3 Import Preview Mobile 390

目标：移动端可审阅小批量导入，不展示宽表格。

结构：

1. 批次摘要折叠卡。
2. 状态分段控件。
3. 导入行卡片列表。
4. 行卡片展示：商户、金额、时间、状态、推荐分类、错误原因。
5. 行操作：编辑、跳过、确认疑似。

验收：

1. 375px 无横向滚动。
2. 每张卡片的状态文字清晰。
3. 错误原因不被省略到不可理解。

### 2.4 Import Row Editor Drawer

目标：让用户把一行候选账单修正成可提交账单。

字段：

1. 标题。
2. 类型和方向。
3. 金额，只读或受控编辑。
4. 时间。
5. 分类。
6. 账户。
7. 标签。
8. 可见性。
9. 跳过原因。

规则：

1. 用户手工修改后，规则不得覆盖 selected 字段。
2. 引用归档元数据时必须提示替换。
3. 保存只更新 import row，不创建 transaction。

### 2.5 Commit Confirm Modal

目标：用户提交前清楚知道本次会写入什么。

内容：

1. 将导入 N 行。
2. 将跳过 N 行。
3. 疑似未处理 N 行。
4. 错误 N 行。
5. 成功后生成正式账单并写审计日志。

阻断：

1. 疑似未处理不能提交。
2. 错误未修复不能提交。
3. 已提交批次不能再次提交。

### 2.6 Import Rule Manager

目标：规则只做推荐，减少重复选择，不替用户提交。

结构：

1. 规则列表：优先级、匹配条件、推荐结果、状态。
2. 新增/编辑规则表单。
3. 归档/恢复。
4. 命中解释预览。
5. 多标签推荐选择。
6. 归档元数据提示。

推荐结果：

1. 分类和账户为单选。
2. 标签为多选 chip/checkbox。
3. 命中解释必须展示推荐分类、账户、标签名称。
4. 不用数据库 ID 作为主视觉文案。

移动端：

1. 390px 下表单单列。
2. 标签可换行，不出现横向滚动。
3. 规则卡片长文案使用 `anywhere` 换行。
4. 归档/恢复/编辑按钮保持可触达，不挤压文字。

验收：

1. 归档规则不再命中。
2. 规则引用归档分类时阻止静默应用。
3. priority 方向和后端一致。
4. 创建和编辑规则都能提交多个 `tag_ids`。
5. 预览行可解释“规则为什么命中”和“建议写入什么”。

## 3. UI 设计稿输出要求

每完成一个 UI 设计稿，必须同步输出：

1. 对应 Figma frame 名称。
2. 适用版本和任务编号。
3. 使用的主题模式。
4. 页面状态：默认、空、加载、错误、权限不足、高风险确认。
5. 与前端组件的映射。
6. 验收方式：截图、CDP 指标、真实操作或无法验证原因。
