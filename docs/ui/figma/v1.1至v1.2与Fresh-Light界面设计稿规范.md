## 1. v1.1 历史设计稿

### 1.1 Dashboard Mobile 390

目标：5 秒内看到本月总支出、结算方向和记一笔入口。

结构：

1. 顶部：账本名、同步/离线状态、月份选择。
2. 月度摘要：本月总支出、我支付、对方支付、待结算。
3. 快捷动作：记一笔、查看结算。
4. 周期账单提醒：待确认、跳过、确认记账。
5. 最近流水：最多 5 条，展示分类、金额、付款人、共同/个人、日期。

历史视觉：v1.1 保持 Dark Glass；Fresh Light 探索不回写 v1.1 冻结事实。

验收：375px 无横向滚动；金额不被长标题挤压；周期提醒为空时不留异常空白。

### 1.2 Transactions Mobile 390

目标：流水从展示页升级为可处理工作台。

结构：月份、关键词、活跃筛选数量、全部/个人/共同/收入分段、交易卡片、卡片动作和筛选 Bottom Sheet。

卡片展示：分类、标题、金额、付款人、标签、日期、可见性。  
动作：详情、复制一笔、存为模板、删除。

验收：不显示 UUID；归档分类显示历史名称和“已归档”；批量模式不破坏卡片布局。

### 1.3 Transaction Drawer Mobile

目标：普通支出 10 秒、共同支出 20 秒。

结构：金额置顶、类型分段、最近分类/账户/付款人、共同支出参与人与分摊、低频字段折叠、底部固定保存和保存并继续。

验收：保存并继续不关闭；金额、标题、备注和附件清空；共同支出保存前看到每人承担金额。

### 1.4 Settlement Mobile 390

目标：理解谁应给谁多少钱，以及复制文案不会改变账本。

结构：顶部结论、paid/share/raw_net/settlement/final_net 解释、复制结算文案、登记结算、本月历史记录。

验收：复制失败有兜底；登记前说明不会修改历史共同支出；空结算状态有解释。

### 1.5 Settings Desktop

结构：账号与登录、账本与成员、分类/标签/支付账户、模板与周期账单、导入导出、备份恢复、系统诊断。

验收：owner-only 前端隐藏或禁用且后端继续拒绝；恢复备份二次确认；诊断信息脱敏。

## 2. v1.2 导入工作台设计稿

### 2.1 Import Entry Desktop

目标：用户知道导入来源，且 preview 前不会写正式账单。

结构：来源选择、文件上传、可行动错误、preview 说明、Owner/Editor 自有批次权限和 Viewer 无权限状态；规则管理与商户学习保持 Owner-only。

格式：微信支持 CSV/XLSX；支付宝和通用模板仅 CSV。

### 2.2 Import Preview Desktop

结构：批次摘要、状态筛选、行表格、行编辑抽屉、固定提交栏。

摘要：总行、新增、重复、疑似、错误、已调整、已跳过。  
表格：行号、时间、商户、金额、状态、推荐、目标分类和操作。

验收：invalid/suspicious 未处理时禁用提交；duplicate 默认跳过；规则命中来源可见。

### 2.3 Import Preview Mobile 390

结构：可折叠批次摘要、状态分段、导入行卡片、编辑/跳过/确认疑似。

卡片显示：商户、金额、时间、状态、推荐分类、错误原因。

验收：375px 无横向滚动；状态文字清楚；错误原因不被截断到不可理解。

### 2.4 Import Row Editor Drawer

字段：标题、方向、金额、时间、分类、账户、标签、可见性、跳过原因。

规则：用户手工修改后规则不得覆盖；归档元数据必须提示替换；保存只更新 import row，不创建 transaction。

### 2.5 Commit Confirm Modal

展示：将导入、将跳过、疑似未处理、错误数量，以及成功后创建正式账单和审计日志。

阻断：疑似未处理、错误未修复、批次已提交。

### 2.6 Import Rule Manager

结构：规则列表、优先级、匹配条件、推荐结果、归档/恢复、命中解释、多标签和归档元数据提示。

规则只做推荐，不替用户提交；名称作为主视觉，不显示数据库 ID。

移动端：390px 单列，标签可换行，长文案 `anywhere`，操作按钮不挤压文字。

## 3. Fresh Light 全应用目标设计

适用：v1.2 收口后的 UI-FL 专项。  
主题：Fresh Light。  
事实源：`2026-07-13-Fresh-Light界面实施规范.md`。

### 3.1 Foundations

#### Fresh Light Tokens

Frame：`00 Foundations / Fresh Light Tokens`，1440×900。

展示：页面/表面/控件背景、边框、主/次/弱文字、绿色主色、深青强调、信息/警告/危险/辅助紫色、阴影和 Focus Ring。

#### Typography Spacing Radius

Frame：`00 Foundations / Typography Spacing Radius`，1440×900。

展示：12/13/14/16/18/22/28 字号，4-48 间距，8/12/16/20/999 圆角，金额等宽数字。

### 3.2 Components

#### Buttons Chips Segments

Frame：`01 Components / Buttons Chips Segments`。

覆盖 Primary、Secondary、Danger、Ghost、disabled、loading；状态 Chip 必须包含文字；Segment 支持可选计数。

#### Transaction Cards and Table Rows

Frame：`01 Components / Transaction Cards and Table Rows`。

同一交易对象在移动卡片和桌面表格中保持分类、标题、付款人、共同/个人、日期、金额和状态口径一致。

#### Drawers Sheets Confirm Dialogs

Frame：`01 Components / Drawers Sheets Confirm Dialogs`。

覆盖桌面右侧抽屉、移动 Bottom Sheet、脏字段关闭确认、删除/结算/导入/恢复高风险确认。

### 3.3 Dashboard Desktop 1440

Frame：`02 Fresh Light Daily Use / Dashboard Desktop 1440`。  
路由：`/`。  
React：`frontend/src/pages/DashboardPage.tsx`。

首屏：页面标题与记一笔、本月总支出、我已支付、对方已支付、待结算、结算行动卡、周期提醒。  
下方：最近流水和分类摘要。

规则：收入降级为次级；跨月范围写清楚；最近流水不显示技术 ID；分类可钻取流水。

### 3.4 Dashboard Mobile 390

Frame：`02 Fresh Light Daily Use / Dashboard Mobile 390`。

首屏必须同时可见月份、总支出、待结算和记一笔入口。周期提醒最多展示 2-3 条；最近流水最多 5 条。

验收：375/390 无横向滚动，FAB 不遮挡列表和导航。

### 3.5 Transactions Desktop 1440

Frame：`02 Fresh Light Daily Use / Transactions Desktop 1440`。  
路由：`/transactions`。  
React：`frontend/src/pages/TransactionsPage.tsx`。

结构：搜索、月份、快速分段、更多筛选、活跃筛选 Chip、导出、表格和详情抽屉。

表格列：日期、类型、分类、标题、付款人、分摊/可见性、标签、金额、操作。

### 3.6 Transactions Mobile 390

Frame：`02 Fresh Light Daily Use / Transactions Mobile 390`。

结构：搜索、筛选计数、分段控件、交易卡片和筛选 Bottom Sheet。

长标题两行截断，金额固定右侧；归档项显示历史名称；不显示 UUID。

### 3.7 Transaction Drawer Desktop

Frame：`02 Fresh Light Daily Use / Transaction Drawer Desktop`。  
React：`frontend/src/components/transaction/TransactionFormDrawer.tsx`。

字段顺序：金额、类型、分类、账户、日期；共同支出展开付款人、分摊和承担预览；标题、标签、可见性、备注、附件和模板管理折叠。

底部：取消、保存并继续、保存账单。

### 3.8 Transaction Sheet Mobile

Frame：`02 Fresh Light Daily Use / Transaction Sheet Mobile`。

接近全屏 Bottom Sheet，金额自动聚焦，主按钮全宽，存在脏字段时关闭确认。

验收：普通支出 10 秒、共同支出 20 秒；保存并继续无脏字段继承。

### 3.9 Settlement Desktop 1440

Frame：`02 Fresh Light Daily Use / Settlement Desktop 1440`。  
路由：`/settlement`。  
React：`frontend/src/pages/SettlementPage.tsx`。

结构：全部未结/仅本月、谁应给谁、影响账单、复制文案、生成结算记录、中文解释表和历史结算。

### 3.10 Settlement Mobile 390

Frame：`02 Fresh Light Daily Use / Settlement Mobile 390`。

主结论优先；解释区可折叠但关键最终未结不可隐藏；危险动作与主按钮分隔。

### 3.11 Analytics Desktop 1440

Frame：`02 Fresh Light Daily Use / Analytics Desktop 1440`。  
路由：`/analytics`。

按趋势、分类、成员和标签组织。成员显示支付、承担和净垫付，不使用“记账人排行”。图表支持钻取并配套可读数据。

### 3.12 Settings Desktop 1440

Frame：`02 Fresh Light Daily Use / Settings Desktop 1440`。  
路由：`/settings`。

左侧设置分区、右侧内容；元数据提供搜索、状态、排序、使用次数、归档和恢复；危险区独立。

### 3.13 States

Frame：`04 States / Empty Loading Error Offline Permission`。

覆盖 Dashboard、流水、表单、结算、导入和设置的加载、空、错误、离线、权限不足和高风险确认。

### 3.14 Code Mapping

Frame：`05 Code Mapping / React Component Map`。

必须列出 Frame、路由、当前 React 文件、建议子组件、API/权限/金额影响和验收方式。

## 4. 输出和验收要求

每个设计稿必须同步记录：

1. Frame 名称。
2. 适用版本和 UI-FL 任务编号。
3. 主题模式。
4. 默认、空、加载、错误、离线、权限不足和高风险确认状态。
5. React 文件和组件映射。
6. 是否影响 API、金额、权限、导入、备份或 migration。
7. 375/390/430/1440 截图、CDP 指标、真实操作或无法验证原因。

不得仅凭 Figma URL 声称完成同步；必须有 Frame/节点、截图或导出物证据。
